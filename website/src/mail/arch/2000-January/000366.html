At 05:46 PM 1/1/00 -0500, digigod@earthling.net wrote:
>wouldn't alt.awt.peer.2 make more sense?

There are at least three products called "AWT". It is most difficult to
undo this kind of mistake. I would like to think of them as "AWT for Java 0
Platform", "AWT for Java 1 Platform" and "AWT for Java 2 Platform".

Why do I think about "peer" as a subpackage of "alt.awt2"? Because each
implementation of "peer" is closely associated with a specific
implementation of "AWT". Since each AWT has a different number of packages,
my proposal is this:

AWT for Java 0 Platform:
alt.awt0=java.awt
alt.awt0.image=java.awt.image
alt.awt0.peer=java.awt.peer

AWT for Java 1 Platform:
alt.awt1=java.awt
alt.awt1.datatransfer=java.awt.datatransfer
alt.awt1.event=java.awt.event
alt.awt1.image=java.awt.image
alt.awt1.peer=java.awt.peer
alt.awt1.print=java.awt.print

AWT for Java 2 Platform:
alt.awt2=java.awt
alt.awt2.datatransfer=java.awt.datatransfer
alt.awt2.event=java.awt.event
alt.awt2.image=java.awt.image
alt.awt2.peer=java.awt.peer
alt.awt2.print=java.awt.print
:
etc.

The alt.* packages are not intended to fix all namespace mistakes. You are
correct. "alt.awt2.peer" does not make sense to enlightened folks. If we
were dealing with a brand new design, I might have used something like this:

alt.datatransfer2=java.awt.datatransfer
alt.event2=java.awt.event
alt.image2=java.awt.image
alt.peer2=java.awt.peer
alt.print2=java.awt.print

Event-driven programs, for example, are not the exclusive domain of visual
applications. We know this from the event-driven device drivers for JOS.
Therefore, the event package should not have been tied exclusively to the AWT.

The print package is obviously non-visual (right?). Since AWT components
can be used for printing, AWT has itself become something where visual is
optional.

Like you said earlier, we can't use "2" as a package name. And why not?
Because a javac tool refuses to compile any package in a directory called
"2". I tried to compile a package called "2" to see what might happen. It
did not work at all. Package names, like other identifiers, must start with
a letter (A-Z, a-z) or an underscore.

These classes seems to be part of the core:

java.lang.Object
java.lang.Class
java.lang.ClassLoader
java.lang.Throwable
java.lang.Exception
java.lang.RuntimeException

java.lang.String is part of the core because it is a final class. And yet,
it has changed in incompatible ways from one platform to the next.

We must not confuse "important" with "necessary". How do we determine if a
class is part of the core or not? Is it possible to run a virtual machine
without it.

There is nothing inherent in a virtual machine that requires a network. The
java.net.* package is not required by a virtual machine; it is not part of
the true core. Just as a command line (CL) program might never use
java.awt, many programs might never use java.net. The java.net package is
important to us as network application programmers. It is not important to
a stand-alone application.

Let me put it another way. Sun Microsystems could have easily implemented a
network package in javax.net and distributed it with every Java Virtual
Machine. A program that needs Sun Microsystems' network package could have
used javax.net.

We can implement network packages in alt.net0, alt.net1 and alt.net2 and
distribute these packages with every virtual machine. Any program that
needs network support can use alt.net0, alt.net1 and alt.net2.
