ArchitectureGroup

Article contributed by GilbertHerschberger (9 February 2000).

Introduction

After reading Linux Programming from Coriolis Group, I began to think about how we might compare and contrast Linux and JOS. Both are both operating systems. Both are open source and free license software. Both are developed from a geographically dispersed group of programmers on the Internet.

Documentation

More documentation is needed. Both jJOS and decaf development depend on Linux. We should make this fact more prominent on our website. Mac, Solaris and Windows are simply not a good choice for jJOS and decaf development today.

The Linux Documentation Project is open source and free license documentation. It is good example, worth immitating. The purpose of the Linux Documentation Project is superficially to document the Linux operating system software. It has deeper meaning. It demonstrates how open source and free license documentation contributes to the acceptance of radical new software.

Changes needed

The JOS Project is a great idea. Beware! A great idea is never enough. We must make day-to-day operations clear to our members. Clarity means success in a very competitive world.

Programmers need to know how they can extend and contribute to an operating system. We need to loosen the architecture to help everyone develop for JOS independently and in parallel.

This must be performed at three levels. The interface design is one level. The design of the interface between subsystems enables different subsystems to be developed independently. The subsystem implementation is another level. While subsystems share an interface design project, the implementation of a subsystem can be performed without the scheduling conflict from other teams. Providing public documentation is the third level. Documenting the purpose of a working interface helps others understand how much of the work has been done and what's left to do.

Documentation under GPL

Like the Linux Documentation Project (LDP), the JOS Project should work toward making all of its documentation available under the GNU Public License (GPL). For examples, authors would retain the copyright of their articles on JOS Wiki. And yet, articles could be distributed -- unmodified -- to anyone who wants them without royalties.

Executable and Library Format (ELF)

More information is needed to use shared libraries in ELF format within the jJOS kernel and decaf virtual machine. If the ELF format were known, a i386 version of jJOS could load ELF libraries created by standard Linux tools.

While the class file format is excellent choice for bytecode, the ELF format is supported by existing tools on Linux. If we had the time and expertise to build our own javac tool and extended programming language, we would be able to use the class file format for both bytecode and native code. Since we would like to continue using the gcc tool to compile C/C++ into the ELF format, it makes sense to add the ELF-related functions to the jJOS kernel.

The jJOS kernel should be able to open a shared library (like libdecaf.so), initialize the library for use within the kernel, invoke methods of the library, and close/unload the library when its is no longer used.

For example, I should be able to re-load libdecaf.so after recompiling it -- without leaving the jJOS kernel. A complete reboot should not be required reload a kernel module.

Make tool in Java

The make tool is yet another example of development in parallel. While the make is under development and testing, we can continue to use the native version on Linux. When the mostly bytecode version is available, we should switch from the native version to the mostly bytecode version.

Shell

I would like to use a mostly bytecode shell program. This program would be my default shell on all my Linux machines.

Shell scripts

A shell must work together with a shell scripts. The first line in a shell script must start with a pound sign (#). The name of the shell script follows the pound sign. The remainder of the shell script is fed to the shell (almost) as if it was typed from the command line.

Shell scripts are an important part of an operating system. A shell must invoke a shell script when its name is entered at the command line. Like shell scripts on Linux, the parameters to the script must be shifted to the next highest index.

People are needed

People are needed to write all of the applications that run on top of the decaf virtual machine. More people are needed to write all of the classes in the standard class library. Even more people are needed to write the device drivers for jJOS so that all hardware devices are compatible with this operating system.

Members of the JOS Project would like the opportunity to contribute. From a list of all the things that we need, people can volunteer for the things they want to do. It will all get done eventually.

Volunteers?

Anyone who wants to help with these subprojects should volunteer. Volunteers are needed to download and re-compile these classes on their machine (to test the distribution). Volunteers are needed to find bugs, submit fixes, and ask questions. Volunteers are also needed to review articles written on JOS Wiki (and elsewhere). If you'd like more information about volunteering, send me an e-mail.

    I'm continuing to work on Smart API. In fact, I just invented a mechanism to enable a pre-processor to work with the javac tool. It is a new scheme (of course!) for the Smart API.

java.lang.InputStream

    This scheme is always given a class name, using a dot separator. The scheme returns an instance of java.io.InputStream to access the source code for the class.

    In the following example, a data input stream is opened for codebase/java/lang/String.java. 

      public void example() {
	 String s = "in:data:source:java.lang.String";
	 URI uri = new URI( s );
	 DataInputStream input =
		  (DataInputStream) uri.getObject();
	 if ( input == null ) {
		return;
	 }

	 for (String line;;) {
		try {
		  line = input.readLine();
		}
		catch( IOException e ) {
		  return;
		}

		if ( line == null ) {
		  break;
		}

		:
	 }

	 try {
		input.close();
	 }
	 catch( IOException e ) {
	 }
  }

    Based on the current configuration of SourceController, the source code might be filtered through zero or more pre-processors. When a javac tool needs the source code for a class, it should consistently use the source: subscheme.

    Where is the source code stored? Anywhere! The definition of codebase and the current configuration of SourceController determines where the source code is stored.

    • Using the http: scheme, a class cam be compiled directly off the network (or Internet).

    • Using the in:zip: subscheme, a class can be compiled directly out of an archive.

    • Using an custom SQL subscheme, a class can be compiled from source stored in an SQL service.

    • Anywhere you can think of in the future...

  • Pure reflection

    I'm continuing to improve the mechanism of pure reflection. I have been working on classed called Opcode and Operand. Reflecting the "Exceptions" attribute enables a class definition file to contain the list of exceptions that might be thrown by a method.
