ArchitectureGroup

Article contributed by GilbertHerschberger.

Introduction

A goal of programming is to make things work. Somewhere in the tradition of programming, programmers are supposed to make things work.

All programmers, beginner and expert, must be able to see the source code in order to make things work. Programmers must see the source code to explain a computer's behavior. Without the corresponding source code, there's no alternative.

All Programmers working in-the-large need to distribute and debug. Weakness in debugging is one reason it has taken so long to gather support for distributed computing. Debugging is a weakness for CORBA/IIOP.

The source code for every Java class must be retained and not casually discarded. For as long as a .class file is used or stored anywhere, its source code must be retained.

Example

Debugging in a distributed environment is more difficult than debugging in a single-machine, single-process environment. For example, sometime during the summer of 1988 I was asked to fix an "insignificant" defect in a production program. It took three months to fix a defect that should have taken a few days. But there was only one thing missing: the matching source code. I was asked to start with the latest source code. It had the latest dates. It didn't take long to realize something was wrong. The latest source code wouldn't compile. It had typos. It called methods that hadn't been written yet. The source code I had was for a future release of the application. Source code for the current release was no where to be found.

Years later, it was a new company, new people, new technology and the same problem. During the summer of 1996, I was asked to figure out why data transfer stopped working. What should have taken a few days could not be done at all. Everyone believed that the source code to the custom program was on a clearly marked diskette. Did anyone bother to check? No. The diskette was empty.

Protect your investment

When you invest your time heavily to write source code in Java, don't casually throw it away. Everyone who reuses the same package names for new versions makes it impossible to see the source code that's responsible for the computer's behavior.

Sure, you can serialize the finished .class file. But, how are you going to fix the .class file if you can't find the source code? Debuggers don't serialize the source code along with the .class file. Debuggers depend on namespace to associate a .class with its source.

When reusing a directory for a new release, the debugger opens the new release source code. It blissfully points to blank lines of the new source. The debugging information in a .class file says that the source code is on that line.

Java has been written for programming-in-the-large. Reusing a package name and its corresponding directory doesn't not translate into programming-in-the-large.

You can't get away with it. Sooner or later, you will find yourself unable to make it work. Why? Because you can't see the source code for three versions of a .class file.

Simple mistake

As an architect, I have been aware of this aspect of programming-in-the-large for 10+ years. I'm sad to see how such a simple thing could overtake and undermine Sun Microsystems and break the promise of Write Once. Their idea of versioning won't solve the problem of distribute AND debug.

Build first

The first step in a maintenance project is always the same. It cannot be skipped. It cannot be rushed. There is no known short-cut. As the first step, a programmer must completely rebuild the product from the supplied source code. Nine times out of ten, a programmer cannot rebuild the product for one reason or another.

Does a product and its source code match perfectly? It is always better to find out before making changes than after.

Problems may come from the programmer's machine. It is much different over time than the environment used to build the product in the first place.

Problems may come from selecting the wrong source code to build a product. It is surprising to find how many lines of code are modified after the "final" version is stored.

For a daemon or CORBA service, this requires a programmer to run the software on a network and make certain that it continues to have the defect(s), which they are supposed to fix. For otherwise, how will they know when it has been fixed properly?

-- GilbertHerschberger (18 December 2000)
