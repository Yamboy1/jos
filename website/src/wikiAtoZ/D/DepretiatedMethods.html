ArchitectureGroup; DeprecatedMethods

Article contributed by GilbertHerschberger.

Introduction

A deprecated method is a method that shouldn't be used any more. If it shouldn't be used any more, why do we have to include it?

For more information on the implications of deprecated methods from Michael Daconta, see also Java...From the Trenches.

When writing a clean-room implementation of the Java 2 Platform, is it necessary to write all of the depreciated methods, so that they don't work in the way they are not supposed to?

When writing a clean-room implementation of the Java 2 Platform, is it necessary to include all of the bugs, too? When programmers count on the defects when they write applications, they are going to be disappointed when the defects aren't there, like they're supposed to be.

The reshape()/setBounds() method for a java.lang.Frame didn't work the way it should in Microsoft's implementation of the JVM. As a work around, a lot of programmers put extra statements in their code to "fix" it. What happens when the "fix" runs in a JVM that works the way it should? What then?

Make things better

It doesn't make sense to me to work so hard on a project without trying to make things better. On the other hand, a clean-room implementation of the Java 2 Platform must implement everything in the standard. It doesn't matter whether I like it or not. It doesn't matter if I agree with it or not. If I have made a commitment to implement the standard, that is the end of it.

I want to build a masterpiece. I can't build bugs into a masterpiece. I can't put bad design into a masterpiece. Just because I agreed to use the Java 1 Platform does not mean I automatically agreed to use everything else produced by Sun Microsystems. Sun Microsystems has proven how little they know about real-world object-oriented standards. Why should I follow their latest API if it doesn't give me what I'm looking for?

Do I have an obligation to use new releases of Java no matter how bad they are?

Needlessly complex

Sun Microsystems depends upon depreciated methods to evolve the Java API. This is bad, very bad. It makes Java needlessly complex. It makes it hard to write stable software.

I switched to Java, in part, because Java was so simple. Simplicity is a strong selling point. I want to keep it simple no matter how big it grows. Yes, it can be done. Java already points the way.

Sun wanted to make the latest and greatest APIs available on the Java 2 Platform and only Java 2 Platform. The market applied tremendous pressure to make new APIs available for Java 1.1. There's no technical reason why SQL, JINI, JNI, etc. isn't available for Java 1.0 API.

By pushing all of the pieces together, Sun can't distinguish between the Java 1.1 JVM and the Java 1.1 API. These are entirely different things. They have entirely different markets and technical implications.

Universal virtual machine

I know what Sun should do. They should create a new name for the JVM. It should not be so closely related to the Java language or the Java API. Although it would be basically the same virtual machine running, a new name would let everyone know that BASIC, C, COBOL, Pascal, and lots of other languages run on the virtual machine. There's a market out there for a universal virtual machine. What Sun calls the JVM could fill that market.

API add-on kit

Plus, Sun should create a new concept for the Java APIs. They should offer an API kit, for example, to the JOS organization so that Sun's API runs on the JOS JVM. And why not? When Sun offers a plug-in to an HTML browser, it's something like that already.

Distributed computing

Depreciated methods are fatal to distributed computing. Why should a distributed component support depreciated methods? If a method is not supposed to be used, don't use it. Ever. Don't enable the method at runtime if it isn't available at compile-time.

Sun isn't playing fair. They will be remembered for not playing fair. And, here we go again.
