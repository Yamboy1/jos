Article contributed by GilbertHerschberger (9 August 2001).

Introduction

I have been writing Java source code for more than three years. I write Java source code in a specialized Java development environment.

My Java development environment does not resemble my C/C++ development environment. While a typical Java compiler enforces a naming convention for its source files, C/C++ does not. While we can safely assume that all Java classes are part of some package (or namespace), namespace is a relatively new concept for C/C++. While every Java class is equivalent to a shared library, functions in C and classes in C++ can be statically linked to an executable, a local library (statically linked) or a shared library (dynamically linked).

Division

I actively develop with both JDK 1.02 and JDK 1.1.6. Physically, I only have two directory trees for all my original Java source code. My java0 directory contains all of the Java source code which is intended to be compatible with JDK 1.02 and compiled with its compiler. My java1 directory contains all source code which is intended to be compatible with JDK 1.1.6 and compiled with its compiler.

Physically, I make little distinction between "products", where a product is a collection of Java packages. All my JDK 1.02 products are originally developed in the java0 directory, using a collection of JDK 1.02-compatible tools. All my JDK 1.1.6 products are originally developed in the java1 directory, using a collection of JDK 1.1.6-compatible tools.

As a goal, I try never to use the same package name twice. By putting all of my source code into two directory trees, I am less likely to "invent" the same package name in more than one product. Since I use two directory trees it is possible, but rare.

I have been unable to reduce my development environment down to one directory tree. Some of my JDK 1.1.6-compatible tools, like JBuilder 2, don't get along with my JDK 1.02 source code. In its attempt to be helpful, JBuilder 2 mistakenly attempts to re-compile any Java source code which has not been compiled with JBuilder 2. This might be a welcome feature if all of your Java source code is supposed to be compiled with JBuilder 2. Of course, I found out the hard way as I watched in horror as JBuilder 2 overwrote all of my old classes!

Reuse

Because all of my Java classes are found in only two directory trees, I am able use one batch file to start all of my Java applications. The browser1b.bat file adds java0 and java1 to the classpath parameter of the java tool and start the UniversalBrowser. From the Universal Browser, I can run any Java application, any Java applet that I write.

The java tool has a 30 year old, UNIX-style command line interface. It is as non-interactive as it can be. I write lots of little Java applications I like to call utilities. The browser1b.bat command hides the usual complexity, automatically linking my Java application with a long list of jars installed anywhere on my computer.

Any time I don't waste on writing a custom batch file is time I can spend developing Java code. Any time I don't waste on building an "executable" jar is time I can spend elswhere, such as writing documentation and test programs.

Products

I don't want to distribute every Java package with every product. Here is how I build a product. Elsewhere, I have a series of product-specific build directories. For each product, I maintain a list of required Java packages. A package copier utility, written in Java, reads this list of required Java packages. It copies Java source code from a development directory tree into a build directory tree. For efficiency, it only copies .java files which have changed.

The source code in a build directory can be discarded. The original is in a development directory tree.

Source code is compiled into class files in both the development directory tree and re-compiled in one or more build directory trees. The class files can be discarded, too. The original class is in the development directory tree.

  • Recompiling the source code in a build directory enables me to determine if the product will compile cleanly using compilers from other toolkits, without disturbing my ability to run my own programs.
  • Recompiling in a build directory enables me to confirm that all packages used directly by an application have been identified. It shows me exactly what a product depends on. Some products depend on other packages. These packages may be distributed as other source code or other jars. When a product depend on other jars, I consistently use a "lib" directory for these.
  • Recompiling in a build directory enables me to double-check the performance of an application outside my original development environment. It is one step closer to the environment of my customer.

Benefits

When I fix a defect in a package, one or more products can be affected. Instead of having to fix a defect in more than one source file, I am able to fix the defect in a single source file. It is automatically copied to a build directory the next time I touch a product.

It is important for me to control the release of products based on the need to release that product. I don't like to release a product arbitrarily, such as when experimenting with a new mechanim in some other product. A product should be released when its release is beneficial to its customers.

About CVS

None of this is reflected in CJOSCvsOnSourceForge. Instead, CVS mistakenly reflects the build directories. It reflects the state of a product after the source code has been copied from my development directory.

I hesitate to change the way the Java source code is stored in CVS because other developers don't appreciate some of the tools I wrote and have come to depend on. The CVS repository is provided so that others can download the source code for a specific product. They should be able to compile it using off-the-shelf tools.

The build directories in CVS are awkward for additional development. The development directories, not yet found in CVS are intended for additional development.

I write open source and closed source software. I use the same technique wherever I go. I have a javaX directory on a customer's computer when I work on-site. I am convinced that this technique helps me write more working Java source code in less time. It works for me.
