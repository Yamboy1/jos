Article contributed by GilbertHerschberger.

When all is said and done, we all want one. Customers, suppliers, architects and programmers want one. We want an mature API that has it all: stability, compatibility, reliability and high quality. We want an API that we can count on, build on and make advanced applications with little effort.

An API is an application programming interface. Like an interface in Java, an application programming interface is an interface only, not an implementation.

Where are we now? After designing and using so many APIs, I thought it might be helpful to describe here how an API matures naturally. Sometimes it helps to think about where the mature API comes from.

When does it become mature? Just as a final draft is not the first step when writing, maturity is not step one. There is no way to go from zero to a mature API in a single step. A mature API cannot be built once, then used forever without change. The point of creating drafts is to use it, to watch it grow and see what's missing.

A mature API comes from the cycle of using it, fixing it, thinking about it. A mature API comes from growing it naturally based on experience. You shouldn't trust an API that you have never used. It is impossible to know its strengths and weaknesses in advance. And if you never use it, it is impossible to improve it.

The growth of a mature API slows over time. A useful API grows to maturity and keeps on growing.

For each production class, I build a test class to push it to its limits. ExtremeProgramming suggests using a class as soon as possible after it has been designed, including writing test programs. The test program provides artificial experience in the use of an API. Under the right conditions, this experience can be translated to real world experience of application programming in the trenches. Typically, the art of writing test programs are so far away from the real world because we can forget that this experience is artificial. Test programs only work in the laboratory.

If you really want a mature API, don't destroy your work. You cannot afford to destroy your work every time you get a new idea. High quality version control means creating a new namespace to work with for each new design. I said it once and I'll say it again, keeping the design that failed may be far more important than keeping the design that succeeded. The best experience comes from working through a failure so that you know exactly what went wrong and how it happened.

If you want a really mature API, you must be able to run every version of the API, side-by-side on a single machine, to compare and contrast the different designs. And for all those architects, designers and programmers that follow, keeping every version of the API in runnable condition allows others to see what works and what doesn't. To boost your confidence, it doesn't hurt to look at the API you created three months ago and see how immature it was compared to what you're using now.

Since it is strongly recommended to use the API early and often, you can use an API in your application program as soon as it is functional. You should put all improvements of the API inside a working application program, queueing improvements towards the next release of the API sometime.

If you want to develop a API to maturity, don't move all of your ideas directly to the API. Work with your ideas in a real applications (test applications don't count) and see if the idea turns out to be as great as you originally thought.

If you want a mature API, don't look backward with regret. There will always be a large hole in the API you hadn't thought about. When you find a hole (and you will), look at it as an opportunity to design and build the next release of the API without that hole. Don't look at it with regret, as if it were a failure. Crying about some missing feature is like crying over spilt milk. Learn from it and go forward.

If you want a iron-clad, top-quality, field-tested API, you have to use it in real world applications. Using an API is what exposes all of details on bad design and/or bad implementation. The architect and designer should use it to build applications, too.

Let's fast forward a little bit. If you want a backward compatible API, you must intentionally create different versions of the API, use them in real world applications and see what happens. The question is not "Are they backward compatible?" but "Can they be made compatible?"

An API must be designed with backward compatibility in mind to give backward compatibility a chance. When designing an API for backward compatibility, there's no guarantee that it will be compatible with your future designs. Great design keeps a window of opportunity open as long as possible. It improves the odds.

For as long as an API continues to be theoretical rather than usable, its chance of success erodes. How can a programmer learn to appreciate an API they can't use? How can a company standardize on an API they can't put into their products? How can a customer use advanced applications based on an API that's not finished yet? When the promised API never materializes, don't be surprised when your customer's patience has worn thin.

There is a world of difference between a mature API and a perfect API. A perfect API has no holes. It is difficult, if not impossible, to create a perfect API. Nobody wants a perfect API anyway. People don't want an architect to work alone in her laboratory for ten years before they get their first look at her design. A perfect API is usually too little, too late.

When I build an API, I follow my own advice. In the beginning, it was hard to tell the difference between my approach and everybody else's. It was kind of if you've seen one API, you've seen them all. It becomes easier to see the difference after using the fourth release of an API.

  • All of the old applications keep on running. Since the old API (interface design) is never changed after it is released, old applications continue to work predictably. Bugs are fixed, of course, under strict guidelines for side-effects and compatibility. So few software defects are simple bugs after all. Most require design changes.

  • Applications are upgraded individually. Applications are individually and intentionally moved from one release of the API to the next, leap-frogging interim releases of the API when possible. Changing the API foundation requires a complete regression test. Who has time to do that every couple of days? I have better things to do. Applications are upgraded as demand and time allows.
  • New designs are free to innovate. With an all new namespace, there is no assumption about the names of classes or methods in each new release of the API. Having said that, a mature API seldom needs drastic name changes.
  • There are no conflicts, like designing around depreciated methods. If programmers are not supposed to use a method after all, the method is deleted completely from the new API. No problem. If a method is supposed to return an int instead of a short, it is fixed. No problem. If a method turns out to be a really bad idea in retrospect, it is dropped. No problem.
  • There are no surprises, like unpredictable run-time errors. Upgrading an application becomes an explicit business decision. The focus is shifted away from upgrading the API to upgrading each application. That is when it is appropriate to perform regression testing on the application and fit everything into the application upgrade project's schedule. You can ask, "Do we have time for this?"
  • There are no application upgrade interdependencies. Upgrading an API just because of one application doesn't break all of the other applications. In the real world, there are many applications, developed at different times, at different releases of the API. Everybody wants a stable API because replacing an API at the end of a project makes the project longer.
  • Completely backward compatible is guaranteed. Backward compatibility is guaranteed by distributing every release (original and unchanged) when distributing the latest release. It sounds excessive, but, it works. Other methods may sound more reasonable, but, they don't work.

When building a new application, programmers must be free to use the most appropriate API possible. The latest and greatest API may not be the best API for the job. The latest API might be too big. The latest and greatest API might be perfectly optimized for something else.

As a disaster waiting to happen, it is irresponsible to upgrade an application to the latest and greatest API simultaneous to making a minor bug fix. It makes more sense to pick your battles. I like to pick the battles I'm going to win.

It doesn't seem possible for an application programmer to get their work done and stay up-to-speed on all APIs all the time. Making the programmer adapt to the new API under deadline pressure might not be the best business decision.
