Introduction

I have been thinking about this for a while. This is my first opportunity to write it down. It seems to me that interfaces in Java can have public static final fields. This is useful for declaring constants.

Warning: final is not constant.

The final keyword in Java is often misunderstood. It is most often misunderstood by C/C++ programmers. While it is often compared to the const keyword in C/C++, the final keyword does not have any of its implications.

public interface ExampleWithList {
  public static final Hashtable list = new Hashtable();
}

Once a variable is declared final, a program is no longer able to assign it. Only an object reference is affected. The assignment statement (=) cannot be used on a final variable. In other words, the following statements are disallowed:

  public void nonexample1() {
	 ExampleWithList.list = null; // disallowed
  }

  public void nonexample2() {
	 ExampleWithList.list = new Hashtable(); // disallowed
  }

Accessor

A final variable is expected to support accessors; and, it does. This works:

  public void example1() {
	 Object o = ExampleWithList.list.get( "abc" ); // allowed
  }

Modifier

A const object is expected to disallow modifiers. Some programmers expect a final object to disallow modifiers, too. But, it doesn't. Final variables are not constant. Are you surprised? Any modifier can change the object itself.

  public void example2() {
	 ExampleWithList.list.removeAll(); // allowed.
  }

  public void example3() {
	 ExampleWithList.list.put( "abc", "xyz" ); // allowed
  }

In Java, there is no exact equivalent to const from C/C++. In C/C++, the const keyword allows only accessors (const methods) when a reference is const. It disallows modifiers. Java does not support const methods, and therefore, allows both accessors and modifiers when a reference is final.

Inter-applet communication

If a browser reuses the same class loader to load two or more applets from your website, an interface can be used for inter-applet communication. More than one applet can add data to a common final object, such as a Hashtable in the example shown above.

A browser is not required to reuse a class loader; but, many of them do. A browser can finalize a class loader between pages, so the data collected this way is not reliable from one page to the next.

Summary

Like classes, interfaces can have static fields, too. When an interface class is loaded by two different class loaders, there are two instances of its static fields.
