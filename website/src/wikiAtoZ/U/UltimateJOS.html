Article contributed by GilbertHerschberger (23 March 2000).

Introduction

The ultimate expression of JOS is an operating system that requires no other operating system at runtime. This article provides more information about what the ultimate architecture might look like when we're done.

Ultimately, the operating system will invoke programs, applications, services, and devices from bytecode and only bytecode. Bytecode is the native code. Outside the kernel and virtual machine, machine code is missing.

Sample

Here is a sample of the start-up sequence for jJOS/decaf.

  • Etherboot or GRUB loads the jJOS/decaf build for a specific machine. A build includes a platform-specific version of jJOS and a platform-independent version of decaf.

  • jJOS loads decaf.

  • decaf loads the system process, a primary domain.

  • Coming soon: The system process will launch a user process through the Runtime.exec() method of the Java standard class libraries. Each user process is a secondary domain.

Where we are

Today, the closest thing we have to the ultimate JOS is the I386Build. Through Etherboot or GRUB, jJOS loads decaf. Then, decaf runs a system process.

The jjos tool runs on the Linux operating system. This is the HostBuild. The architecture of a host build does not represent the ultimate version of JOS. A host build is only for the purpose of transition. We need a stable host platform while we are developing the ultimate expression of JOS.

As more tools are rewritten in bytecode, fewer tools from the Linux operating system will be required to run JOS.

Reuse

Extreme software reuse is a very powerful strategy. We can reuse the software created by all Java programmers. Many non-members are working on JOS and have no idea how much they are contributing to it.

JOS is a host for Java. Therefore, every pure bytecode Java application is a JOS application.

We don't have to re-write every Java application to run it on JOS. We only need a standard class library that takes full advantage of our architecture. All those class files can be interpreted by our virtual machines. Java applications need a standard class library and they'll find one on JOS.

23 March 2000

Look at the virtual memory manager. A virtual memory manager needs a pre-existing file subsystem. A virtual memory manager reads and writes to a swap file/partition. Without a file subsystem, the virtual memory manager cannot be written. And yet, a virtual memory service can be hosted on Linux or Windows. It is possible to implement a swap file across TCP/IP.

Look at the standard class library. A standard class library needs to implement all of the methods from the Java Language Specification and/or Java Virtual Machine Specification. Our primordial class loader can be different. It does not have to use a file subsystem. It could use a device manager.

Look at a file subsystem. And yet, a file subsystem service can be hosted on Linux or Windows. It is possible to implement a file subsystem across TCP/IP.

Look at the Java Experimental Protocol Stack, or JEPS. These classes can download package files across a TCP/IP connection.

Look at the object reference. The operating system needs to generate unique object references for connections between low-level and high-level devices. While a low-level device is owned by the system process and under the care of the system process' garbage collector, its methods can be invoked by objects in a user process through inter-process communication. Each low-level device has a unique object reference that does not change for the lifetime of the system process. This is how you cross the persistence barrier.

Look at the device: scheme. The start-up sequence can create a device factory. What does a device look like? A "device" is an instance of any class. Any class can be turned into a device and managed by the device manager.

The ultimate JOS is closer than ever before, now that there is a binary distribution of the system we have so far. -- GilbertHerschberger (23 March 2000).
