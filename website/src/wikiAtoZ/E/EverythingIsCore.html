Article contributed by GilbertHerschberger (8 September 1999).

Introduction

Everything is core? But, everything can't be core.

Core and non-core is the separation of critical and non-critical, necessary and unnecessary, important and unimportant. It is the difference between essential (won't work without it) and non-essential (optional). It determines our priorities.

Every platform (and operating system) has a core. Each Java(tm) Platform has a core. JOS(tm) has, or will have, a core. This article considers the implications of adding all class libraries to the core of the Java platform.

The definition of core hasn't changed. The effects of core haven't changed. It has many implications. As technicians, the core affects everything we do.

Classic core test

Delete it and see what happens. This is a classic core test. Either something is core, or it's not. If your platform starts without it, it must not be core.

If you're careful, you can perform a core test on a copy of the rt.jar (or classes.zip). Delete a package and see what happens. If your JVM starts without it, it must not be core.

  • java.lang

    This package is core because the JVM won't start without java.lang.Thread. And besides, a package is the smallest unit of distribution. If one class in a package is required, the whole package is required.

  • java.net

    This package might be core because some JVMs won't start without it. And besides, there are differences in the definition of core from one JVM to the next. What is core for one JVM might not be core for another.

  • javax.servlet

    This package is not core because a JVM works without it. And besides, this is a great test case for a core test. In spite of the fact that your servlets won't run without this package, the JVM does not depend on this class library. It is not core.

  • java.sql

    This package must not be core because a JVM works fine without it. Delete this package and nothing bad happens. In spite of the fact that some of your applications won't run without this package, the JVM does not depend on this class library. The JDBC class library could have been called javax.jdbc as far as the JVM is concerned. It is not core.

  • javax.swing

    This package is not core because a JVM works without it. Moving it to java.swing doesn't change the fact that it is not core. In spite of the fact that some of your applications won't run without this package, it is not core. The Swing class library could have been called com.sun.swing or javax.swing as far as the JVM is concerned.

  • java.applet

    This package must not be core because a JVM works fine without it. Delete the java.applet package and nothing bad happens. In spite of the fact that some applets won't run without this package, it is not core. The Applet class library could have been called javax.applet as far as the JVM is concerned.

  • The list goes on and on. Unfortunately, there are more classes in JDK 1.2 that are non-essential, optional and non-core.

Complexity

A platform must have complexity. There is no way around it. The core minimizes complexity. A clear definition of the core determines the minimum complexity of a platform for other people.

People resent needless, or arbitrary, complexity. It is a waste of time. Saying that something is core when it is not, introduces additional complexity. It makes it more difficult for other people to know what is needed and what is not. People need to know what it takes to run a minimally complex platform. After wasting precious time, people are angry that it took so long to find out.

Plan A

Sun Microsystems, along with their licensees, had a great plan for future expansion of the Java Platform. The JVM would stay the same, with no need to change it. Additional class libraries would be developed elsewhere, anywhere but the java.* packages.

The Java 0 Platform was celebrated for its simplicity. It provided a few standard class libraries (packages), in the style and elegance of C, not C++. It had half a dozen core packages and a great plan for future expansion. Java, the class libraries, would have a managed namespace.

This plan is a great plan. It would have worked. Unfortunately, with the introduction of the JDBC class library, this plan was quickly discarded.

Plan B

Out of the ashes of a great plan, another plan was created. It is Plan B. Unlike plan A, this plan is bad, very bad. The Java Platform would change frequently, without concern for compatibility. Even standard extensions are moved around. Additional class libraries would be developed exclusively inside the java.* packages and require hooks into a new JVM.

Where does Plan B comes from? It comes from a business plan designed to frustrate "competitors". Throwing everything into the core will make it more difficult for someone to clone the core libraries. Changes to the core will make it difficult for the competition to keep up.

Unfortunately, customers and competitors are indistinguishable in software. What frustrates the competition also frustrates customers. These tactics undermines principles of software reuse, eliminating the advantage of an object-oriented approach. Legitimate customers are victimized. Applications suddenly stop working. Customers should be outraged when they are asked to rewrite applications, when they don't have to.

What went wrong?

History

It all started with the need for SQL. SQL is a very popular tool for serious business applications. SQL might be critical to the success of Java as a business platform.

Originally, a serious business platform was not the highest priority for the Java 0 Platform. Expectedly, SQL was missing. While an built-in SQL class library might have been missing, nothing would prevent a client/server SQL application from using TCP/IP sockets.

Let's examine the history of the JDBC class library. A half-hearted attempt was made to make JDBC available on the Java 0 Platform. In fact, you can download and install a JDBC class library for JDK 1.0.2. What went wrong? Why was the JDBC class library available on Java 1, not Java 0?

JDBC is not a core package. There are many Java-based applications that never use the JDBC class library. There are many embedded devices and products, like HTML browsers, that should not use the JDBC class library.

JDBC is a clear violation of Plan A. Sun Microsystems put the JDBC class libraries in java.sql and java.math packages. JDBC should have been developed outside the java.* packages. As a standard extension, JDBC could have been distributed with JDBC-specific applications.

Sun Microsystems should have put JDBC in a single javax.jdbc package. Two separate packages are not required.

JDBC itself does not require a new JVM. There are no Java 1 Platform-specific bytecodes. When installed correctly, JDBC runs on the Java 0 Platform.

JDBC-ODBC bridge

Everyone agrees that the JDBC-ODBC bridge is critical to the transition from ODBC to JDBC. The JDBC-ODBC bridge is not equal to JDBC class library. The JDBC-ODBC bridge is a JDBC driver, found in the sun.jdbc.odbc package. The JDBC-ODBC bridge is not a core package.

Since the JDBC-ODBC bridge from Sun Microsystems depends on the Java Native Interface (JNI) for Java 1, it did not work on Java 0.

Other class libraries

Other class libraries, like RMI and Swing, followed the path of Plan B. They were added to the core of the Java 1 and Java 2 platforms.

Future

As a Java developer, the future of the Java platform is important to me. I think about what might happen if Plan B is used for the next fifteen years. Let's take a quick look at the Java 17 Platform.

  • The Java 17 Platform has integrated the custom class libraries from every software developer on the planet. The core is 286GB (that's gigabytes). It is so big that it won't fit on any existing disk drive. It can't be backed up.

  • The Java 17 Platform cannot be used for embedded devices. There is no ROM big enough to hold it.

  • The Java 17 Platform cannot be distributed across the Internet. Even with 100Mbps connections, it can't be downloaded. It would take 2,860 separate archives to download it one piece at a time.

  • The Java 17 Platform cannot be loaded on any personal computer. There is not enough RAM to hold it.

  • The core Java 17 Platform cannot be taught to programmers. It would require 14,000 separate volumes to document the Java API. It would take 30,000 technical documentors to write the text.

Why core

Java needs focus. Java needs a killer app. What can Java do that no other platform can do? The purpose of a core for the Java platform is to focus, concentrate on the class libraries that are essential to the platform. It provides focus for Java programmers and third-party vendors. It provides focus for refining the Java API and standardization. It provides focus for removing critical bugs first from class libraries.

Everything is core. When everything is core, it means that nothing is core. It is just as when everything has become your priority, nothing has priority.

When Java has no core, it has no focus.

Plan A-2

JOS(tm) should stick to the original plan. It is a great plan. It works. The JVM must stay the same. Additional class libraries would be developed elsewhere, anywhere but the java. and jos. packages.

My proposal is this. If we are going to rewrite JDBC for JOS(tm), we should do it right. Since JDBC is non-essential to the JOS JVM, we should write it inside a package called josx.jdbc.

With a source code tool, there is original source code and a few machine-generated copies. While we write a josx.jdbc package, we can compile both josx.jdbc and java.sql. While it is written once, it is compiled (at least) twice.

We should do a similar thing for Swing. We can have one copy of the source code for Swing and compile it to the javax.swing, java.swing and josx.swing packages. While it is written once, it is compiled often.

Compatible across platforms

When we put JDBC in josx.jdbc, it is compatible with all Java platforms. We can write JOS applications directly to the josx.jdbc package and ship the josx.jdbc package for use with JDK 1.0, 1.1, 1.2 and 1.3.

With this approach, josx.* packages are more reliable across all Java platforms. There is no code to port.
