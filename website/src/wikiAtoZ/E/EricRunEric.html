ERICProject

Article contributed by GilbertHerschberger (11 August 2001).

Introduction

One eric tool can run another. There are three very good reasons to do this.
  1. View standard output.
  2. Switch to an alternative codebase.
  3. Take a new eric tool for a test drive.

View standard output

The eric tool is available for Microsoft Windows. It is compiled as a Win32 GUI application. It is compiled in such a way that it is disconnects itself from a command line shell. When disconnected, standard output is redirected to a standard sink (or null device).

Within your Java application, the System.out and System.err objects is connected to standard output by default. Java programs may write informative messages to System.out and error messages to System.err. To see these messages, you could

  • use redirection at the command line, as in eric > ggg.txt, or
  • run one eric tool from within another, as in run:os:eric.exe. The UniversalBrowser enables you to enter a location using the run:os: scheme. Unlike redirection into a file, this technique enables you to easily see standard output while the program is still running.

Switching to an alternative codebase

The eric tool manages the codebase automatically. The codebase determines what classes are available to your virtual machine. Microsoft Windows mistakenly limits you to associate only one copy of the eric tool with the .eric file extension. Your default eric tool may not use the codebase required for a specific application.

Earlier today, I ran the Deric utility from an alternative codebase. My default eric tool is a non-developer version on this machine. It includes few development tools. Yet, I would still like to double-click on an ERIC Shortcut and run these tools.

My solution is this. My alternative eric tool includes lots of development tools. I start my default eric tool and use it to start an alternative eric tool. I can continue to put up with Windows and get what I want.

Taking a new eric tool for a test drive.

The eric tool has been thoughtfully designed so that multiple copies and multiple versions of the tool can coexist. This approach provides maximum margin for error. You do not have to overwrite the old tool during installation. You are not required to un-install your old tool to make room for a new one. You can choose to run a new version if it pleases you.

I cannot afford to be without my eric tool for very long. I have come to depend on it to get my work done. When I am experimenting with a new eric tool, I neither overwrite nor un-install the old one. Rather, I use the old tool to take a new eric tool for a test drive.

I use an old tool to display the standard output from the new tool. I can count on the old tool to do this properly. I can set up a desktop with a variety of locations to test each aspect of the new tool. Backward compatibility is very important to me. I use an old tool to help me perform regression testing.
