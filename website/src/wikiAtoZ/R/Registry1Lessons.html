RegistryPages

Article contributed by GilbertHerschberger (7 July 1999).

Introduction

There were good ideas and not so good ideas inside my registry API. Both good and bad ideas should be expected when building anything from the bottom up. This article is about the lessons learned (so far) from building my registry API.

It works. The registry factory really works. While it might not be as crisp as I'd like, it is functional. I am already using it in an application program. There are many benefits of starting at the bottom. Being able to use a package as soon as possible is just one of them.

The registry URL is spectacular. I like it a lot. Many parts of the regular Java API would benefit from a URL-based factory interface (like streams, AWT/Swing and zip). A fully URL-based operating system is the way to go!

If I had to do it all over again (and I do), I would do it a little differently. Knowing what I know now, some of the demands of the registry.Registry interface are a challenge.

While I can use my registry in lots of programs, it isn't as crisp as I expected. It is an API that needs to mature. During detailed design, for example, I got confused about where conversion should be. I didn't follow my own advice. While writing my specifications, I clearly stated that data conversion is not a responsibility of a registry object. It's true. I should have stuck to it. Concerns about conversion would complicate each implementation of a registry. While it can be done (I've done it), it isn't the most effective use of application programming time.

Future

A future registry API should build on the strength of the registries we already have. A registry design comes down to the following three layers (top-down):
registry.RegistryIterator
registry.RegistryNode registry.RegistryEntry
registry.Node..Helper registry.Entry..Helper

Registry Iterator

A registry iterator provides an interator for any registry, tracking a "current" registry node. All of the methods of RegistryNode are dispatched to the "current" registry node. This is very much like the proposed RegistryIterator.

public class RegistryIterator 
	 implements RegistryNode {
  public RegistryIterator( String url ) {
	 this( RegistryFactory.getRegistry( url ) );
  }
  public RegistryIterator( RegistryNode v ) {
	 ptop = v;
	 path = "";
	 pcurr = ptop; // or resolveCurrent();
  }
:
  public void setPath( String v ) {
	 path = v;
	 resolveCurrent();
  }
  public void setPathToRoot() {
	 path = "";
	 resolveCurrent();
  }
  public void setPathToParent() {
  :
  }
  public RegistryIterator getIterator() {
	 return new RegistryIterator( pcurr );
  }
  private String path;
  private RegistryNode ptop;
  private RegistryNode pcurr;
}

Registry Node

A registry node is an interface like registry.Registry, only simpler. It provides objects and nothing but objects to the registry cursor. It is a little more than an interface for java.util.Hashtable.

public interface RegistryNode {
  // accessors
  public Enumeration getKeys();
  public Enumeration getEntryKeys();
  public Enumeration getNodeKeys();
  public Enumeration getObjects();
  public Object getObject( String key );
  public Object getObjectFor( String key, String type );
  public String getType( String key );
  public boolean isKey( String key );
  public void saveAs( String url );
  public void copyTo( RegistryNode v );

  // modifiers
  public void createEntry( String key );
  public void createNode( String key );
  public void setObject( String key, Object v );
  public void removeAll();
  public void removeKey( String key );
}

Design notes:

  • This RegistryNode interface must not expose the use or non-use of RegistryEntry. An implementation may or may not use RegistryEntry to store its data.
  • A custom registry might discard the original source type inside setObject().
  • When conversion is part of RegistryWrappers, it is important for getObjectFor() to provide the name of target type.
  • createEntry() and createNode() make a registry easier to use. They must create a null entry and an empty registry node respectively.
  • Entries and nodes participate in a single namespace.
  • Keys are case sensitive.
  • The forward slash (/) is a special character and should not be used in a key. This character provides navigation within registry nodes.
    • An iterator can navigate directly to a specific registry node. For a registry node stored as "editor" inside another registry node called "programs", it is possible to set the path to "user" with setPath( "programs/editor" ).
    • setPathToParent() removes the last key in a path by searching for last forward slash.
  • The open and close braces ([]) are special characters and should not be used in a key. These characters provide navigation within arrays.
    • It is possible to set a current registry node to a registry node within a registry array. The key is in a form of name[index]. When a child node is stored as "name", it is possible to set the path to registry 25 with setPath( "name[25]" ).
    • It should be possible to get an individual item from an array. When a byte array is stored as "name", it should be possible to get byte 100 with getByte( "name[100]" ).

Node Helpers

A registry node can use node helpers--if it makes sense. A node helper will provide containers to preserve the original data type stored, even when a registry converts the data to another type for storage.

Registry Entry

A registry entry holds a single object. It provides all of the accessor (getXXX()) and modifier (setXXX()) methods that I want. The registry cursor returns registry entries and nothing but registry entries.

Entry Helpers

In order to perform all kinds of reasonable conversion from any type to any other type, there should be a series of entry helpers. An entry helper is selected based on the type of data stored. It returns the data for the type requested (if possible) without a registry entry having to know how the conversion was done.

public abstract class StringEntryHelper 
	 implements ItemAccessor, ArrayAccessor {
  public StringEntryHelper( Object v ) {
	 value = object2String( v );
  }
  public boolean getBoolean() {
	 int v = getInt();
	 if ( v == 0 ) {
		return false;
	 }
	 else {
		return true;
	 }
  }
  public byte getByte() {
	 return (byte) getInt();
  }
  public char getChar() {
	 return getString().charAt( 0 );
  }
  public short getShort() {
	 return (short) getInt();
  }
  public int getInt() {
	 // *A!

  public long getLong() {
	 // *A!
  }
  public float getFloat() {
	 return (float) getDouble();

  public double getDouble() {
