RegistryPages

Article contributed by GilbertHerschberger (22 July 1999).

Introduction

It isn't enough for your high quality application to store its configuration inside a registry. You must deal with registry redirection, too.

Alternatives

A high quality application should provide alternatives. An application might provide for themes, different collections of colors, sounds, images and optimization settings. Otherwise, your customers would be stuck with one configuration (not likely) or left on their own to swap configuration files manually (that' how it's done). Instead of ignoring turn-over and experimental configurations, a high quality application uses a registry to choose the current configuration.

Choice

Alternative configurations is a direct consequence of freedom of choice. When you start a program, you should be able to choose the configuration you want.

Named

How do you keep track of all these alternative configurations? We can continue to do what we do now. We provide a unique, human-readable name for each configuration. Only one configuration can be the default (no questions asked) at any point in time.

Start up

When your high quality application keeps track of all its alternative configurations, you might decide to let your customer decide which configuration to use at start up. Some applications are like that. What theme would you like to use today?

Or, you might want to start your application using the default, and then, offer a mechanism to switch to an alternative configuration.

Keeping track

You should use a registry to keep track of alternative configurations. Each alternative configuration can be given a name, a key into your top-level application registry.

If you choose to use the system-wide registry, the list of alternative configurations don't take up much space. But remember that everything you put in the system-wide registry is difficult to remove after your application has been deleted.

Example

Here is an example of a program that maintains three alternative configurations for its color scheme, with the blue theme as its current default. These configurations are stored in read-only configuration files. All three files are in the application's conf subdirectory.

red=conf/redTheme.conf
blue=conf/blueTheme.conf*
green=conf/greenTheme.conf

At start up, the application has to resolve this redirection. It has to load one registry in order to find out the name of another. When the short list of alternatives is stored, you provide a standard mechanism for choice.

Instead of hard-coding the name of your program configuration, design a redirector. You can hard-code the name of the redirector to maximize the flexibility for your customer.

Un-install

Your program should be self-contained. Code and configuration should be stored in a directory set aside for your program. There are good reasons to un-install a program. A high quality operating system uses the delete command to remove software from a machine. Once a program has been removed, there is no good reason to store its configuration.

On a poor quality operating system, your program can check out but it can never leave. It will leave behind its useless configuration in your system-wide registry.

On the other hand, all of your JOS applications should use the same registry API to read (and write) configuration data. It is the API that is critical, not a system-wide service.

For the OS and its security, it needs a system-wide registry service. The API to the system-wide registry can be identical to the interface used by all of your JOS applications, and should be.
