RegistryPages

Article contributed by GilbertHerschberger (21 July 1999).

Introduction

Property files (.properties) were made for configuration data. The registry API can easily read property files. This is a map from property files to the JOS registry.

File format

A property file is a plain ASCII text file. It is an otherwise binary file divided up into lines ending with CR or LF or CRLF. Each entry must be on a line by itself. It uses 8-bit encoding exclusively. There are only a few special characters:

  • CR This control character is used to mark the end of each line.
  • LF On some platforms, this control character is used redundantly to mark the end of a line.
  • = This character separates a name from a value.
  • # A single line comment start with this character.
  • This character is used for sub-registries.
  • TAB Every line in a sub-registry must start with this character.

Specifically, the single quote ('), double quote ("), and slash (/) are considered user-data.

  • blank line A blank line is ignored. It is just as a single line comment.
  • leading and trailing spaces Leading and trailing spaces are trimmed. The space character within a value is considered user-data, too.

Properties

A property is a name/value pair. The maximum length of a property name is unknown. A string is the only type of data value supported. In the standard property file, each property must fit on a single line.

  • name-1=value-1
  • name-2=value-2
  • name-3=value-3

Flat

The standard property file is flat. There is no mechanism for nested sub-registries. A sub-registry cannot define a sub-sub-registry.

There are no preprocessor and no preprocessor directives. One property file cannot automatically "include" another.

Comments

In the style of configuration (.conf) files, a property file supports single-line comments and blank lines. Single line comments and blank lines are ignore, not imported.

# single-line comment

Mapping to a registry

An property file can be imported into a registry like this: 

String url = "x-registry:file:demo1.properties";
Registry r = RegistryFactory.getRegistry( url );

When imported, a property file becomes a registry object. This is a top-level registry object. It may contain sub-registries, too.

When imported, each line becomes a registry entry. Characters before an equal sign (=) becomes a key. Characters after an equal sign become a value.

When value equals backslash ()

When the backslash character is used by itself as a value, a subregistries is created. The name of a subregistry is used as a key into the top-level registry object.

section-1=
<TAB>name-1=value-1
<TAB>name-2=value-2
<TAB>name-3=value-3

Each property line in a subregistry must start with a TAB (0x09) character.

Example 1

The following code demonstrates how to manually build a registry equivalent to this property file.

Property file (demo1.properties): 

#
# Demo 1 Properties
#

name-1=value-1
name-2=value-2
section-1=
	  name-3=value-3

The equivalent Java code: 

  public Registry example() {
	 Registry section1 = new BasicRegistry();
	 section1.setString( "name-3", "value-3" );

	 Registry property = new BasicRegistry();
	 property.setRegistry( "name-1", "value-1" );
	 property.setRegistry( "name-2", "value-2" );
	 property.setRegistry( "section-1", section1 );
	 return property;
  }

Once a registry has been created, either manually or by importing an property file, a program must be able to use the values stored. Here is an example of printing the imported property file: 
  public void printProperty( Registry property ) {
	 String[] list = property.getKeys();
	 int iMax = list.length;
	 for ( int i = 0; i < iMax; i++ ) {
		String key = list[ i ];
		String type = property.getType( key );
		if ( type.equals( "String" ) ) {
		  String value = property.getString( key );
		  System.out.println( key + "=" value );
		  continue;
		}
		if ( type.equals( "Registry" ) ) {
		  System.out.println( key + "=" );
		  printSection( property.getRegistry( key ) );
		}
	 }
  }

Registry URL example

It is important to understand where the pound sign (#) is used to separate the name of a registry from its contents. The pound size (#) is never passed to the property file import mechanism. The entire property file may be loaded by the subsystem. The pound sign is always processed after a registry has been loaded by a registry factory. Basically, the registry factory ignores all characters after a pound sign.

  • To view one section of a complex initialization file in a registry editor, use the pound sign between filename and section name: 
    x-registry:file:/usr/local/demo1.properties#section-1
  • To go directly to a specific property in a registry editor, use a slash (/) after the section name: 
    x-registry:file:/usr/local/demo1.properties#section-1/name-1
The propertiesregistry1a.PropertiesRegistryController provides seemless access to property files. The subprotocol for this controller is properties: but also recognises the .properties extension. 
x-registry:properties:file:/usr/local/example.properties
x-registry:file:/usr/local/example.properties
The properties: subprotocol is needed if you want to store a properties file without the .properties extension. You can store data in a .txt file and still read it into a registry: 
x-registry:properties:file:/usr/local/demo1.txt
The propertiesregistry1a package isn't ready yet; it might be available with release 1c. -- GilbertHerschberger (21 July 1999).
