RegistryPages

Article contributed by GilbertHerschberger (23 June 1999).

Introduction

The registry was made for configuration data. Since there is already a lot of configuration data stored in initialization (.ini) files, there must be a map from .ini to the JOS registry.

File format

An initialization file is a plain ASCII text file. It is an otherwise binary file divided up into lines ending with CR or LF or CRLF. Each entry must be on a line by itself. It uses 8-bit encoding exclusively. There are only a few special characters:

  • CR This control character is used to mark the end of each line.
  • LF On some platforms, this control character is used redundantly to mark the end of a line.
  • [
  • ]
  • =

Specifically, the single quote ('), double quote ("), slash (/) and backslash () are considered user-data.

  • blank line A blank line may be significant. A blank line may be required to mark the end of a section.
  • leading and trailing spaces Leading and trailing spaces may be significant. The space character may be considered user-data, too.

Sections

The standard initialization file is (apparently) divided up into sections. A section is a container for properties. The name of each section must be unique. The name of each property must be unique within a section. Sections participate in a section index.

Each section starts off with a section name. Here is how a section name is formed:

  • [section-name-1]

An opening brace ([) and closing brace (]) is required. The maximum length of a section name is unknown. Spaces in a section name are allowed. Section names are case sensitive.

Properties

A property is a name/value pair. The maximum length of a property name is unknown. A string is the only type of data value supported. In the standard .ini file, each property must fit on a single line.

  • name-1=value-1
  • name-2=value-2
  • name-3=value-3

Flat

The standard initialization file is flat. There is no mechanism for nested sections. Each section participates in a single unique section index.

There is no preprocessor and no preprocessor directives. One initialization file cannot automatically "include" another.

Mapping to a registry

An initialization file can be imported into a registry like this: 

String url = "x-registry:file:demo1.ini";
Registry r = RegistryFactory.getRegistry( url );

When imported, an initialization file becomes a registry object. This is a top-level registry object. It is expected to contain other registry objects.

When imported, each section becomes a registry object. The name of a section is used as a key into the top-level registry object.

When imported, each property is stored in a registry object for its section. The property name is its key; the property value is its value.

Example 1

The following code demonstrates how to manually build a registry equivalent to this .ini file.

The initialization file (demo1.ini): 

[section-1]
name-1=value-1
name-2=value-2
name-3=value-3

The equivalent Java code: 

  public Registry example() {
	 Registry section1 = new BasicRegistry();
	 section1.setString( "name-1", "value-1" );
	 section1.setString( "name-2", "value-2" );
	 section1.setString( "name-3", "value-3" );

	 Registry ini = new BasicRegistry();
	 ini.setRegistry( "section-1", section1 );
	 return ini;
  }

Example 2

Once a registry has been created, either manually or by importing an .ini file, a program must be able to use the values stored. Here is an example of printing the imported .ini file: 
  public void printINI( Registry ini ) {
	 String[] list = ini.getKeys();
	 int iMax = list.length;
	 for ( int i = 0; i < iMax; i++ ) {
		String key = list[ i ];
		System.out.println( "[" + key + "]" );
		printSection( ini.getRegistry( key ) );
		System.out.println( "" );
	 }
  }

Registry URL example

It is important to understand where the pound sign (#) is used to separate the name of a registry from its contents. The pound size (#) is never passed to the INI file import mechanism. The entire initialization file may be loaded by the subsystem. The pound sign is always processed after a registry has been loaded by a registry factory. Basically, the registry factory ignores all characters after a pound sign.

  • To view one section of a complex initialization file in a registry editor, use the pound sign between filename and section name: 
    x-registry:file:/usr/local/demo1.ini#section-1
  • To go directly to a specific property in a registry editor, use a slash (/) after the section name: 
    x-registry:file:/usr/local/demo1.ini#section-1/name-1
The iniregistry1a.INIRegistryController provides seemless access to my own format for initialization files. I'm pretty sure this format is non-standard. The subprotocol for this controller is ini: but also recognises the .ini extension. 
x-registry:ini:file:/usr/local/example.ini
x-registry:file:/usr/local/example.ini
The ini: subprotocol is needed if you want to store an initialization file without the .ini extension. You can store data in a .txt file and still read it into a registry: 
x-registry:ini:file:/usr/local/demo1.txt
The iniregistry1a package isn't ready yet; it will be available soon. -- GilbertHerschberger (6 July 1999).

In another book about Java, the author uses initialization files (.ini) for applets, servlets and applications. He consistently uses slash-slash (//) for single line comments. This is in the style of C/C++ and Java. Is this standard?

While mapping a full registry tree to initialization files, there is a slight problem with sub-sub-registries. A file maps nicely to a registry. The section maps nicely to a sub-registry. But what about sub-sub-registries (and sub-sub-sub-registries)?

I think these can be mapped to an initialization file by using the slash (/) inside a section name. 

[sub-registry/sub-sub-registry]
name-1=value-1
name-2=value-2
name-3=value-3

In fact, both a slash and brackets ([]) could be significant in a section name. We can store registry arrays in an initialization file. 

[sub-registry[0]]
name-1=value-1
name-2=value-2
name-3=value-3
[sub-registry[1]]
name-1=value-1
name-2=value-2
name-3=value-3
[sub-registry[2]]
name-1=value-1
name-2=value-2
name-3=value-3
-- GilbertHerschberger (16 July 1999).

According to a book called IRQ, DMA and I/O, the semicolon (;) is the Microsoft standard for single line comments. In order to write Microsoft-compatible (and thus JOS-compatible) initialization files, the semicolon should be used if single line comments are used at all. Comments are used sparingly in initialization files.

Think about this: The standard registry editor, provided with JOS of course, will be able to read (and write) all of the initialization files on your Windows 3.1/NT/95/98/2000 computers.

The first non-standard implementation of importing/exporting for initialization (.ini) files is available today. For more information, see also Release 1b in RegistryPages. -- GilbertHerschberger (21 July 1999).
