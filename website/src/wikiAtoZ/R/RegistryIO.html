RegistryPages

Article contributed by GilbertHerschberger (19 June 1999).

Introduction

I have been thinking a lot about problems with a persistent registry. A registry should be able to fit into the model of I/O stream. And yet, a registry might be based on RMI, CORBA or future technology.

Current direction

At this point, it makes more sense for a registry to exist apart from its persistent storage mechanism. I am looking for ways to provide an interface between a registry and all kinds of storage mechanisms.

There are so many different kinds of registries.

  • A transient registry must be explicitely stored. A transient registry can be copied to a file using the registry.RegistryInputStream and registry.RegistryOutputStream. Since both of these interfaces are part of the registry API, the conversion from transient registry to a file format is completely open. A transient registry can be stored in any file format.
  • On the other hand, a persistent registry might be used without ever being explicitely stored. It is stored anyway. Every change to the registry is written automatically and seemlessly to permanent storage.

Copy

When a registry object is a self-contained, self-sustaining object, it must manage its own memory. For primative methods, it is easy to see how a registry must store a copy of data inside its own structures. But for registries inside registries, this is a little harder to see.

The setRegistry() method makes a copy of a registry object and all of its child registry objects (and all of their child registry objects). At the end of each setRegistry(), a copy of a given registry is stored within another registry. A registry is not stored by reference.

It is easy to see how registry.RegistryInputStream and registry.RegistryOutputStream must make a copy of registry data when reading and saving from a stream. This self-contained registry allows us to copy data from one transient registry to another; from a transient registry to a persistent registry; from one persistent registry to another.

Adding

When you put data in a transient registry, you can add the transient registry to a persistent registry. The normal setRegistry() method makes a deep copy of all the data in a source registry. Both the source registry and a target registry exist. The source registry is useable by your program. The target registry is created inside another registry.

Example

Here is an example of the code you might use you want to create an object graph and store it inside the JOS system-wide registry. The city of Marietta, Georgia (USA) is the county seat of Cobb county.

  • USA (country)
    • Georgia (state)
      • Cobb (county)
        • Marietta (city)

Here's the code.

  public void example() {
	 Registry city = new TransientRegistry();
	 city.setInt( "example", 1000 );

	 Registry county = new TransientRegistry();
	 county.setRegistry( "Marietta", city );

	 Registry state = new TransientRegistry();
	 state.setRegistry( "Cobb", county );

	 Registry country = new TransientRegistry();
	 country.setRegistry( "Georgia", state );

	 Registry globe = SystemRegistryFactory.getSystemRegistry();
	 globe.setRegistry( "USA", country );
  }

Notice how there is only a single call to store the graph inside a system-wide registry. When a system-wide registry is distributed, this approach is much more efficient. Of course, this is only one of hundreds of different ways to do this.

Framework

There are many file formats capable of storing registry data. In order to connect a registry with one of its possible file formats, registry.RegistryInputStream and registry.RegistryOutputStream must be implemented. And yet, these two object are used together. It may be possible to change the file format of a persistent registry while it's running. When a new registry I/O stream is assigned to a persistent registry, it could automatically and seemlessly make a copy of itself before closing its original streams.

While persistent registries could have non-standard ways of keeping their data persistent, we should build a framework to allow any implementation of a persistent registry to use any implementation of registry I/O streams.
