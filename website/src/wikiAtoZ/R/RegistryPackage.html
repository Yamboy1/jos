RegistryPages

Article contributed by GilbertHerschberger (9 June 1999).

Introduction

Sometimes you might need permission to do the right thing. You have all of our permission to build experimental APIs using the right naming convention. What is the right nameing convention? You must mind your own packages. You must put your packages in your own namespace, neither in jos nor org.jos.

We must build on strength. A large namespace is one of the most important strengths of Java. We must take advantage of the large namespace as we develop JOS over the long haul.

We must think big. When Sun Microsystems finds out about the JOS registry, they will build a registry API, too. In an open project, you expect others to use our best ideas. Hopefully, they won't take our worst ideas. And neither should we.

In many Java projects today, development process is exactly backward. The default development process encourages us to keep the worst and throw away the best. That's not healthy. That's not what we want. We need to think about this. Sun Microsystems defined this development process. It doesn't work for them, so why should we use it?

Naming convention

We should find a naming convention that works for everyone. The current naming convention only works for a few. We should think big. Everyone that volunteers their time to refine and promote JOS and Java should have their own space.

We can get along better by choosing a fair naming convention. The current JOS naming convention is not fair to either the individual or the community. The recommended process is broken. It doesn't work. It can't work. Why? Because the first person to lay claim on an official package wins at the expense of everyone else. That is not how a mature API is developed. For more information on a mature API, see also MatureAPI.

It is unfair to the individual for a number of reasons. An individual must be able to challenge a proposal at any time. An individual must be able to create a counter-proposal. An individual should be able to use a candidate standard if they choose to. An individual should be aware of the risks and trade-offs. An individual should not have to wait, keeping all their projects on hold, until a candidate standard becomes an official standard. Using a candidate standard is how the Internet was made. In fact, standards must be implemented before they can be pushed through the standardization process.

It is unfair to the community for a number of reasons. A community must benefit from its best ideas. We must be able to pick the best-of-breed among APIs and implementations. We should not pick the first idea. We should not suffer under the weight of a first draft API. A first draft API is rarely the last word on a subject. A community must have a choice among many different APIs. A community that undermines competition will always suffers from mediocrity. Maximize the discussion of different perspectives.

The best API today surely won't be the best API tomorrow. A community that promotes the best of the best will be prosperous and thrive. A community that embraces mediocrity will die.

The JOS naming convention doesn't have to be fair. We could choose to mangle the JOS namespace in the same way other operating systems do. Namespace is one of the reasons why I don't like other operating systems. The Microsoft Windows namespace for DLLs only works if you work at Microsoft. The Java namespace only works if you work at Sun Microsystems.

Important lesson

This issue was raised with the development of multiple (and currently incompatible) registries. There must be multiple implementations of both the registry API and registry-related classes.

We should learn this lesson early in the development of the JOS API and its related packages. We can learn the lesson now when it is less expensive, or wait to learn it down the road. It gets more expensive every day that goes by.

How it should work

First, here is the reality. I have a registry package. You have a registry package. Three more people each quarter will have their registry package. When Sun Microsystems figures out that a registry package is a good idea (and it is), they will build a registry package in their java.* hiearchy somewhere.

In order to support diversity, each registry API should be named for what it really is: a proposal.

  • My registry API should be called com.realignsoft.registry1a because that's what it is. It is one of many different registry APIs that I'm planning.
  • Your registry API should be called your-name-here.registry.. because that's what it is. It is one of many registry proposals you are planning.
  • Sun Microsystems registry API should be called com.sun.registry.. because that's what it is.
  • If IBM wants to propose a registry API, they should call it com.ibm.registry..

Immediate benefit

When a registry API is named properly, there are many benefits. You can develop real tests. You can run the code. You can compare different registry APIs side by side within one JVM. You can get your hands on it. And if you are bold, you can use the leading edge registry API in your own production code long before the dust settles on which registry API is the "best", which registry API is good enough to be the official JOS registry API.

Sequence

In the proper light, every JOS API is a proposal. Proposals should follow the sequence that makes the Internet great. An author publishes a request for comments (RFC). The proposal is put together in a draft specification and reference implementation, using the proposal names.

After the proposed API has been used by JOS programmers, they might be qualified to respond to an RFC. There will always be things that they like about the API and things they don't. Every JOS programmer should be free to say what they like and don't like, as well as put together their own counter-proposal.

A proposed API must never occupy the space of an official package. The first proposed registry API should definitely not go in the org.jos.registry space. The first proposed API doesn't offer all of the benefits of a mature API. How can a proposed registry API occupy the space for the official registry API?

JOS programmers should compete for the most mature API for JOS. JOS will be better for it. We must use competition to push the technology forward. By this mechanism, all of us get everything we want.

About implementations

The implementations of classes in an API should be open for all. That is the point of an application programming interface, after all, for application programmers to write their own implementations the interface.

In order to support diversity, each registry package should be named for what it really is: a unique implementation.

  • My favorite self-describing-binary-file-based registry package is called com.realignsoft.masterregistry1a because that's what it is. It is one of many different registry implementations that I'm planning.
  • Your registry implementation should be called your-name-here.your-product-name-here.. because that's what it is. It is one of many registry implementations you're planning.
  • Sun Microsystems registry implementation should be called com.sun.customregistry.. because that's what it is.
  • If IBM wants to build a registry implementation, they should call it com.ibm.customregistry..

Future

The future is bright. A bridge should be planned between different registry APIs. Just as we have seen the strength of the JDBC-ODBC bridge, we can look forward to a bridge between org.jos.registry and java.lang.registry. When a bridge implementation is contructed between two competing APIs, everybody wins. All of the implementations of two competing APIs become instantly compatible.

Java is different. It has the power to get the work done now and into the future. All of the power hinges on keeping a consistent naming convention.
