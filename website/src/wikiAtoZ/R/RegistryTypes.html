RegistryPages

Article contributed by GilbertHerschberger (20 June 1999).

Introduction

There must be more than one registry API simply because there is more than one registry-like concept. At least two registries are possible because one stores objects by reference and the other stores objects by value.

  • Registry API Type 1 This kind of registry stores objects by reference. You can create an object of any type. You can store any kind of object in a registry. When it comes time for persistence, only serializable objects will be saved.
  • Registry API Type 2 This kind of registry stores objects by value. You can store primative Java data, strings and other registries in this kind of registry. The registry API is used for persistence. The registry itself is persistent, so that your objects don't have to be.

Object store

A type 2 registry is a data store. In contrast, the type 1 registry may be thought of more like an object store.

There are good reasons for JOS to have a object store API. JOS is expected to have quite a few standard objects, such as JOSSystem and other object-based devices. An object store would enable a process to pick a system object by name out of the system-wide object store. An object store, with persistence, would enable JOS to save its runtime state in ways that other operating systems can only dream about. When object stores are daisy chained among multiple computers, JOS can take full advantage of seemless distributed computing.

Item names

Currently, in a type 1 registry, an item (or entry) is expected to have only one name. This registry closely associates an entry with its name. A registry entry object is used to hold any object. The registry learns an object name by asking a registry entry.

In contrast, the type 2 registry closely associates a registry and item names. The registry manages the names; an item is entirely unaware of its own name. This provides a mechanism for an item to have more than one name.

Why are multiple names important? With Java Beans and other Java objects, a single property may be exposed by two (or more) names. In order to map a registry to a bean, the registry must be able to return a property when given any one of its names.

For example, an imaginary system-wide registry for JOS might have a JOS version 1 well-known name for a registry (x-registry:users/joe/browser) and a version 2 well-known name for the same registry (registry:jos/users/joe/applications/browser). When both of these are object references to the same object, only one real object should exist.

Other registry APIs may be possible. Each API is a valid approach to the registry. We should not choose one over another in an operating system until all of these are implemented and can be demonstrated side by side.
