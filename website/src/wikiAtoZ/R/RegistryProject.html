About this project

This is a registry API project. This registry project is focused on a pure registry API, not an implementation of a registry. All choices for interface and implementation are left up to application programmers.

  • While it makes assumptions about using an Internet URL as recommended by W3, it does not assume an OS file subsystem.
  • While it makes room for anyone to distribute their registry, it does not make assumptions how a registry will be distributed.
  • While it is compatible with Serialization, RMI, and CORBA, it does not require these.
  • While it is compatible with XML and SQL, it does not require these either.

Goals

The result of this project will be a proposal for a pure JOS registry API and a set of reference implementations:

  • .conf, .ini and various text files
  • RMI,
  • XML,
  • SQL,
  • ORKA,
  • XDC, and
  • various self-describing-binary-files.

Registry types

Both type 1 and 2 registries implement a common registry interface. There are two types of generic registries so far.

  • Type 1 This type of registry works primarily with Java primatives. The getObject() and setObject() methods are secondary and call other getXXX() and setXXX().
  • Type 2 This type of registry works primarily with objects. Other getXXX() and setXXX() methods are secondary and call getObject() and setObject().
