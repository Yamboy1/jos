Registry API - Project Update

Created: 99.07.23Last Updated: 99.12.08Project Page: RegistryPagesMailingList: arch@jos.org

Project Summary:

The purpose of the Registry API project is to build a smart registry, a unified registry design with pluggable registry implementations and pluggable registry protocols/file formats.

  • You can use it now. The registry API available for download (See RegistryPages). Your feedback is appreciated.
  • Discussion of the registry is focused on the clarification of "what is a URL?" and "how do we use the smart API?".
  • We're still working on the registry API. Based on Registry1Lessons and e-mail collaboration, we are working on the design of Registry - release 1c. In addition to initialization (.ini) files, we're adding file support for property (.properties) files.
  • We still need help for a full implementation of a generic registry, a registry iterator, and configuration (.conf) files.
  • Future: SmartRegistry - release 2a will depend on SmartAPI; StreamFactory; ObjectFactory; and ClassFactory.
-- GilbertHerschberger (23 July 1999)
For those worried (as I am) about backward compatibility, the Registry1bWrapper is an example of how you can provide bridges from one API to another. Yes, this means that any data you store in a release 1b registry is run-time compatible with a release 1c registry. This wrapper is (will be) part of release 1c.

The registry factory may need serious redesign in release 2a. Release 2a depends on both the smart API and stream factory. Of course, I'm leaving it alone in release 1c. -- GilbertHerschberger (26 July 1999).

Registry - Release 1c has been distributed as part of the JOSDistribution1d. A smart registry packages contain prototypes of a registry iterator and persistent registry. Looking back at the need for a system-wide registry and registry-compatible file/database formats, it seems that a registry is a difficult concept because there is more than one kind of "registry".

  • data registry - This is where all registries start. A data registry does not deal with custom objects at all. It deals only with primative data. A string registry, for example, uses only String and Registry objects.

  • object storage registry - This is where persistent objects are stored. Whey an object is stored, it has no behavior, only data. The object storage registry provides an mechanism to serialize objects, or convert objects to primative data.

  • shared object registry - This is where objects are created in a distributed computing environment. The registry provides a mechanism to keep objects running. Objects continue to have behavior. A shared object registry is known as an agent environment. It is also known as a service environment. CORBA provides shared objects across a network.

So, there isn't a need for one kind of registry; we need them all. A object storage registry needs a data registry. A shared object registry needs a object storage registry. -- GilbertHerschberger (8 December 1999).
