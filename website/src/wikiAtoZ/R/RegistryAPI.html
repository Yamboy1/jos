RegistryDesign; RegistryExamples

Article contributed by GilbertHerschberger (4 June 1999).

Introduction

Quotes

I personally think that a key component of the operating system will be a registry. All OS API's should be interfaces only and there should be a way to obtain the "current implementation". In this way we allow vendors to replace parts of the JOS operating system without having to modify our code. It also provides for a clean future migration path. -- ClarkEvans [1]

Developing a great registry is critical to JOS project for a number of reasons. The most obvious reason is we would like to use it in our programs.

Other reasons might not be so obvious. The registry is a good example of a pattern we'll continue see in everything else we develop inside JOS. The registry project is about learning. We must learn how to develop a SmartAPI. We must tame each of the following challenges:

  • multiple, concurrent registry APIs,
  • multiple, concurrent implementations of a registry interface,
  • using a well-know registry factory to obtain objects that are compatible with a given registry API,
  • refining the step-by-step development model used by the JOS project, and
  • learning to cooperate and compete in an open source, Internet project.

An operating system's API must be strong on interfaces. An object-oriented API gives us additional opportunities to provide adapters and base classes for each interface (as we have time). For more information on adapters and base classes, see also Adapter.

There must be a mechanism to obtain the "current" implementation for an interface. The mechanism developed by the registry project can help us refine the mechanism used by future JOS projects. It will benefit the Java community as a whole.

So far, the mechanism works like this:

  • Choose an API. An API is developed in a package corresponding to its own name, such as org.jos.demo,
  • Choose an interface within the org.jos.demo package, like org.jos.demo.Demo. Your program uses an objects that implements the org.jos.demo.Demo interface.
  • Your program should obtain an object by using a factory corresponding to its interface, like this: 
    import org.jos.demo; // the API
:
  public method() {
	 Demo t = DemoFactory.getDemo();
  }
:

This mechanism enables any programmer to replace parts of the JOS operating system. This provides a way for anyone to make a better part, make it available on the market, and compete on performance, reliability, and support. Both individuals and big vendors can plug in. Everybody wins.

The future of this mechanism depends on a strong model for version control. Without version control and a naming convention that everyone can live with, you can kiss this promise goodby. Versioning isn't strong enough to keep an operating system stable over the long haul.

A small suggestion here. Let's add a version control and security bit to the registry. That way parts of the classes could be replaced with newer versions if the security bit would allow it. -- FenixBird [2]

Security is important to the success of the registry. Even a stand-alone JOS system should discourage un-authorized access. A non-JOS network server should not be required to implement an appropriate security mechanism.
