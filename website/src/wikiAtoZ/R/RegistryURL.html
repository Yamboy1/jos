Article contributed by GilbertHerschberger (4 June 1999).

Introduction

A uniform resource locator (or URL) should be provided for a registry, among other things. After all, a registry is just another digital resource.

What is a registry URL? How is the registry URL going to be used in JOS?

Browser example

A browser test should be used to see if a new resource locator is good enough to be a uniform resource locator. A browser test is a simple sequence that goes like this:

  1. Open your favorite browser;
  2. Type a resource locator into the location field;
  3. Press the Enter key (or whatever works).
Results? If it is theoretically possible for your browser to correctly display the resource, you may have a uniform resource locator. Otherwise, you may have a resource locator that won't work inside a generic browser.

When developing a new kind of URL, your old browser probably won't display the resource correctly. It won't understand a protocol that hasn't been built yet. A browser test is theoretical. It tests to see if it could be possible. The browser test is a good way to think through your URL design.

A registry URL is a uniform resource locator for registries. It is something you can type into the location field of a browser and get good results.

Ultimate browser

The ultimate browser should be able to automatically load a registry editor when the registry-type URL is entered into the location field. When there are more than one registry editor, a browser should be configured to ask which one to use or automatically select your favorite.

Eliminate ambiguity

Ambiguity is bad, very bad. Ambiguity exists when two or more things have the same name. If you have two or more text editors, you have two or more things with the same name: text editor.

As URL designers, we must work hard to eliminate ambiguity from uniform resource locator in order to get predictable results. When a resource can be displayed in more than one program, there is an ambiguity.

When you have installed more than one browser on your computer, which browser should your computer launch? This is a reason for operating system configuration. You should configure your operating system to ask you which browser to launch or indicate which of your browsers you prefer.

Uniform

Resource locators have been used for about 50 years in the software industry. Resource locators are also known by its other names, like file names, device names, and command lines. Uniform resource locators are a more recent evolution. There are good resource locators; and then there are great resource locators.

  • A good resource locator is going to find a document from within a specific program. When you open a text editor, you can load a text file called /usr/local/example. Within a text editor, you also may be able to use universal network computer (or UNC) names, like //host/usr/local/example. This may be all the resource location you'll ever need.
  • A great resource locator is going to find a resource from anywhere. By typing a URL into the location field of your favorite browser, you can potentially find any kind of resource stored anywhere on a network. When you type in a URL for a text file, your operating system or browser should load a text editor. Which text editor? You can tell your browser which text editor you prefer.

In search of uniformity

The first step on our evolution to URL is the file extension. A file extension is another kind of protocol selector. While protocol is a suffix to a file name, protocol is a prefix for a URL. There may be nothing inherently wrong with a protocol as a suffix. Prefix or suffix, it doesn't really matter.

Many products work with file extensions only on a limited basis. Three character file extensions have many limitations.

  • One for one It is a bad assumption to think there is one and only one program capable of dealing with a file. For example, an .html file may be viewed by a browser; but, it is viewed and edited with an HTML editor, viewed and edited with a text editor, uploaded and downloaded with ftp, imported and exported with a word processor, generated or scanned by a custom program. In fact, there are many programs at once that are capable of dealing with each kind of file.
  • Version It is a bad assumption to think there is one and only one version of a program installed on a computer. An .html file, for example, may be viewable by any version of any browser. When I have more than one version of a program installed, I should be able to choose which one I want to use today.
  • tiny namespace Experience has shown that three characters isn't enough namespace. There are a hundred thousand different protocols and more on the way. There isn't enough expression in three characters to make file extensions meaningful to people that need to read them.
  • reuse Reusing a file extension for a new purpose makes matters worse. With such a small namespace, reuse is unavoidable. Did you ever notice how config.sys, a text file, has a binary file extension? It conflicts with conflicts with io.sys, msdos.sys, ansi.sys, himem.sys, mouse.sys, etc., listed inside config.sys.

While I have text files with about a hundred different file extensions, my stupidest operating system recognizes only one: .txt. I'll list a few extensions for text files just for fun:

  • .
  • .000 through .999 -- sometimes
  • .bak -- sometimes
  • .bat
  • .c, .cc, .cpp
  • .conf
  • .h, .hh, .hpp
  • .htm, .html
  • .ini
  • .java
  • .log
  • .mak
  • .sys
  • .txt

All of these are text files. Shouldn't they be recognized by a browser and an operating system as text files? A file extension isn't enough.

Refinement

By providing host and port, the URL provides a unform resource location worthy of a global Internet. The World Wide Web Consortium (W3) continues working to refine URLs and URIs.

With the introduction of an Internet URL, the World Wide Web on the Internet was born. According to Tim Berners-Lee, the URL is significant to the long term success of the World Wide Web, more so than HTML, HTTP and browsers. The uniform resource locator made it easier to access all kinds of resources across the Internet. The browser makes resources easier to see.

Experimental protocols

The World Wide Web Consortium recommends using x- to indicate an experimental protocol. An experimental protocol is any protocol that has not reached the status of Internet standard. Since our registry protocol is experimental, it should start with x-.

Protocol

Ultimately, the registry protocol will be a real distributed computing protocol. It will be a protocol in the style of SMTP or FTP. It will be a protocol in the sense of sending and recieving data across an Internet connection.

The registry: protocol indicates a resource is an implementation of the registry interface. A protocol, like this one, should be implement-able across platforms. A registry should still be a registry no matter what operating system you use.

More directly, this protocol also explains to operating systems and browsers that a resource is viewable and editable with a registry editor. A file extension cannot explain registry files to a browser because a registry can be stored in at least .ser, .conf, .ini and .xml files.

Subprotocols

Selecting a specific implemenation of registry storage is provided by a subprotocol. Subprotocol is optional when one of these subprotocols is configured as the default registry protocol. There are an unlimited number of registry subprotocols. Registries can be implemented in all sorts of different ways. Opening up the registry to many different storage options strengthens the position of JOS as an operating system.

  • The proposed new: subprotocol is useful for non-persistent registries. A new non-persistent registry can be created, stuffed full of data, and passed to some method of another object. It doesn't have to be saved at all.
  • The proposed file: subprotocol is useful for save-on-demand registries. The file subprotocol is based on the idea of import and export. The getRegistry() method performs an import. The saveAs() method performs an export to any URL. This kind of registry probably uses whatever file format is indicated by file extension. If the extension is .xml, the registry is loaded/saved in .xml file format. If the extension is .conf or .ini, the registry is loaded/saved in configuration file format.
  • The proposed ftp: and http: subprotocols are useful for uploading/downloading a registry from a service across the Internet. Imagine this: As part of our work at the JOS organization, we will build registries, upload them to CVS, making them available for others to download.
  • The proposed net: subprotocol is used to indicate the network-wide registry. The network registry should be able to discover a registry on any kind of network device, not just personal computers running JOS, the operating system.
  • The proposed os: subprotocol is used to indicate the operating system registry. Participation in a system-wide registry is optional. Since this is the default subprotocol, both of these URLs return the same object: 
    x-registry:/users/user-1/browser
x-registry:os:/users/user-1/browser
  • The proposed rmi: subprotocol is used to indicate an RMI-based registry service.
  • The proposed sql: subprotocol is useful for storing a registry using an SQL service. For all those client/server SQL applications, the benefits of a SQL-stored registry would be consistent with the rest of the application. SQL programmers wouldn't have to learn a new storage model. Applications can warehouse and analyse registry data with all the power of standard SQL.

Host and port

To get a registry proxy using the registry protocol, a registry service must be running on another host. How does the registry protocol work? How is the registry protocol secured? These are questions better left to future work. A registry protocol should be developed to meet the unique needs of a volatile remote registry. You should be able to use a registry editor running on machine A to edit a registry on machine B.

The host and port are optional parts of a registry URL, after a subprotocol. A registry can be stored on this host or another.

  • this host When a registry is stored locally, there is no need to indicate host and port. When /usr/local is the current directory, all three URLs import the same file. 
    x-registry:file:example
x-registry:file:./example
x-registry:file:/usr/local/example
  • another host When a registry is stored remotely, a mechanism is required to indicate a host, and as with all experimental protocols, a TCP/IP port. 
    x-registry:file://www.jos.org:2000/usr/local/example

Bad examples

There are plenty of bad examples of URLs. Sun Microsystems implemented bad examples into their own java.net package.

For more information, see also the W3 website.

To understand the proper design of a URL, you should be aware that the most popular implementation of the HTTP URL is technically incorrect. For the HTTP protocol, host is always optional. Requiring localhost to download local files is needlessly redundant. The HTTP URL should have fully supported all of these: 

http:index.html
http:./index.html
http:/usr/local/index.html
http://localhost/usr/local/index.html
