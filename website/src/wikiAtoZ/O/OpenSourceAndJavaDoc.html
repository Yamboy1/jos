ArchitectureGroup

Article contributed by GilbertHerschberger (17 December 1999).

Introduction

When you contribute your code to an open-source project, you would like others to see both the source code and high-level view. The source code would provide the details. The high-level view would provide useful information on how to use an interface and class. Source code might be the wrong level of detail when you're most interested in using a high-level class from another package. The source code and high-level view should be integrated so you can switch from one to the other, and back.

JavaDoc

JavaDoc translates source code into HTML pages. It provides a high-level view of interfaces and classes. It is especially useful for a closed-source project, where only the JavaDoc and class files are delivered to other people.

Source-to-HTML

To translate source code into HTML pages, there are tools such as SourceServer, Source-Doc and JavaToHTML. These tools show all the details. I would like a unified view of my source code from design -- through source -- to documentation.

Links

When source code is displayed in HTML, it should have a link to JavaDoc-like table-of-contents, method index, variable index, package index and such. When documentation is displayed in HTML, it should have a link to the source code. Just as JavaDoc should be linked to additional documentation for programmers, administrators and customers, the design elements of a project should be consistently presented.

I would like to see all of this to use a consistent look-and-feel. In fact, the idea I like most is the integration of JavaDoc-like pages and SourceServer-like pages directly into the JOS Wiki mechanism.

Why Wiki

When JOS Wiki displays an "article", the contents of an article are stored separately from its HTML wrapper. There are no <HTML> or <BODY> tags store in an article. Rather, a wrapper and an article are combined only when a page is requested. If a tool were to write "articles" for JOS Wiki instead of complete HTML pages, output from a tool would become a read-only article.

Forums

Mailing lists work for theory and discussion. JOS Wiki is useful for documentation, news and distribution links. Mailman supports different mailing lists. JOS Wiki supports different "forums". We prefer to use only the Main forum. Having more than one forum for discussion is difficult to manage for contributing authors. We prefer to use mailing lists instead.

But what about a forum for JavaDoc and SourceServer "articles"? These articles could be generated by a tool off-line and integrated into the look-and-feel of the JOS Project website.

Navigation

When navigating by package, a flat list of all packages should appear in a "package index". Packages should be listed in alphabetical order. Click on a package to see more information about that package.

When navigating by inheritence, a tree of inheritence relationships should appear on a "tree" page. An interfaces should be listed, along with every implementation of it. A class should be listed, along with every class derived from it -- even classes from other packages. Such an inhertence tree is available through pure reflection from .class files. Classes must be compiled successfully before appearing in such a list.

To break this process into two parts, a list of all interfaces would be displayed on one page. Each interface would have a link to a page showing all its implementations. A list of all classes would be displayed on a page. Each class would have a link to a page showing all its subclasses.
