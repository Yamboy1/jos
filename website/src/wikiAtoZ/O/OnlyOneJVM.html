Does JOS support only one bytecode interpreter?

I have my doubts about the relationship between the JOS kernel and the bytecode interpreter. All this time, I had assumed that JOS would provide both a kernel and BytecodeInterpreter, and the kernel would support multiple, concurrent bytecode interpreters.

I like choice. I like what I already have. I want to keep what I already have. I have collected all of these JVMs.

  • For Linux:
    • Blackdown JDK - Java 1.0.?

  • For Microsoft Windows 95:

    • Sun Microsystems JDK 1.0.2
    • Sun Microsystems JRE 1.1.7B
    • Borland Java Add-On 1.0.2
    • Borland JDK 1.1.6 (from JBuilder)
    • Novell Intranetware - Java 1.0.?
    • Netscape Navigator 3.x JVM - Java 1.0.?
    • Netscape Navigator 4.x JVM - Java 1.1.?
    • Microsoft Internet Explorer 3.x JVM - Java 1.0.?
    • Microsoft Internet Explorer 4.x JVM - Java 1.1.?, and
    • many more.

  • IBM AS/400
    • IBM JRE - Java 1.0.?, and

Why? Because my customers use many different operating systems. They have many kinds of networks and equipment. That's just the way it is. It isn't going to change anytime soon.

When my customers have so many different environments where they want to deploy Java bytecode, I need to find the best JVM for the job. Some JVMs don't quite work the way they are supposed to. Some JVMs are great for development and pathetic at production.

I don't want to choose between the three versions of Java. No, I need to switch between them at a whim on my own development machine. The faster I can switch between them, the more work I can get done.

I don't want JOS to assume that all of the Java work is going to be performed in Java 2.0--because it isn't. Yes, I have Java 1.0.2 projects in the planning stages for release over the next three to five years. Why? Because Java 1.1 and 2.0 aren't feasible. My customers don't have them and won't have them any time soon. I'm not going to build applications for a fantasy platform that doesn't exist yet.

I don't want JOS to assume that Java 2.0 is completely backward compatible with Java 1.1.x or Java 1.0.2--because it isn't. When we are building an application targeted for Java 1.1.x, the Java 2.0 classes should not "exist" for that version of the bytecode interpreter.

JOS should support all three versions of Java. Maybe it could have one JOS kernel and many different JOS-compatible bytecode interpreters. The JOS kernel could select a unique bytecode interpreter to run for each process.
