SmartAPI

Article contributed by GilbertHerschberger (20 Jul 1999).

Introduction

Yes, I am jumping ahead to something I've been wondering about. While I'm sure I can create objects with an object factory, I wonder if we can create objects using a serialization subprotocol. Here's what I have in mind.

Serialization

Using java.io.FileOutputStream and java.io.ObjectOutputStream, you can write any serialize-able object to a serialization file (.ser). Using java.io.FileInputStream and java.io.ObjectInputStream, you can read any serialized object from a serialization file.

Note: You cannot count on this part of the Java API. Related interfaces and classes are not available on every Java platform.

Object factory

The factory.ObjectFactory class could use another controller. The default configuration for the object factory should include the serial object controller. While a primative controller would support the object: protocol, the serial object controller would support the ser: subprotocol.

This controller needs to know how to open a serialization-compatible stream. It extracts an object from that stream.

Protential URLs

object:file:/usr/local/example.ser

Based on the .ser extension, the serial object controller would open the /usr/local/example.ser file and return the object it contains. The object factory doesn't care what kind of object is inside the file.

object:ser:file:/usr/local/example

Based on the ser: subprotocol, the serial object controller would open the /usr/local/example file as a serialization file even through it doesn't have a .ser extension.

Across the network

Since serialization files might be on other machines, the serialization controller should be able to use a standard stream factory.

object:ser:ftp://host/download/example
object:ser:http://host/download/example

The serial object controller should support the ftp: and http: sub-subprotocols, too.

Removing assumptions

Since an object can be created with a serialization file, you might know the name of a file, but not know the object's class name.

Serial object controller

This controller would implement the object factory's controller interface or extend the object factory's base controller class.

public class SerialObjectController 
	 extends BaseController {
  public static final String SUBPROTOCOL = "ser:";
  public static final String EXTENSION = ".ser";
  public static final String MATCH = PROTOCOL + SUBPROTOCOL;
  public SerialObjectController() {
  }
  public boolean acceptsURL( String url ) {
	 if ( url.startsWith( MATCH ) ) {
		return true;
	 }
	 if ( url.endsWith( EXTENSION ) ) {
		return true;
	 }
	 return false;
  }
  public Object getObject( String url ) {
	 StreamFactory factory = StreamFactory.getFactory();
	 ObjectInputStream is;
	 if ( url.startsWith( MATCH ) ) {
		f = url.substring( MATCH.length() );
		is = (ObjectInputStream) factory.getObjectInputStream( f );
	 }
	 else {
		if ( url.startsWith( PROTOCOL ) ) {
		  f = url.substring( PROTOCOL.length() );
		  is = (ObjectInputStream) factory.getObjectInputStream( f );
		}
		else {
		  return null;
		}
	 }
	 if ( is == null ) {
		return null;
	 }
	 try {
		return is.readObject();
	 }
	 catch( InstantiationException e ) {
		return null;
	 }
	 catch( Illegal..Exception e ) {
		return null;
	 }
	 catch( IOException e ) {
		return null;
	 }
  }
}

What about beans?

When a bean is instantiated, it needs extra-special attention. Does the ObjectInputStream do this? Or, does the serial object controller need to do this? Or, does the requesting application need to do this?
