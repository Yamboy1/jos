Introduction

Trying to get all members of this project to agree on a single software license is like trying to get the software industry to agree on a single software license. It is not only unnecessary, but a bad idea.

Our members do not agree on a single software license, and do not have to. No member should be excluded from the discussion simply because of their license preference. One license for everyone is bad. Our members come from different backgrounds. Choice of license may be greatly influenced by other projects. Some members contribute to more than one open source project and will continue to do so. They contribute to the advancement of a Java-based operating system by research and expertise. This is sometimes reflected in sister projects.

It is not necessary nor desireable to force an umbrella project, like the JOS Project, to agree to a single license. It is not necessary because we already have sister projects distributed under a variety of software license and it works. It is not desireable because license is always a sister project issue. A sister project should apply a license consistently.

Collectively, members of this project are not responsible for the content or license of any sister project. We cannot imagine what might be best for a sister project. Only a sister project can make that determination. A sister project must make that determination for itself.

Never happen

Look at it practically. It isn't going to happen. This Project has not agreed to a single license for all sister projects in three years. It is not going to agree to a single license in years to come.

If you think this project is going to agree on a single software license, think again. Advocates of one license cannot possibly switch to another after the fact. This causes too many broken promises.

Like a magazine

When you publish an article in a magazine or newspaper, anyone can use your information for any purpose. Publishing is an honorable act of sharing information.

This project publishes information on its mailing list and website. Anyone can use this information for any purpose. While copyright is retained by the author, its content can help others to know what we've learned.

One size fits none

There is no such thing as one-size-fits-all, especially at the umbrella project level. The JOS Project is an umbrella project. Members of a sister project are responsible for how they distribute their products. Each product has its own licensing issues.

A sister project is formed to advance some specialty of a Java-based operating system. Members volunteer for a sister project as they have time and it suites their taste in software license. While a sister project might be concerned about software license, this umbrella project is not.

A subproject is formed to experiment with part of the Java platform. Members volunteer to work on a subproject. Often enough, the results a subproject is applicable to everyone because its product is experience, not source code.

There are plenty of specialties. There is much room for the advancement of Java-based operating systems.

Other projects have been formed on the Internet and behind closed doors. For whatever reason, some choose not to become members of this project. When articles are contributed and messages are posted on a mailing list, everyone benefits. The information is free.

Examples

The JJOSProject, for example, uses its own license. This license was written by JohnMorrison and adopted by ToddMiller (as far as I can tell). This is a project suitable for those who cannot afford the GNU General Public License ( GPL ).

On the other hand, the CjOSProject, SimpleJOS and RJK uses the GPL. My ultra-conservative corporate customers prefer this software license because it is a well-known. They do not write commercial software. Their lawyers reviewed the it once for all time and found it acceptable.

  • SimpleJOS uses the GPL because Linux uses the GPL.
Many sister projects can co-exist. Many can promote and advocate the idea of a Java-based operating system. Members of one project are not necessarily members of another.

Summary

If you are going to build an implementation and distribute it under a software license, you should join (or start) a subproject or sister project. That is where the license issue should be resolved.
