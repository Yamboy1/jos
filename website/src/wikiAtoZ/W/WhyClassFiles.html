Introduction

While looking into the startup sequence for JOS, the following question keeps coming up:

Why are bytecodes stored in .class files?

There are two assumptions here. The first assumption is the Java virtual machine has an OS file subsystem available. The second assumption is one file per class.

When Java sits on top of a foreign OS, the foreign OS provides a convenient OS file subsystem. Java uses .class files because it was convenient, not because it was the right thing to do.

Have you ever used FTP to upload a large Java applet using .class files? It is exasperating. There are so many .class files. It is so exasperating that many switched to .zip and .jar files without hesitation. Anything must be better than .class files, right? Maybe not. See also WhyJarsAreBad.

Does JOS have an OS file subsystem available? No. And JOS isn't the only one. There are other Java virtual machines with no OS file subsystem, too.

The second assumption is one file per class. One file per class is exactly why Java failed to scale up when installed on an HTTP service. It is expensive to open a hundred thousand little .class files through an HTTP service. Each file must be downloaded in a separate connection.

JOS should put more than one class in a file. JOS should put one package per file. See also PackageManager and WhyPackageFiles.

Are .class files required?

Apparently, .class files are not required at runtime when using the Java virtual machine.
