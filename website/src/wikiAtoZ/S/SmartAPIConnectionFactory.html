SmartAPIPages; SmartAPIStatementFactory

Article contributed by GilbertHerschberger (25 August 1999, updated 27 December 2001).

Introduction

Java Database Connectivitity ( JDBC ) uses an open-ended API. Does it go far enough? I don't think so. It complicates the first thing you'd like to do with JDBC: create a connection.

This article illustrates how the SmartAPI can simplify JDBC and its plug-in JDBC drivers.

Registering drivers

There are lots of SQL servers already and new SQL servers are expected. Each SQL server might need a unique driver. Sun Microsystems did not want to gather and redistribute a JDBC driver for every SQL server ever made. The JDBC API had to be open-ended. While one-stop-shopping might be great for end-users, Sun built an open-ended API. The JDBC driver became a plug-in component for the JDBC API.

The JDBC API provides an SQL connection and connection factory. The SQL connection is called Connection and the connection factory is called DriverManager. The driver manager uses a driver to create connections.

As is, application programmers must be concerned about registering JDBC drivers before using the JDBC connection factory. Unfortunately, the default connection factory is unaware of any JDBC drivers. Drivers are expected to register themselves with the driver manager in a static method: 

  static {
    DriverManager.registerDriver( new DemoDriver() );
  }

Attention JDBC programmers:

If you would like to contribute your connection controller to the JOS Project or have more information about specific JDBC drivers, please send an e-mail to me, GilbertHerschberger, at gchii@mindspring.com .

Recommended

Applications are expected to load a driver class, so their static method will register the driver: 
  public void example() {
    try {
      Class.forName( "org.jos.demo.DemoDriver" );
    }
    catch( ClassNotFoundException e ) {
    }
  }
This looks strange to application programmers, especially beginners. Why would you create a class with Class.forName and not do anything with it? It is because the class registers itself with the java.sql.DriverManager.

Not recommended

You should use the Class.forName() technique to register JDBC drivers. Unlike what you might read elsewhere, you should not register each driver with DriverManager. 
  public void badExample() {
    /* do not use
    DriverManager.registerDriver( new DemoDriver() );
    */
  }
But wait! A JDBC driver would be registered twice. It would be registered automatically by the driver's static method, and again, by the explicit call to registerDriver(). That doesn't make it any easier.

In a tight loop, you're code may register hundreds of copies of a driver. The DriverManager should not eliminate duplicate JDBC drivers based on class name, because each JDBC driver might be uniquely configured (connection pool, timeout, secure sockets, proxy host, etc.).

System-wide configuration

The better connection factory would abstract the individual JDBC drivers away from application programming. A smart connection factory would be able to return an Connection object without writing code in each application to register drivers.

A smart connection factory would be configured (once) with a system-wide configuration. A configuration would associate a sub-protocol with a JDBC driver. The configuration look something like this: 

[jdbc-connection]
odbc=sun.jdbc.odbc.JdbcOdbcDriver
as400=com.ibm.as400.access.AS400JDBCDriver
db2=com.ibm.db2.jdbc.app.DB2Driver
JDBC drivers are added to your system, not your application. It is unnecessary to create a new way of configuring drivers one application at a time. Adding a new JDBC driver to your system makes that JDBC driver available to all applications running on your system.

Note: JDBC subprotocols are unmanaged. Unlike protocols on the Internet, there is no authority to register new JDBC subprotocols.

JDBC URL

All JDBC connections are created by passing a URL to a connection factory. The JDBC-compatible URL must start with the jdbc: scheme. When we want to connect to a ODBC datasource, we must provide a JDBC-compatible URL. The URL for a JOSDEMO datasource might look like this: 
jdbc:odbc:JOSDEMO?user=joe;password=xxx

Without SmartAPI

To create a connection without the SmartAPI, you're code looks like this: 
  public void example() {
    try {
      Class.forName( "sun.jdbc.odbc.JdbcOdbcDriver" );
    }
    catch( ClassNotFoundException e ) {
      return;
    }

    String url = "jdbc:odbc:JOSDEMO?user=joe;password=xxx";
    Connection c;
    try {
      c = DriverManager.getConnection( url );
    }
    catch( SQLException e ) {
      return;
    }
  }

With SmartAPI

To create a connection with the SmartAPI, the equivalent code looks like this: 
  public void example() {
    String s = "jdbc:odbc:JOSDEMO?user=joe;password=xxx";
    Connection c = (Connection) new URI( s ).getObject();
    if ( c == null ) {
      return;
    }
  }
The smart object factory passes all requests for the jdbc: protocol to the JDBCConnectionFactory. Since this factory is responsible for creating connections, it is called the JDBC connection factory. This factory is smart enough to automatically load a driver based on a sub-protocol. It uses the sub-protocol to look up a JDBC driver in its configuration. This factory is smart enough to register each driver once and only once.

The smart connection factory is compatible with the JDBC API. It uses DriverManager to create connections. It uses the Class.forName() technique to load drivers that register themselves. While it is compatible with the JDBC API, it provides a no-nonsense mechanism to get what your application really wants: a JDBC connection.
