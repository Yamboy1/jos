SmartAPIPages

Article contributed by GilbertHerschberger (25 August 1999).

How far

I have been working on the José(tm) project for over a year. I have been working on the JOS project for about six months. These two projects have restarted my career. I look back to see how far I've come.

Introduction

The José(tm) project started with two simple interfaces, Console and Application. The Console interface is simple enough. It has only one method: 

public interface Console {
  public void println( String v );
}

The original console was created when the utility of such an interface could not be determined. It has proven to be useful in practice. It shows up in every program I build.

At the time, a console was experimental. A factory for a console would have been too much. Why build a factory when the value of an interface is still unknown?

Until a few days ago, I didn't have the energy to think about a console factory. With the greatest of ease, I reused the components of the SmartAPI to build a console factory. The factory provides an implementation of the console interface in all its various forms.

The console factory uses the x-console: URI scheme. It supports the class: subprotocol, so I can create a console from a class name.

Console Studio

The console interface, console factory, and built-in implementations of the console interface are found in the Console Studio. Java classes are found in the consolestudio package.

Improvement

Over the last year, we have used a continuous improvement model to build better software. We have been methodical. We have built from the bottom up. The synergy we are looking for in extremely reusable components has been growing. And, there is a lot of work yet to do.

The console factory is yet another example of how the SmartAPI gets you directly what you're looking for. In order to create a new console, you use code that looks like this: 

  public void example() {
	 String s = "x-console:class:org.jos.demo.DemoConsole";
	 Console c = (Console) new URI( s ).getObject();
	 if ( c == null ) {
		return;
	 }

	 c.println( "Hello, World!" );
  }

Custom console

End-users have preferences for the kind of console they would like to use. Some might prefer a Swing-based console over something in AWT. Others might prefer AWT. You can provide them whatever they want.

When an end-user configures your application, they can configure it with their own console URI. When your application is written to the well-known Console interface, that's all your application needs to know.

Because any class can implement the Console interface, your application can write to any kind of console. You can have a file-based console, log-based console, remote console, and more.

Make your own HTML console

You can make your own HTML console. You extend an HTML component and implement the visual console interface. To display HTML on the console, you use the println() method. This kind of console can be passed to a JOS program and its HTML is output directly to an HTML component.
