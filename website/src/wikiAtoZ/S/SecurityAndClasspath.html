ProcessPages

Article contributed by GilbertHerschberger (1 May 2000).

Introduction

A static form of CLASSPATH is used in a classic virtual machine. This is an aspect of security in the JavaVirtualMachineSpecification. A class loaded through CLASSPATH can be trusted.

One virtual machine can create another. As each new virtual machine is created, it gets a new CLASSPATH and it is trusted. So, a trusted application (or applet) can download any code and run it through a new virtual machine. When code is downloaded, it is trusted.

Bytecode process

When a new primordial class loader is created in a MPCL-compatible virtual machine, the rules are identical. A MPCL-compatible virtual machine is neither more nor less secure than a classic or JNI-compatible virtual machine.

The ability to create a separate primordial class loader must be tied to the security manager. Creating a primordial class loader is equivalent to creating a machine code process. Both use the Runtime.exec() method.

About CLASSPATH

CLASSPATH is passed to a new bytecode process as part of the environment parameter. By default, the CLASSPATH of a child process is inherited from the CLASSPATH of its parent. When specified, the CLASSPATH of the new process can be anything.

The ability to create a process implies an extreme level of trust. An application, once trusted, can download and install all kinds of bytecode from anywhere (or anyone) and then execute it on your machine. Bytecode verification is not required.

In the end, a Java Virtual Machine is neither more nor less secure than machine code. When one class is trusted, all classes are trusted.

System property

Is the CLASSPATH field reflected in system properties? If so, can it be changed? When changed, does a primordial class loader use it from that point forward?

With a dynamic CLASSPATH manager, changes to CLASSPATH should be reflected in system properties.
