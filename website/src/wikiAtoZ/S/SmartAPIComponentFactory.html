SmartAPIPages

Article contributed by GilbertHerschberger (2 August 1999).

Introduction

I want a component factory. Rather than use the AWT directly in a visual designer, I want to use an AWT component factory.

Protocol

If we use a component: protocol, we should be able to install an AWTController. All of the URIs for components would return an instance of java.awt.Component. Wouldn't it be nice if both the AWTController and SwingXXXController were compatible? Then, I could build a button, like this: 

x-component:button

Parameters

Even the parameters for a button could be compatible. If I want a button with "OK" as a caption, I should be able to use this URI: 

x-component:button?caption=OK

Real component API

In order to put together a real component API, and not a component framework, we need common interfaces for all these things. In other words, a button object should implement the button interface: 

public interface Button {
}

If there are truely across-the-board methods that a button should implement, these methods should be part of the button interface.

No base classes

The current problem with the AWT and different releases of Swing is the use of base classes. Base classes make it difficult to reconcile AWT and Swing. It makes it difficult to reconcile different releases of Swing. Swing hasn't really changes that much; but, different releases are incompatible anyway.

If there is a component API, it would be heavy on interfaces and light on abstract classes and concrete classes.

Sizeable

Visual components may be size-able. There should be a Sizeable interface implemented by any object that's sizeable.

Visual

All visual components should implement the VisualComponent interface. A visual component may be required to implement show() and hide(), or setVisible().

Focus-able

All visual components that can retain focus should implement the Focusable interface. Any visual component that does not implement this interface cannot obtain focus.

Enable-able

Any component that can be enabled and disabled should implement the Enableable interface. For components that are always enabled, they should not implement this interface

Container

For those objects that can contain other components, both visual and non-visual components, they should implement the container interface. Is there also a corresponding Containable interface?

Why so many interfaces?

If there is an interface for each perspective of our visual components, the compiler could check the consistence of our design. That is what a compiler is for, isn't it? It may seem like a lot of work; but, it would mean that you can build one system that's compatible with AWT and each release of Swing. That is a future-oriented system. When the next Swing is released, you don't have to do so much work.

Layout manager

With such an interface, implementations of a layout manager work with all component libraries, not just one. The layout manager does its work through the Sizeable and Locatable interface.

Lightweight components

You can implement lightweight components through the Paintable interface. Such a interface is implemented by any object (any object at all) that can be painted. A Graphics object is passed to the paintable object. And paintable doesn't mean a rich, peer-based component either. An application object could be paintable. The z-order manager keeps track of its objects through the paintable interface.
