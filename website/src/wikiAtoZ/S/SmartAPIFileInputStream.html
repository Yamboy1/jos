SmartAPIPages

Article contributed by GilbertHerschberger (3 July 2002).

Introduction

The in:file: scheme is implemented by a file input stream controller. What is a difference a 2a plug-in and a 4a plug-in? It is easy to switch your program from version 2a to 4a at your leisure because the URI is backward compatible. Any exception "thrown" by a 4a-compatible controller is "caught" by the URI class. 
:
; *D! import org.jos.smartapi2a.URI;
import org.jos.smartapi4a.URI;
:
Take a look at the file input stream plug-in for Smart API 2a. It implements the in:file: scheme. Its getObject() method simply returns null if a file input stream cannot be created. 
  public Object getObject( String v ) {
    :
    try {
      return new FileInputStream( v );
    }
    catch( IOException e ) {
      return null;
    }
  }
The plug-in for Smart API 4a also implements the in:file: scheme, but its getObject() method returns an instance of the java.lang.Exception class. 
  public Object getObject( String v ) {
    :
    try {
      return new FileInputStream( v );
    }
    catch( IOException e ) {
      return e;
    }
  }

Improved read-ability

By itself, the file input stream plug-in might not look like much. But there are two very important benefits that exceed the ability of Sun Microsystems' runtime classes.
  1. This plug-in converts a given path from platform-independent to platform-specific. It converts the slash character to a backslash (on Windows) and converts the backslash to a slash (on Linux).
  2. This plug-in works with scheme chaining so that you can create a buffered data input stream for a given file. The in:data:buffer:file:example.txt URI returns an instance of the java.io.DataInputStream class.
