SmartAPIPages

Article contributed by GilbertHerschberger (30 August 1999).

Introduction

You'd like to make your class library a little smarter. You'd like to reuse the components in the SmartAPI class library. This article is a good place to start.

How do I build my first SmartAPI?

Let's go through an example. Our scenario is simple so that the essential elements of this process are outstanding.

Process

A typical class library is made a little smarter by using the following steps:

  1. Build a CustomController interface.

    It is easy to do. You build an extension of the Controller interface, like this: 

    import org.jos.smartapi2a.Controller;
public interface <EM>Custom</EM>Controller
	 extends Controller {
  public static final String <EM>CUSTOM</EM>_PROTCOL
		= "<EM>custom-protocol</EM>:";
}

    This interface provides a type-safe way to add your custom controllers to your custom smart factory. That is the essence of this step.

  2. Build a BasicCustomController class.

    This is a very important step in providing a factory for components inside your class library. A basic controller class implements the CustomController interface.

	 implements <EM>Custom</EM>Controller {
  public Basic<EM>Custom</EM>Controller() {

  public boolean acceptsURI( String v ) {
	 if ( v == null ) {
		return false;
	 }
	 /* *L!
	 if ( v.startsWith( <EM>CUSTOM</EM>_PROTOCOL ) ) {

    Your BasicCustomController depends upon other classes in your class library. Your controller might support one or more custom classes and/or classes from other class libraries. Your controller might support the configuration of an object through a query string. (Fragments are not supposed to be passed to a controller.)

  3. Build a CustomFactory class.

    The smart factory provides a mechanism to return an object created by your controller(s). Other programmers write other controllers. The smart factory provides a mechanism to return object from other controllers. Your smart factory class should extend the SmartFactory from the Smart API class library.

    import org.jos.smartapi1b.Controller;
import org.jos.smartapi1b.SmartFactory;
public class <EM>Custom</EM>Factory
	 extends SmartFactory {
  public <EM>Custom</EM>Factory() {
	 super();
  }
  public void addController( Controller v ) {
	 if ( v == null ) {
		return;
	 }
	 if ( v instanceof <EM>Custom</EM>Controller ) {
		super.addController( v );
	 }
  }

    Some smart factories are not plug-ins for the system-wide smart object factory. The InputStreamFactory and OutputStreamFactory, for example, must not be plugged directly into the smart object factory; they implement the same URI scheme. If your smart factory is a plug-in for the system-wide smart object factory, it must extend SmartFactory and implement ObjectController.

  4. Build a test suite.

    As with any high quality software, you should build a test suite to test both the URIs that should work and those that should not. It might not be possible to test for all combinations; that is the nature of a URI scheme. Test as many as you need.

    At the first level, you should run your test suite directly linked to your BasicCustomController. 

    	 Controller controller = new Basic<EM>Custom</EM>Controller();
	 :

    Next, you should run your test suite linked to your CustomFactory. 

    	 Controller controller = new <EM>Custom</EM>Factory();
	 :

    If necessary, you should run your test suite linked indirectly through the URL class. 

    	 String s = "<EM>custom-protocol:path</EM>";
	 Object o = new URI( s ).getObject();

Default configuration

Inside the constructor for your CustomFactory class, you should provide a default configuration for your BasicCustomController with the addController() method.

  public <EM>Custom</EM>Factory() {
	 super();
	 <STRONG>addController(
			new Basic<EM>Custom</EM>Controller() );</STRONG>
  }

A future release of the SmartAPI class library should define a more specific mechanism to configure smart factories. Meanwhile, a custom factory should configure itself in its own constructor.

Note: The system-wide smart object factory is responsible for configuring itself and all of the system-wide smart factories added to it. This configuration should be stored a single registry-compatible configuration file. Meanwhile, the smart object factory should configure itself in its own constructor, using the addControllerFor() method.

Foreign controllers

Sometimes you should use addController() and sometimes you should use addControllerFor().

How do I know when to use which?

The addController() method should be used for controllers your factory depends on. If a controller is critical to the operation of your factory, use addController() In practice, a controller in the same package as a factory can use addController() because all classes in a package are distributed together.

The addControllerFor() method should be used for foreign controllers, controllers your factory is aware of. If a controller is optional, use addControllerFor(). In practice, any controller in some other package is optional. It is a foreign controller.

Development kit

We should work on developing a development kit for other programmers. It should include more information on building smart API. Such a kit should focus on how to build SmartAPI-compatible class libraries for JOS.
