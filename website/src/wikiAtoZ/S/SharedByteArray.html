ProcessPages

Article contributed by GilbertHerschberger (27 April 2000).

Introduction

The most primative of the Java primatives might be the byte array. A MPCL-compatible virtual machine must be able to pass a byte array cleanly from one process to another.

Scenario

Two user processes are running in a MPCL-compatible virtual machine. One application needs a byte array found in the other. Since no object can be owned by both processes, how do they communicate? They communicate the old-fashioned way, through a network protocol stack.

Using TCP, the first application creates a server socket and the second creates a socket. With an open channel, one application can send a byte array to the other by writing the byte array to the socket's output stream. The other application can receive a byte array by reading a byte array from the socket's input stream.

Protocol stack

Even a fully optimized protocol stack has to run somewhere. Likely, it runs in a user process, isolated from all others. The following diagram illustrates the arrangement of processes required to implement a protocol stack.

A MPCL-compatible and JNI-compatible virtual machine is potentially more efficient for communication than a classic virtual machine. More than one bytecode process is running under the control of a single heap. It is the single heap that crosses the persistence barrier in a virtual machine without crossing the persistence barrier in a bytecode process.

You can have your cake and eat it too.

JNI-compatible

Optimization of a virtual machine is not limited to MPCL. The same thing is true of a JNI-compatible virtual machine. As more than one virtual machine can run within a machine code process, optimization is possible.

Such a virtual machine is not more efficient by default. It has the potential to be more efficient. If we start with a fully optimized classic virtual machine and add support for JNI, we would have a better virtual machine. If we start with a fully optimized JNI-compatible virtual machine and add support for MPCL, we would have a better virtual machine.

Write and read

When one process writes a byte array to a native stream, the byte array is usually copied by a kernel. The kernel takes data out of a process by copying it. That is the purpose of the file subsystem API.

  public void example() {
	 FILE *fp = fopen( "x" );
	 fwrite( fp, buf );
	 fclose( fp );
  }

When one user process writes a byte array to a inter-process stream, the byte array is copied by the virtual machine. Somewhere in machine code, a virtual machine must be able to copy a byte array from one bytecode process to another.

A user process cannot copy a byte array to another process using System.arraycopy() because it cannot own (or obtain) a byte array from another process. But, something like System.arraycopy() is needed.

The solution is already provided in java.net.Socket, java.io.FileOutputStream and java.io.FileOutputStream. The write() method of an output stream and a read() method of an input stream enables a byte array to be copied from one bytecode process to another. A handle can be obtained for a file stream through a file descriptor. A file subsystem runs in a user process. It provides only a file handle (a long value) to other user processes. A handle is how an object should be proxied. Open a shared object, use a shared object, close a shared object. Such a stream can be buffered.

Summary

We must build shared objects from the bottom up. First, we must design a mechanism to pass primatives by value from one user process to another. The following primatives have exactly the same definition on all three Java platforms.

byte		byte[]
char		char[]
short	  short[]
int		 int[]
long		long[]
float	  float[]
double	 double[]

After a primative mechanism has been designed, it will support all manner of object sharing using traditional, off-the-shelf inter-process communication.
