Article contributed by GilbertHerschberger (28 March 2000).

Introduction

The process: scheme is part of the SmartAPI. The ProcessController is found in the org.jos.gchii.smartprocess1a package.

Like the run: scheme, the process: scheme invokes a JOS program. Unlike the run: scheme, the process: scheme creates a separate thread.

In fact, it creates a separate class loader. The new class loader creates a separate thread.

Backward compatible

The process: scheme itself is backward compatible. It does not require a custom virtual machine. Any off-the-shelf virtual machine will do. In an off-the-shelf virtual machine, the process: scheme returns a light-weight process. There is still a problem with static fields. There is still a problem with Runtime.exit().

  • The process: scheme always returns an instance of java.lang.Process or null.

    The parent program finds out more about a child process through the Process abstract class.

  • By the time a process is returned, it has already started running. The process: scheme starts the new thread.

  • The new program is fully identified by the run: scheme.

Example

Here is an example of the run: protocol to start the Find application:

run:program:org.jos.gchii.find2a.Find

Here is the process: protocol to start the same program in a separate thread:

<STRONG>process:</STRONG>run:program:org.jos.gchii.find2a.Find

The process: scheme works with any run: subscheme, such as jose:, os: and program:.

About JOS

In a custom virtual machine, like decaf, the exec() method of java.lang.Runtime can use the smart process components to build, start and return a heavy-weight process. For compatibility, the os: subscheme returns a heavy-weight process in an off-the-shelf virtual machine, and will continue to do so. A custom virtual machine gives a new process its own class path, its own class loader, its own thread group. For more information, see also the article called ProgramExec.

The smart process components provides the building blocks for both light-weight and heavy-weight processes. For compatibility with off-the-shelf virtual machines, the process: scheme must returns a light-weight process.

Within an off-the-shelf class library, the exec() method is native, linked to machine code. What should the machine code do inside JOS? Here is a pseudo-example from JavaNativeInterface.

jobject *Java_java_lang_Runtime_exec(
	jenv *env, jobject *runtime, jstring *cmd, jstring *environ ) {
  // create a custom class loader
  jclass *c = env->getClassForName( "decaf.CustomClassLoader" );
  jobject *new_loader = c->newInstance( cmd );
  jobject *process = new_loader->invokeVirtual(
		"getProcess(Ljava/lang/String;Ljava/lang/String;)V"
		cmd,
		env
		);

  old_loader = runtime->getClass()->getClassLoader();
  jobject *proxy = old_loader->loadClass( "decaf.ProcessProxy" );
  proxy->invokeVirtual(
		"setProcess(Ldecaf.CustomProcess;)V",
		process
		);

  return proxy;
}
