BytecodePages

Article contributed by GilbertHerschberger (31 January 2000).

Introduction

It is possible to run a bytecode verifier as a batch process. Such a verifier would be independent from any implementation of a virtual machine. A batch-oriented verifier would be easy to write. A modular verifier would be easy to expand. Such a bytecode verifier would verify each class file statically, generating a comprehensive list of all problems it detects.

A verification suite would provide a framework for building and plugging in your own verification tests. Anything you'd like to verify can be added without changing the architecture.

Verification

A class file is a binary file. It can contain anything. While it is supposed to contain highly structured bytes emitted from a javac tool, a class file can become garbled during storage or transmission.

A process of verification is used to make certain that the data inside a class file is reasonable. A verifier is used to prove that a javac tool is creating class files a virtual machine can understand. In addition, the virtual machine can take many things about a class file for granted, speeding up the interpretation of a class file at runtime.

Typically, verification is never performed on the standard class libraries (in java.* packages). The javac tool is trusted to create valid class files all the time. The bytecode that ships with many virtual machines is not verified by a virtual machine at runtime. The virtual machine expects you to trust the archives that came with it: (classes.zip and rt.jar).

Typically, verification is performed at runtime on classes that are downloaded at runtime, as in the case of browser-based applets. Verification of an applet makes sure a class is reasonable before it begins to execute on a virtual machine.

Repeately verifying an unmodified class file is inefficient. A verified class file only needs to be verified if it has changed or the verifier has changed. A static verifier might update the class file with an attribute, showing when the class was verified and which verifier was used. A robust virtual machine would only skip verification for classes with such a "Verify" attribute.
