SmartAPIPages

Article contributed by GilbertHerschberger (21 July 1999).

Introduction

Everything seems to be dependent on everything else. While I want to use an object factory, an object factory depends on many other smart APIs. You have to write a smart API before the system-wide object factory can use it.

On the other hand, the object factory can be configured for a smart API that hasn't been written yet, or one that doesn't exist on a specific machine. How's that? The addControllerFor() method of a smart factory does not throw an exception; it ignores any request it cannot fulfill. So you can configure your smart factory in anticipation of writing or installing a subsystem.

This is how we break the circular reference. This is how we can build device drivers and applications without having all these things built.

About class factory

Fundamentally, you must use classes when you use a Java virtual machine. These classes may be stored in different locations and in different file formats.

JOS needs a fundamental class factory. The class factory is a smart factory that uses a collection of class controllers. You can extend the class factory if you want by writing your own class controllers. So JOS is future-oriented. It is primed and ready for Java 3 and Java 4 and whatever direction Sun Microsystems wants to take the bytecode storage.

Controller, expecially for classes

The base class for all controllers that create classes is smartapi.ClassController. 

public abstract class ClassController 
	 implements Controller {
  // all class controllers are nested 
  // under the class: protocol
  public static final String PROTOCOL = "class:";

  public ClassController() {
  }
}

From classpath

The first kind of class controller is based directly on Class.forName(). If a class isn't on the classpath, this controller won't be able to load it.

public void SystemClassController
	 extends ClassController {
  private static final String SUBPROTOCOL = "system:";
  public boolean acceptsURL( String v ) {
	 Resource r = new Resource( v );
	 /* <A HREF="TeamTags#L">*L!</A>
	 if ( r.isProtocol( PROTOCOL ) ) {
		if ( r.isSubProtocol( SUBPROTOCOL ) ) {
		  return true;
		}
	 }
	 return false;
	 */
	 return ( r.isProtocol( PROTOCOL ) &&
				 r.isSubProtocol( SUBPROTOCOL ) );
  }
  public Object getObject( String v ) {
	 if ( !acceptsURL( v ) ) {
		return null;
	 }
	 String cn = new Resource( v ).getDocument();
	 if ( cn == null ) {
		return null;
	 }
	 try {
		return Class.forName( cn );
	 }
	 catch( ClassNotFoundException e ) {
		return null;
	 }
  }
}

Class factory

You don't have to worry about the configuration of the system-wide class factory when you write your own applications. It is automatically given a default configuration from a well-known key into the system-wide registry.

The default configuration for the class factory uses the SystemClassController. 

public class ClassFactory 
	 extends SmartFactory {
  public static ClassFactory getFactory() {
	 if ( factory == null ) {
		factory = new ClassFactory();
	 }
	 return factory;
  }
  private ClassFactory factory;
  protected ClassFactory() {
	 addControllerFor( "smartapi.SystemClassController" );
  }
  public addController( Controller v ) {
	 if ( v instanceof ClassController ) {
		super.addController( v );
	 }
  }
}

Using class factory

The following example shows how to create a button.

  public void example() {
	 String url = "class:system:java.awt.Button";
	 Class c = ClassFactory.getFactory().getObject( url );
	 if ( c == null ) {
		// maybe the AWT is not installed on this machine...
		return;
	 }
	 Button b;
	 try {
		b = (Button) c.newInstance();
	 }
	 catch( IllegalAccessException e ) {
		return;
	 }
	 catch( InstantiationException e ) {
		return;
	 }
  }

Using object factory

The following example also shows how to create a button. Instead of the class factory, it uses the object factory.

  public void example() {
	 String url = "object:class:system:java.awt.Button";
	 Button b = (Button) ObjectFactory.getFactory().getObject( url );
	 if ( b == null ) {
		// maybe the AWT is not installed on this machine...
		return;
	 }
  }

How does it do this? The object factory has a simple object controller. This simple object controller can use Class.newInstance() to create a new instance of any class. The primative object controller uses the class factory to create all of its classes.

If you use a Java virtual machine equipped with the reflection API, you may prefer to configure the object factory with a reflection object controller. This controller can use Class.getConstructors() to create a new instance of a class using any constructor. Some virtual machines don't have the reflection API. We may need to find some other way to create a new instance.
