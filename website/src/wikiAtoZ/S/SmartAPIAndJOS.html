SmartAPIPages

Article contributed by GilbertHerschberger (15 Jul 1999).

Introduction

I can imagine a distribution of Java based on the SmartAPI. Can you imagine it, too?

  • All of us would have the latest in Java packages created by all members of the Java community, including Sun. Everyone could participate and compete freely in the development of new APIs and frameworks.
  • We could reuse all the code that others wrote.
  • We could reuse all those file formats designed by others.
  • We could reuse all those protocols designed by others.
  • We could install components on a machine without recompiling.
  • For JOS, our virtual machines would be able to use the Java API verbatim--without re-compiling or re-writing--most of the classes developed by Classpath (or Sun).
  • There would be distinct (unmistakable) packaging for Java 1.0, 1.1 and 1.2.
  • With a smart API, Versioning is eliminated completely, with no need to fix it.
  • The world of AWT and Swing would be a lot different now. Both the AWT and Swing would participate in a common smart API for GUI components.
  • We would not need to upgrade from one virtual machine to the next to the next, with no end in sight.
  • We would not need to port our Java applications from one version of Java to another. We could count on APIs to be available on every machine.
  • We would not need to give up on making applets work in every deviant browser.

A time to follow

When a vendor makes something better, it might be time to follow. When IBM made SQL, QBE and the PC, maybe that was a good time to follow. When Microsoft made MS-DOS, maybe that was a good time to follow. When Sun Microsystems made Java, maybe that was a good time to follow.

How long?

How long do we have to follow them? It would be better for everyone if Sun would show the initiative to write a real Java API for Java programmers. It would be heavy on interfaces and light on implementations. It would not be the Java framework we have now.

Why isn't Sun moving toward a smart API?

If we really like Sun a lot, we can imagine that they just don't know any better. If they don't know any better, I might make up excuses them, at least for a little while. Eventually, my patience must come to an end.

Meanwhile, Sun Microsystems is making it more and more difficult to develop a Java application. Why are they making it more difficult? They won't fix glaring errors some of us have known about for three years (like no formal java.lang.Application interface). For more information, see also FixJava.

Do you think Sun is going to fix Java or not? Do you think everyone is going to drop Java 1.1 in favor of 1.2?

I hesitate to break away from Sun Microsystems. They are king of the hill. They are the current world champions. They have captivated our attention. They are doing some of the things that Microsoft should have done, but never got around to.

Showing leadership

Leadership is passed from one person to another as circumstances demand it. Leadership is transferred from one president to the next, one congress to the next, one govenor to the next.

Political, not technical

Sun Microsystems did not support the java.sql package on Java 1.0 for political--not technical--reasons. They wanted to strongly encourage everyone to upgrade to Java 1.1. Why? Because that's what's best for Sun.

Sun Microsystems isn't going to support the java.awt package over the long term, for political--not technical--reasons. They want to strongly encourage everyone to write for Swing. Why? Because it's cool? No. Because it's best for Sun.

Sun Microsystems isn't going to support Swing for Java 1.1 for political--not technical--reasons. They want to strongly encourage everyone to upgrade to Java 1.2. Why? Is it because Java 1.2 is that much better?

There are technical reasons for Sun Microsystems to do what they should have done. They should have made it possible to Write Once and Run Anywhere. They promised they would. Instead, Sun has made it possible to write once, per version, per browser and run anywhere in theory.

Keep in mind

It seems to me that JOS should use the smart API pattern in as many of our sub-projects as possible. We can create a smart API for many things:

  • applications - They need to be create-able by a smart factory. It is not necessary to install every application on every machine.
  • Java Beans - They need to be create-able by a smart factory. It is not necessary to install every bean on every machine.
  • device drivers - They need to be create-able by a smart factory. It is not necessary to install every driver on every machine.
  • components - They need to be create-able by a smart factory. It is not necessary for every component to be installed on every machine.
  • editors - They need to to be create-able by a smart factory. It is not necessary for every editor to be installed on every machine. It is not necessary for the JOS project to collect and redistribute every editor that's JOS-compatible. In fact, it is important for the long term to take the JOS project out of the loop as quickly as possible.
  • daemons - They need to be create-able by a smart factory. It is not necessary for every daemon to be installed on every machine.
  • registry - They need to be create-able by a smart factory. It is not necessary for every implementation of a registry to be installed on every machine.
  • objects - They need to be create-able by a smart factory. A smart object factory would be able to create any object from a class name and parameters.
  • documents - They need to be create-able by a smart factory. For more information on documents, see also your favorite HTML browser.

Uniform expansion

By correctly identifying everything in the operating system as a digital resource, we can benefit from an infinitely extendable operating system, using a smart API and uniform resource locators.
