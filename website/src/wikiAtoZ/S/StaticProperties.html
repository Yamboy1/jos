Introduction

A property is part of an object. Like a field, a property has a name and is a container for a data value. A property uses custom accessor(s) and modifier(s) instead of a field name or assignment operator.

Some properties are instance properties. Like instance fields, an instance property is closely associated with an object. An object, an instance of class, has its own container to hold a data value.

Some properties are static properties. Like static fields, a static property is closely associated with an object's class. All objects share a single container to hold a data value.

The Java programming language uses properties as a convention. It is not part of the language. The names of a method can be interpreted as an accessor or modifier. The getXXX() method is expected to be an accessor, used instead of a property name. The setXXX() method is expected to be a modifier, used instead of a property name.

Unlike the Java programming language, Object Pascal (Delphi) provides a property keyword. The property keyword in Object Pascal enables a class to override the accessor and modifier for a property. Elsewhere, a custom accessor is automatically used when a property name is an L-value. And elsewhere, a custom modifier is automatically used when an assignment is made to a property.

Premise

The rest of this article is based upon the following premise.

All classes have static properties.

Static properties are part of the real complexity of distributed computing. As a container, an object can include more than just one instance of a class.

Objects

An object in Java has a well-known interface defined by the java.lang.Object class. What are the properties of this class and how many of its properties are static? Using the JavaDoc page for java.lang.Object as a reference, we can identify the visible properties of an object.

  • getClass()

    This method qualifies as a property because (1) it takes no parameter and (2) it returns a value. The name of the property is Class because this method follows the getXXX() naming convention.

    This property is not static. The static keyword is missing from the signature of this method. Each instance of java.lang.Object has-a unique class.

  • hashCode()

    This method qualifies as a property because (1) it takes no parameter and (2) it returns a value. The name of the property is HashCode because this method does not follow the getXXX() naming convention. getHashCode() would have been a better name for this method.

    This property is not static. The static keyword is missing from the signature of this method. Each instance of java.lang.Object has-a unique class.

  • equals()

    It would be a stretch of the rules to say that Equals[] is a property of an object. And yet, this method qualifies as a property because (1) it takes only one parameter and (2) it returns a value. This method does not follow the getXXX() naming convention.

  • clone()

    It would be a stretch of the rules to say that Clone is a property of an object. And yet, this method qualifies as a property accessor because (1) it takes no parameter and (2) it returns a value. This method does not follow the getXXX() naming convention.

  • toString

    This method is a conversion method. It converts an object to a string. A conversion method should follow the toXXX() naming convention.

  • notify()

    This method cannot be a property accessor method because (1) it returns no value. It is not a property modifier because (2) it takes no parameter.

  • notifyAll()

    Like notify(), this method cannot be part of a property because (1) it returns no value and (2) it takes no parameter.

  • wait()

    It would be a stretch of the rules to say that Wait is a property of an object. And yet, this method qualifies as a property modifier because (1) it returns no value and (2) it takes one or two parameters. This method does not follow the getXXX() naming convention.

  • finalize()

    Like notify(), this method cannot be part of a property because (1) it returns no value and (2) it takes no parameter.

An instance of java.lang.Object has no static properties. Or does it?

Experiment

Compile and run the following code. What output does it produce?

package org.jos.demo;
public class Demo {
  public static void main( String[] args ) {
	 int iMax = 10;

	 System.out.println( "Object" );
	 Object o = new Demo();
	 for ( int i = 0; i < iMax; i++ ) {
		System.out.println( "" + o );
		o = o.getClass();
	 }
	 System.out.println( "" );

	 System.out.println( "Class" );
	 Object o = new Demo();
	 for ( int i = 0; i < iMax; i++ ) {
		if ( o == null ) {
		  break;
		}

		System.out.println( "" + o );
		Class c = o.getClass();
		o = c.getClassLoader();
	 }
	 System.out.println( "" );
  }

  public Demo() {
  }
}

Proof

Neither object nor class has static properties, right? Wrong! The java.lang.Object and java.lang.Class are a special case. They are part of the fabric of the Java platform. They provide a model of the relationships between object and class. They cannot be examined as if they were just like other custom classes. While these model classes have no explicit static properties, all classes have static properties.

As we have seen, all classes have instance properties. Instance properties of a class are static properties of some object. A class is an object, too. Instance properties of a class object are static properties of some class. Therefore, all classes have static properties.

About MPCL

A MPCL-compatible virtual machine has a difficult job to do. It must provide a comprehensive environment for a bytecode process. A bytecode process must run as if it were running on a separate virtual machine, inside a machine code process, on a foreign operating system. A part of the puzzle may be solved by multiple primordial class loaders.

Summary

MPCL has been introduced into virtual machine design. It extends the specification of a virtual machine.

What are its implications?

An MPCL-compatible virtual machine might provide a vm-wide address space, a vm-wide garbage collector, a vm-wide JIT compiler, a vm-wide class verifier, a vm-wide bytecode/class definition cache. In theory, it can provide multiple bytecode processes, as if each process were running in its own address space, with its own garbage collector, with its own class verifier, with its own bytecode/class definition cache, inside its own machine code process, on a foreign operating system.

Isn't MPCL about sharing objects?

No, quite the contrary. It was introduced into virtual machine design as a new persistence barrier so that -- as a rule -- objects cannot be shared. Each bytecode process has its own "address space" and runs independently. MPCL was designed to prevent the arbitrary or automatic sharing of objects.

While MPCL puts up a barrier, an object-sharing mechanism is expected. An object-sharing mechanism needs to define when it is safe for objects to break through. An object-sharing implementation must tightly control object-sharing between two or more bytecode processes.
