PlatformAPIPages

Article contributed by GilbertHerschberger (7 April 2000).

Introduction

With the introduction of BytecodeResource, it becomes possible to link bytecode to a virtual machine. Linking bytecode to a virtual machine simplifies the startup sequence.

As a resource, bytecode is stored in an array of octets. Bytecode is given a Bytecode interface, not a I/O stream interface. It does not need to be stored in a class file. Without class files, a virtual machine only needs its configuration and dynamic bytecode as an external resource.

Everything that can be written in bytecode should be written in bytecode. Write a class in bytecode to read and write the zip file format. Machine code is not required. It is not necessary to store bytecode in classes.zip (or rt.jar).

One file

The virtual machine and its boot package should be linked into a single boot image. Here is a general outline of this procedure:

  1. A virtual machine goes into a static link library.

  2. Each boot package goes into its own static link library.

  3. The virtual machine and its boot packages are statically linked.

  4. The executable image is converted into boot image format.

Alternative boot packages

By changing link options, a virtual machine can use alternative boot packages.

  • JVM-specific classes

    JVM-specific class should be statically linked to a virtual machine. They are, after all, specific to a virtual machine.

  • Core classes

    The following packages should be statically linked, too.

    • java.io
    • java.lang
    • java.net
    • java.util

Whole packages

If one class in a package needs to be statically linked, the whole package must be statically linked. All classes in a boot package are inseparable.
