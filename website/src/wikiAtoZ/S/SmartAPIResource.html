SmartAPIPages; ObjectFactoryURLClass

Article contributed by GilbertHerschberger (28 July 1999).

Introduction

It puts a new perspective on what you should expect from an operating system. With the SmartAPI, your programs can do things that other programs can't do. With this approach, JOS can do things that other system programmers can only dream about.

resource
A resource takes on a new dimension. A resource is any digital resource. Both a document (object) and a program (verb) are digital resources. Both deserve the same kind of treatment. With the SmartAPI, they are treated like digital resources, and should be.

Release 1a

The first release of the resource package provides interfaces today where there will be classes tomorrow. These interfaces need to be implemented to do what has been described.

  • URI

    The URI class provides a very nice constructor to build a URI from any string. And it also provides a getObject() method that connects directly to the smart object factory. You can get any digital resource you want, something like this: 

      public void example() {
	 String s = "any://JOS/compatible?uri#string";
	 CustomClass c = (CustomClass) new URI( s ).getObject();
  }

Hang on to your seat!

When you have the smart object factory at your command, imagine the kind of text editor you could build. You can download the code and build it today, too.

Throw out the traditional File | Open dialog box. Put up a location field. When someone wants to edit a text file on any machine in an intergalactic environment, they should type in its URI.

Instead of your obsolete text editor giving you a choice of local files (and only local files), you can open any file. If you have a network (Internet/intranet), you can bring text file directly into your editor through HTTP and FTP.

Because the smart object factory is so smart, you can create your own controllers to download a text file from any format: past, present and future!

Revolutionary

I don't like the word revolutionary. People use revolutionary far too often. I have to take exception with the potential for JOS. It is nothing short of revolutionary.

When you combine the massively dynamically linked model of the JVM and its bytecode loader with well-known interfaces, the very large namespace, and the advantage of the SmartAPI, you can do things easily that cannot be done on other platforms.

Who would have thought? You can display any text file using a stream tokenizer, too. You can enter a list of special characters the stream tokenizer should use on a file and see the results--instantly.

in:token:file:/usr/local/example?delim=%20%0a%0d

Think about that the next time you're trying to determine the list of delimiters to parse an HTML file, a Java source file, or a .conf file.

Beyond cat

Let's think about the cat command. When you use the cat command, the cat command should be able to display any file, too. The cat command should be able to display any file from anywhere. If the cat command uses the smart object factory, it should pass a URI string to the object factory. For arguments without a scheme (protocol), the cat command should prefix the request with in:data:file:, convert the given string to a platform-independent form, and use the smart object factory.

In fact, the cat command is easier when the data: sub-protocol is used. Each line of a text file is read from the input stream with readLine().

Should the cat command also display one token on each line when I use the token: subprotocol? Yes, I should think so.

Get ready

You can lower your expectations or raise them. You can put up with a poor quality operating system, or not. But, as for me and my next OS, we will be URI-based and future-oriented. Would you care to join us?
