Introduction

A secret namespace is harmful. It can severely reduce the quality of your software. It is hard enough to work with hundreds of thousands of components using a visible namespace. Accuracy itself may become impossible using a secret namespace.

Why does Sun Microsystems promote a secret namespace?

Programming in-the-large

Java was designed from the ground up with all of the power to do programming in-the-large. The key element of this power comes from Java's very large namespace. There are classes in packages and packages in packages. They can be nested many levels deep. Java 1.0 has all of the namespace you'll ever need, right?

Distributed computing

Java has the power to do distributed computing like no platform before it. With a large namespace and a java.net package, personal distributed computing is within our grasp. It is possible to build dependable, long-lasting components in a distributed computing product with no need for incompatibility.

Recommendation

Java provides namespace as an appropriate hierarchy. Sun gives us a strong recommendation for keeping everything straight. You must not invade another organization's namespace. When our website is jos.org, our assigned namespace is org.jos. It is clean, simple and works.

Secret namespace

If you can't see it, it's a secret. If you make it hard for someone to find out, you're keeping it secret whether you intend to or not.

If you keep it a secret, it is a mystery. While you may want mystery in a cinema thriller, you don't want mystery in your business application. You don't want to hide something that is so important to your customers and administrators. Hiding it is annoying. You want your customers to use the software you write, yes?

Sun has given Java a secret namespace. They say it is because the very large Java namespace isn't large enough. You can't see a version. A version cannot be seen through the OS. It is hard to find out about a version. It is hidden away from all the people that need to know. It is a mystery why we need versions anyway.

With such arbitrary complexity, many administrators will give up, pack it up and go home. They would rather wait for the next platform. They hope it will be simpler.

It was simple when there were classes in packages and packages in packages. But now, there are classes in versions and packages in versions and versions in packages.

It was simple when there was a direct map between packages and directory names. It was a clean map. All packages could be stored in the OS file subsystem. But now, the map is lost. The map is muddied. All packages cannot be stored in the OS file subsystem at the same time. Why? Because version is ignored in a directory name.

With the introduction of Versioning, Sun has invented yet another solution for programming in-the-large. They have invented the secret namespace. The secret namespace is the insertion of a version number in a package that's not reflected in the OS file subsystem.

Debugging

Java was designed from the ground up with all of the power to do programming in-the-large. The key element of programming in-the-large is debugging in-the-large. The original Java provided a clean map from a class back to its source code. A symbolic debugger could should you the lines of source corresponding to the bytecode that's running.

The secret namespace is not reflected in the source code for Java. The debugger loads whatever file is in the directory. How should it know which version of the class you're running?

Versioning makes it difficult, if not impossible, to debug the application that's running. It anticipates, expects and assumes everyone practices editing-in-place. You are going to overwrite your source code when developing a new version, aren't you?

Where are you going to go when the application doesn't work? And as you know, the application won't work on the first try. A secret namespace makes the administrator's job impossible. They'll be pulling their hair out, swearing about the day someone talked them into putting Java on the network.

For more information on debugging and distributed computing, see also DistributeAndDebug.

Cascading effect

When programming in-the-large, you must be prepared to load every class available. When working on a large program, you want to eliminate the possibility of loading a bad class. Squashing a bug may require you to replace a bad class with a good one. Which class are you going to replace? How do you replace every instance of the class, if you're going to support more than one? How many times do you want to fix the same bug?

The secret namespace has a nasty cascading effect. We have seen bugs that won't die in distributed computing. We take out a bug, restart the system. A few days later the bug is back. What happened? When we restart the system, of course we can't restart thousands of machines all at the same time. By the time we shutdown the last machine, the first machine has already been reinfected.

A bug goes away only if you make it impossible for the class to come back. It means that every class must participate in a single namespace, with no secrets.
