Introduction

This article illustrates how an SQL service can provide bytecode for a class loader.

Design

The following design is an oversimplified example of a function to download bytecode from an SQL service. It assumes a pre-existing, permanent JDBC connection.

  public byte[] findBytecodeByName( String cn ) {
	 return findBytecodeBySpec( cn.replace( '.', '/' ) );
  }
  public byte[] findBytecodeBySpec( String cs ) {
	 String sqlStr =
"select PAYLOAD from JOSDEMO.JAVA0" +
" where CLASSSPEC=" + sqlChar( cs );

	 ResultSet rs = performSQLex( sqlStr );
	 if ( rs == null ) {
		return null;
	 }

	 byte[] buf = null;
	 try {
		if ( rs.next() ) {
		  buf = rs.getByteArray( "BYTECODE" );
		}
	 }
	 catch( SQLException e ) {
	 }

	 try {
		rs.close();
	 }
	 catch( SQLException e ) {
	 }

	 return buf;
  }

Armed with only its boot packages and an SQL service on another machine, a virtual machine can use this SQL-based bytecode provider to download other packages. It can download packages from the Java standard class library (java.) and standard extensions javax..

Previously, a JAVA0 table was created in the JOSDEMO database.

  create table JOSDEMO.JAVA0 (
	 CLASSSPEC varchar(128),
	 PAYLOAD varchar(32000),
	 PRIMARY KEY (CLASSSPEC)
	 )

And, class files were inserted into the JAVA0 table.

  insert into JOSDEMO.JAVA0
  (CLASSSPEC,PAYLOAD)
  VALUES ?,?

When a PLATFORM column is added, classes for all Java platforms and their variations can be stored together in a single SQL table.

Bytecode provider

A bytecode provider is not a class loader. It always returns raw bytecode. It might be used by a class loader. A bytecode provider can also be used by a bytecode analyser, a static bytecode verifier and other development tools.

The javac tool could be re-implemented as a bytecode provider. When given a class name or class spec, even a compiler must produce a byte array.

Package provider

With package-oriented tools like ERIC, the smallest unit of bytecode is a package. While a package provider always loads one package at a time, it implements the bytecode provider interface.

When an SQL bytecode provider is package-oriented, it would execute only one query for each package, not each class. While the package name and class might be stored in a single field, the query would retrieve classes for exactly one package. The where clause would be something like this:

  public void example( String cn ) {
	 String pn = cn2pn( cn );

	 String sqlStr =
"select PAYLOAD from JOSDEMO.JAVA0" +
" where CLASSSPEC like " + sqlChar( pn + ".%" ) +
" and not CLASSSPEC like " + sqlChar( pn + ".%.%" );
	 ResultSet rs = performSQLex( sqlStr );  
	 :
  }
