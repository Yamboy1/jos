Introduction

The semantics surrounding System.exit() must be preserved inside of JOS.

The lifetime of a process looks like this:

  1. A process is started, kicked-off by the kernel.
  2. A process runs for a while.
  3. When a process stops, it returns an exit code. A process stops itself when it runs out of things to do. A process is asked to stop by the kernel.

It is possible for JOS to go where no operating system has gone before and do something else entirely. But, there are good reasons for a process to return a result code when it stops. The tradition of process and sub-process should continue.

  • Chaining A parent process can start a child process, in such a way that a child process completely replaces the parent. The result code of the child process becomes the result code of the whole process.
  • Calling A parent process can start a child process, so that the parent waits until the child is finished. It is very important to the parent to know the result of the child process.
  • Spawning A process can start another process. The flow of control has just forked, so both processes are running in parallel. A process doesn't know the result code of its peer.

When running processes in a batch through a shell, it is important for the shell to find out the result code of each child process. A scripting language may allow for the script to evaluate the result code to figure out what to do next.
Preservation

How does a Process object preserve the semantics of a call to a static System.exit()?

A custom Process should be able to call System.exit() with a final result code. At this point in a process, all of its threads should be stopped, all its objects should be garbage collected. And then, the result code should be returned to the calling process.

What about the main process? The main process could call something like System.exit(), too. In this case, the system should (0) halt, (1) reboot, or (2) re-launch a new main process.

  1. Shutting down the computer gracefully should be supported by JOS. If there is an API for system_shutdown(), such a request should be passed to the main process, which calls its System.exit( 0 ).
  2. Rebooting the computer gracefully should be supported by JOS. If there is an API for system_reboot(), such a request should be passed to the main process, which calls its System.exit( 1 ).
  3. Relaunching a main process should be supported by JOS. Relaunching a new main process should be performed after an upgrade of any system class, after any configuration change that requires a new main process. If there is an API for system_recycle(), such a request should be passed to the main process, which calls its System.exit( 2 ). Recycling does not reload the JOS kernel. It only restarts the stuff written in Java.

Relaunching any process should be supported by JOS. When any custom Process changes its own configuration, it should be able to relaunch itself (with preservation of the original command line).
