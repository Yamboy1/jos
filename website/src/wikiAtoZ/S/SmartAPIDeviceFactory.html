SmartAPIPages

Article contributed by GilbertHerschberger (2 August 1999).

Introduction

I'll get right to the point. If we have a device interface, we should create a DeviceFactory.

Device interface

Suppose we have a device interface, like this:

public interface Device {
}

Suppose this interface is implemented by any JOS-compatible devices. We can create a device factory to load any kind of device. Each kind of device needs its own special-purpose interface, like this:

public interface CustomDevice
	 extends Device {
  public void customMethod();
}

A pointer device, for instance, is capable of supporting an entirely different set of methods than, say, a CD-ROM drive. Why should the pointer device implement methods required by a CD-ROM?

Factory

A smart device factory should be capable of creating any kind of device from a URI. While the smart factory provides devices to the device manager, it dispatches all requests to its controllers.

You can configure the smart device factory to include or exclude different kinds of devices. If you want to experiment with a new kind of device in JOS, you re-configure your device factory and load the new device. You may not have to restart your OS or your computer.

The configuration of the smart device factory is an entirely different issue from configuring the devices to install automatically at boot-time. The configuration of your smart device factory determines which drivers can be installed. The boot-time configuration determines which drivers your OS will attempt to install automatically.

The interaction between the two configurations is this. If your boot-time configuration attempts to install a device that is not supported by your smart device factory, the attempt fails. Your OS won't halt if a device is not loaded automatically at start-up. It skips over any device that is not "installed".

It also means that you can attempt to install devices when bytecode for the driver is not installed on your machine. When you install the bytecode, the device starts working.

Controllers

A controller should be capable of creating one kind of device from a JOS-compatible URI. Each controller is plugged into the smart device factory.

This is future-oriented. New kinds of devices can be added. All you need is a new controller. Add the controller to the configuration of your smart device factory.
