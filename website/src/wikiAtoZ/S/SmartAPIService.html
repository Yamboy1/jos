ServicePages; SmartAPIPages

Article contributed by GilbertHerschberger (16 October 2000).

Introduction

The Smart API Service is the cornerstone of an operating system for Serverfarm-in-a-Box(tm). It enables a web of services on a farm of server machines.

Protocol

The network protocol for this service is a lot like TELNET. A TELNET-compatible client is useful for invoking digital resources within this service.

Scheme

The x-smartapi: scheme supports the Smart API Service. It acts like a TELNET-compatible client. Here is the syntax diagram for this scheme. 
x-smartapi: 
  [ '//' [ <user-name> [ ':' <password> ] '@' ]
  <machine-name> [ : <port> ] '/' ]
		<location>
The default machine for this service is localhost. The default port for this service is 8023.

Location

The location inside the x-smartapi: scheme is any string. A uniform resource identifier/locator is expected.

This service may be configured to dis-allow any location for any account. One account might be able to invoke an location that another account cannot invoke.

By default, there are few restrictions on programs invoked inside this service. A program can read/write all files on the server machine. A program can be based on AWT or Swing and create a window (frame or dialog box). Any program can shut down the service by invoking System.exit().

Examples

Here are a few examples of a uniform resource locator that uses the Smart API Service.
  • x-smartapi:file:demo.txt - Contents of a file are returned relative to a document root directory. A document root is determined by service configuration.
  • x-smartapi:/usr/local/share/demo.txt - Contents of a file are returned relative to a false root directory. A false root is determined by service configuration.
  • x-smartapi://demo.jos.org/file:demo.txt - Contents of a file are transferred from a computer called demo.jos.org.
  • x-smartapi://demo.jos.org/http://www.jos.org/ - This location instructs demo.jos.org to download a file using the HTTP protocol. In turn, the content of this file is transferred from demo.jos.org.
  • x-smartapi:applet:org.jos.demo.DemoApplet - This location displays an applet on this machine. An applet browser is automatically invoked.
  • x-smartapi://demo.jos.org/applet:org.jos.demo.DemoApplet - This location displays an applet on demo.jos.org, another computer. An applet browser is automatically invoked.

More examples

The most exciting part of this service may be its subprogram interface.
  • x-smartapi:run:os:start.exe notepad.exe - Any machine code program can be invoked on this or another machine. This location invokes notepad.exe program from within a Smart API Service.
  • x-smartapi://demo.jos.org/run:os:start.exe notepad.exe - This location invokes notepad on a machine called demo.jos.org.
  • x-smartapi:run:program:org.jos.gchii.find2a.Find - Any subprogram can be invoked on this or another machine. This location invokes the Find utility.
  • x-smartapi://demo.jos.org/run:program:org.jos.gchii.find2a.Find - This location invokes the Find utility on a machine called demo.jos.org.

Serverfarm

One machine is designated as a security server. This is where the serverfarm-wide configuration is stored. The security server is one of the few machines with a physical drive, such as an SCSI- or IDE-compatible hard drive. This is where system administration functions are performed. The security server is not required to run the Smart API Service.

Whenever a machine is connected to the serverfarm, the security server runs an appropriate initialization script. Other servers in the farm run the Smart API Service automatically upon boot up. The security server uses the x-smartapi: scheme to configure additional machines on the subnet.

The security server software can restart a Smart API Service when its configuration changes. This can be tied to load-balancing on a large serverfarm, to dynamically start daemon processes on other machines as needed.

Security

A serverfarm-in-a-box has unique security needs. The whole farm acts as if it were one computer. Signing into the security server is like signing in to all machines in the farm. All machines in the farm are in close physical proximity. The farm is isolated by a firewall. Each farm has a unique IP subnet.
