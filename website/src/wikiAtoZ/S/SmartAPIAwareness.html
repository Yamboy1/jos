SmartAPIPages

Article contributed by GilbertHerschberger (31 August 1999).

Introduction

In the spectrum of aware of to depends on, let's talk about aware of. This article describes a loosely-coupled class library when programming in-the-large.

Awareness

When one class library is aware of another class library, the relationship is an optional dependency. Given two class libraries, A and B, A is aware of B when A can use classes from B if they are available; A continues to work correctly with limited functionality even without B.

At the introductory level, an interface is a example of awareness. A requestor is aware of an implementation indirectly through an interface. The interface is one of the more important strengths of an platform. As its strength, the Java platform formalizes an interface to an object by implementing a special kind of class, literally called an interface.

Pushing the envelop

The Java pushes the envelop. The Java platforms enable more application programmers to work with programming in-the-large. This kind of programming uses far more class libraries than ever before. Each class library in Java is a collection of classes and called a package. How are class libraries connected? At this level, two or more class libraries may be tightly- or loosely-coupled.

  • tightly-coupled

    When two libraries are tightly-coupled, both libraries must exist for either library to work. Tightly-coupled class libraries are the norm for building applications. As a rule, tightly-coupled reduced the potential for runtime software reuse.

  • loosly-coupled

    When two libraries are loosely-coupled, one library works even when the other is missing. An architect and system programmer search for loosely-coupled class libraries. They increase the potential for building applications through runtime software reuse.

About JOS

The JOS Project is building an operating system. We are building an object-oriented API. Since we're building a system, our class libraries should be loosely-coupled. Direct dependencies between class libraries should minimized.

Coupling

There is direct and indirect coupling. To illustrate the difference between direct and indirect coupling, a layout manager will serve as our example. A layout manager is part of the AWT class library.

A class is directly coupled to another class library when it uses the new keyword, like this:

  LayoutManager m = new CardLayout();

A class is indirectly coupled to another when it uses dynamic class loading, like this:

  String s = "java.awt.CardLayout";
  LayoutManager m = (LayoutManager) Class.forName( s ).newInstance();

Example: layout managers

Think about a layout manager and all its potential implementations. The layout manager is one of the strengths of the AWT class library.

The following example demonstrated direct coupling: 

import java.awt.CardLayout;
import java.awt.Panel;
public class CardPanel
	 extends Panel {
  public CardPanel() {
	 setLayout( new CardLayout() );
  }
}

Note: This is the kind of code you write in an application, and should.

The following example demonstrates indirect coupling: 

import java.awt.LayoutManager;
import java.awt.Panel;
public class CardPanel
	 extends Panel {
  public CardPanel() {
	 String s = "x-layout:awt:card" );
	 setLayout( (LayoutManager) new URI( s ).getObject() );
  }
}

As you know, CardLayout is inside the java.awt package. When an application is dependent on java.awt it is dependent upon the whole package.

So, what's the difference?

In a loosely-coupled class library, CardLayout does not have to be inside the java.awt package. In fact, it doesn't have to be inside the java.* packages. With the loosely-coupled class library, well-known interfaces are in a core set of packages; implementations of a well-known interface are elsewhere.

Third-party software vendors should provide compatible replacements for the standard AWT layout managers. End-users should plug in alternative layout managers to use with their applications. In a bean box, you should select any layout manager configured for your machine. Like a pluggable look-and-feel, layout managers should have been in a separate class library.

Maybe you can come up with a better example. This just one example of the kind of thing you should be able to do in reusable object-oriented software. You should be able to replace the implementation of (almost) any class library.

About SmartAPI

The SmartAPI class library is an indirect link between class libraries. This design provides a methodical way to design libraries that are as loosely coupled as possible.

Interfaces should be kept together in core packages. SmartAPI is one of the core packages. All other reusable class libraries should minimize their dependency on other class libraries by using the SmartAPI where appropriate.
