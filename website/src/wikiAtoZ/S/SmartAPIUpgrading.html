SmartAPIPages

Article contributed by GilbertHerschberger (23 July 1999).

Introduction

For those of you who haven't yet had the opportunity to work with a smart API, this example should help explain how you can upgrade your application to take advantage of the latest and greatest class libraries available.

Scenario

Here is a typical scenario for a text editor application. Imagine I have written a tiny text editor.

About version 1

The first version of the text editor was created long before I installed the SmartAPI. So the text editor was complicated with code to open text files in a platform-independent way. Meanwhile, the editor works. I wrote a lot of imaginary text with my imaginary text editor.

Download SmartAPI - Release 1a

I downloaded the first release of the SmartAPI package. It took a while to get familiar with the way the object factory can create any object. It took a while to understand that my text editor could use the stream factory to open text files.

About version 2

Moving from version 1 to version 2 was a major revision. There were things built into the stream factory that went way beyond editing local text files. By typing in an URL starting with http:, I can bring any HTML file into my text editor. I didn't have to know anything about Internet/intranet client/server programming to do it either.

Download SmartAPI Add-ons

After I finished version 2, a minor bug-fix was posted on the JOS website. I just installed the patch and add-ons. Without recompiling my editor, it benefits from patches and add-ons. I can distribute the patch to my customers.

Download SmartAPI - Release 2

A major rewrite for the SmartAPI is posted on the JOS website. It was necessary to change some of the class names and signatures of public methods. Do I want to use it, or not?

I download and install release 2. Release 2 does not overwrite release 1. I can experiment with release 2. I can learn about the changes to release 2. I can see if it makes sense to change my text editor to take advantage of the refinement.

Because both release 1 and 2 are stable, self-contained packages, I may skip a generation of the SmartAPI if it pleases me.

I can compare, on a single machine, the performance of releases 1 and 2. My text editor keeps on working. During all of the time I'm considering the impact of release 2, I have no business interruption.

And so it goes.

Ten years later...

We're up to Java 14 now. JOS has cornered the market for high quality operating systems. I still have a few hundred 1000MHz processors in my antique collection. Bill Gates has retired to his own South Pacific island. IBM bought out what's left of Microsoft's struggling software division. The Year 2012 glitch has made the headlines.

While I'm working on release 47.3 of my favorite text editor, I might notice that I have no applications that depend on SmartAPI - Release 1. What now? I can delete SmartAPI - Release 1 if I want to. Since deleting it will saving 50K on my 1000 terabyte drive (cost $US100), it hardly seems worth the effort.
