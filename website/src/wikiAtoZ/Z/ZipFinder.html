SmartAPIPages

Article contributed by GilbertHerschberger (4 October 2000).

Introduction

This finder was introduced in release 14 of the SmartAPI. Like any finder, it uses AsynchronousIO to find files. This kind of finder reads a ZIP-compatible archive. It reads both .zip and .jar files.

Although directories may not be stored inside an archive, this finder dispatches a directory event as if they were.

Finder interface

This class implements the Finder interface found in org.jos.finder1a.

Like basic finder

This class is like the BasicFinder in the org.jos.gchii.find2a package. In fact, this class depends on on that package for matching wildcards in file names.

Controller

Instances of this class are available throught SmartAPI, a multiple-stage object factory. The ZipFinderController processes a zip: subscheme for the x-finder: scheme. Here are a few examples of valid locations.

  • x-finder:zip:[file:src.zip]?show=all

    With show=all, this location displays all directories/files within the src.zip archive.

  • x-finder:zip:[file:src.zip]?pattern=*.java&show=all

    With a pattern=*.java, this location displays only files that end with .java.

  • x-finder:zip:[file:src.zip]org/jos/finder1b

    A directory (path) is written after the closing bracket (]). All files in the org/jos/finder1b directory are displayed. The first slash in a directory name is optional. Note: The file separator is always a slash (/) inside an archive.

  • x-finder:[file:src.zip]?show=all

    When using brackets ([]), the zip: subscheme is optional.

zip: subscheme

When a ZIP-compatible file has a .zip extension, the zip: scheme is optional.

  • When optional

    A ZIP-compatible archive is implied when an archive file has a .jar or .zip extension.

    x-finder:[file:src<STRONG>.zip</STRONG>]?show=all
  • When required

    A ZIP-compatible file can have any extension. The ZipFinder can be explicitly used by entering the zip: subscheme.

    x-finder:<STRONG>zip:</STRONG>[file:src.bak]?show=all

Note: The ZipFinder implementation inside release 14 of Smart API is not quite correct. It mistakenly opens all archives as if they were ZIP-compatible archives. A new implementation is required.
