Article contributed by GilbertHerschberger.

Introduction

Browser security has gotten stupid. It has gotten way out of hand. The Java-enabled browser is crippled by an inappropriate security policy. When a security policy prevents people from doing legitimate work, it is inappropriate. I want a security policy to reflect the needs of an intranet or stand-alone browser, not just an Internet browser.

It has often been said and often repeated that it is dangerous to download programs from the Internet. You run the risk of downloading a program from someone who doesn't have your best interests at heart. They might write a virus to wipe out your hard drive. Oooh. That would be bad.

This isn't quite right.

  • It is dangerous to run any program. You should be always concerned about all new software. It doesn't matter if it was provided by friends or strangers. What is the difference if its a bug or a virus that takes out your hard drive? The results are the same.
  • Programmers, who write their own software, are at the greatest risk. It is dangerous to run a native compiled program for the first time, even if you wrote it. When you're a beginner programmer, it is dangerous especially because you wrote it. ;)
  • Installing software has little to do with the Internet. It has little to do with downloading. The most dangerous program I have ever seen came from a brand-name CD-ROM I bought at the local computer store!
  • The only real protection against a virus is a backup. You're living dangerously if you're running virus protection software and have no backup. A backup is the only way to recover if the hard drive is lost for any reason, including lightning, virus, or a bug.

Isn't this a double standard? While Sun Microsystems makes it difficult for legitimate Java programmers to download their Java bytecode to a browser, they recommend you download and install their native compiled software off the Internet. But wait! It is dangerous to download programs off the Internet. Doesn't that include programs from Sun?

The Java browser add-in has an automatic update feature. It downloads native compiled programs off the Internet. But wait! That's dangerous, isn't it?

Tightening

The security manager won't allow me to run a program that I wrote on my own computer. The HTML page is stored on my computer. The applet code is stored on my computer. A class loader is considered to be so dangerous that you're not allowed to create one in a browser. Sun Microsystems closely associates bytecode security with the class loader so the most powerful feature of Java has been crippled by concerns about security.

I would like my browser to let me type in the domain names of websites I trust. While there are legitimate concerns about running a browser on the Internet, what about an intranet or stand-alone computer? The security manager doesn't distinguish between file: and http: protocols.

The security policy for Navigator 4.0 is tighter than security policy for Navigator 3.0. Navigator 4.0 won't let you read and write files on your own computer when the bytecode is stored on your own computer.

A reasonable policy

I want a reasonable security policy. It should reflect the needs of an Internet, intranet, and stand-alone browser. When browsing the Internet, my browser should ask me if I would like to trust Java bytecode based on domain name.

My browser shouldn't ask me about any Java bytecode found through the file: protocol. My browser shouldn't ask me if I would like to trust Java bytecode I explicitely installed on my computer. Do you think I want to run the Java bytecode if I took the time to install it? Duh.

About JOS

Bytecode factories should be trusted, not individual classes. The class manager has a responsibility to create a java.lang.Class object for trusted code. Where does trusted code come from? Trusted code must come from trusted bytecode factories. It should be the bytecode factory that's trusted and not the individual classes loaded by a factory.

The class manager should trust bytecode from a factory that trusted. The class manager should sandbox bytecode from a factory that's sandboxed. I should be able to configure my browser so that it trusts ALL code loaded by a bytecode factory. All other bytecode factories are sandboxed.

It begins to make sense that the class manager should have a place for one trusted bytecode factory and a place for one sandbox bytecode factory. The security policy should enable trusted bytecode to read and write files, etc.
