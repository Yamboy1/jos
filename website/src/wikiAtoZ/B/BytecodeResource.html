Article contributed by GilbertHerschberger (7 April 2000).

Introduction

Linking bytecode to a virtual machine is possible and desireable. It is possible because any binary data, such as a graphic icon, can be linked to a machine code executable image and a virtual machine is a machine code executable image.

It is desireable because a virtual machine and its core classes are inseparable. The rt.jar or classes.zip from one virtual machine does not always work with another.

It is also desireable because a virtual machine with bytecode resources can load enough of the core to download additional bytecode from across the network.

Factory

The BytecodeResource object is a multiple stage object factory. It always returns an instance of Bytecode. To get an instance of Bytecode from machine code, invoke the getBytecode() with a class name.

void example() {
  CharBuffer cn = "java.lang.Object";
  Bytecode *bc = BytecodeResource::getBytecode( cn );
  :
}

The BytecodeResource class maintains a vector of package objects, one object for each package. When asked for bytecode for a specific class, the name of the class is passed to each package.

About java.lang.Object

Let's look to the java.lang.Object class as our first example. This class is critical to the operation of a virtual machine. It is so closely coupled with the virtual machine that switching from one classes.zip to another is likely to crash a virtual machine.

Create the java/lang/Object.class file in the usual way. It can be compiled with javac or assembled with jasmin.

The Object.class file is just another resource. It is a binary resource. It contains a collection of octets. The octets are treated like payload from the viewpoint of a machine code linker. Here is the corresponding C++ source code.

#include <rc_BasicBytecode.h>
#include <rc_java_lang.h>

static const char cd_Object[] = {
	 0xca, 0xfe, 0xba, 0xbe, ...
};

const rc_Bytecode *rc_java_lang::Object =
	 rc_CreateBytecode( cd_Object, sizeof( cd_Object ) );

Bytecode

A bytecode class is needed to decode a binary class file resource. In C++, the rc_Bytecode class is a pure virtual class and is fully implemented in a header file only. BasicBytecode is an example of a basic vm-specific implementation.

// rc_Bytecode.h
typedef unsigned short u2;

class rc_Bytecode {
  public:
	 virtual const void *getPayload() const = 0;
	 virtual size_t getSize() const = 0;

	 virtual u2 getCodepoolCount() const = 0;
	 virtual ByteArray getCodepool( u2 v ) const = 0;

	 virtual Access getAccess() const = 0;
	 virtual CharBuffer getThisClassSpec() const = 0;
	 virtual CharBuffer getSuperClassSpec() const = 0;

	 virtual u2 getInterfaceCount() const = 0;
	 virtual CharBuffer getInterfaceSpec( u2 ) const = 0;

	 virtual u2 getFieldCount() const = 0;
	 virtual Access getFieldAccess( u2 v ) const = 0;
	 virtual CharBuffer getFieldName( u2 v ) const = 0;
	 virtual CharBuffer getFieldDescriptor( u2 v ) const = 0;

	 virtual u2 getMethodCount() const = 0;
	 virtual CharBuffer getMethodName( u2 v ) const = 0;
	 virtual CharBuffer getMethodDescriptor( u2 v ) const = 0;
	 virtual ByteArray getMethodExceptionsAttribute( u2 v ) const = 0;
	 virtual ByteArray getMethodCodeAttribute( u2 v ) const = 0;

  protected:

  private:

};

Package

In C++, the rc_java_lang class is derived from the rc_PackageResource class.

class rc_PackageResource {
  public:
	 virtual CharBuffer getName() const;
	 virtual void setName( const char *v );

	 virtual rc_Bytecode *getBytecode(
		  u4 v ) const;

	 virtual rc_Bytecode *getBytecode(
		  const char *v, size_t s ) const;

	 virtual rc_Bytecode *getBytecode(
		  const char *v ) const;

	 virtual void addBytecode( const rc_Bytecode *v );

  protected:

  private:
	 CharBuffer name;
	 size_t size;
	 const rc_Bytecode **list;

};

Header

In C++, the header file for the rc_java_lang class becomes something like this:

#include <rc_Bytecode.h>
#include <rc_PackageResource.h>

class rc_java_lang : public rc_PackageResource {
  public:
	 static const rc_Bytecode *ArrayIndexOutOfBoundsException;
	 static const rc_Bytecode *Class;
	 static const rc_Bytecode *ClassLoader;
	 static const rc_Bytecode *Exception;
	 static const rc_Bytecode *Object;
	 static const rc_Bytecode *Runtime;
	 static const rc_Bytecode *System;
	 static const rc_Bytecode *Thread;
	 static const rc_Bytecode *ThreadGroup;
	 static const rc_Bytecode *Throwable;
	 :

	 rc_java_lang();

  protected:

  private:

};

Constructor

In C++, the constructor for the rc_java_lang class becomes something like this:

#include <rc_java_lang.h>

rc_java_lang::rc_java_lang() {
  setName( "java.lang" );
  addBytecode( rc_ArrayIndexOutOfBoundsException );
  addBytecode( rc_Class );
  addBytecode( rc_ClassLoader );
  addBytecode( rc_Exception );
  addBytecode( rc_Object );
  addBytecode( rc_Runtime );
  addBytecode( rc_System );
  addBytecode( rc_Thread );
  addBytecode( rc_ThreadGroup );
  addBytecode( rc_Throwable );
  :
}

Code generator

Most of the Java-platform-specific classes are generated by the bytecode resource application. When provided with a directory and package name, it writes C++ code. For each package, it writes a header file and source file.

./include/rc_<EM>package-name</EM>.h
./rc/<EM>package-name</EM>/<EM>package-name</EM>.cpp

For each class it writes out a C++ source file.

./rc/<EM>package-name</EM>/<EM>class-name</EM>.cpp

Each package is linked into its own static library. The static library can be linked to a virtual machine, such as decaf.

The following classes are not generated automatically:

  • CharBuffer
  • rc_Bytecode
  • rc_PackageResource
  • rc_BytecodeResource
  • BasicBytecode
  • BytecodeUtility

My implementation of rc_CreateBytecode() returns an instance of BasicBytecode. While BasicBytecode extends rc_Bytecode and is always const, BytecodeUtility is an implementation-specific class and is always mutable. The BytecodeUtility class actually does all the work. At this point, my implementation of BytecodeUtility is able to check the payload, check the magic number, check the version and return the number of entries in the constant pool. -- GilbertHerschberger (4 May 2000).
