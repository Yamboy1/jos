About this document

This document describes the purpose and how to use the BindGen program.

Audience

This document might be important for a Java programmer using the Bind Library. It might be interesting for a Java programmer using JDBC.

Author

Purpose

Introduction

This section explains the purpose of the BindGen program.

Scenario

A fairly complex JDBC-compatible database has already been designed and implemented. Now you are asked to get information in and out of that database. How are you going to do it?

Bind library

The bind library enables you to build Java classes that correspond to database queries, tables and views. It is possible to handcraft these classes.

Speed and accuracy

Automating the process shortens the time required to build a model for your tables and views. The model can be re-generated as the database design is altered over its lifetime.

Extend, not modify

Because it is possible to generate and re-generate these Java classes, two separate packages are recommended. In one, the generated package, all of the generated xxxRecord classes extend a class called CustomRecord. You must provide the CustomRecord class, which implements the Record interface. Typically, your CustomRecord class extends BaseRecord.

In other packages, you extend a generated class to provide additional and override existing functionality.

  • BaseRecord implements Record
  • CustomRecord extends BaseRecord or implements Record directly.
  • XXXRecord extends CustomRecord
  • YourCustomClass extends XXXRecord

Re-bind

The bind library provides the following classes for fields:
  • DateField
  • FloatField
  • IntField
  • StringField
Use a patch to apply changes to the generated source code. It is not yet possible to re-assign the type of a generated field.

In C++, the rebind() method enables a program to swap one field object for another. There is no corresponding const expression in Java. 

  public void rebind( Field source, Field target ) {
    target.setNext( source.getNext() );
    source.setNext( null );
    Field pcurr = searchForParent( source );
    pcurr.setNext( target );
  }

Using with Apache Ant

Introduction

This section explains the recommended use of the BindGen program using Apache Ant as a guide.

The taskdef target

In your Apache Ant script, you should create a target called taskdef. This is where you should put your task definitions.

Define the smartapi task, something like this: 

  <target name="taskdef">
    <taskdef name="smartapi" classname="gchii.tools.ant1a.SmartApiTask"/>
  </target>

The bindgen target

In your Apache Ant script, you should create a target called bindgen. It depends on the taskdef target. You should invoke this target as often as your database design changes.

Invoke the generator program using the smartapi task, something like this: 

  <target name="bindgen" depends="taskdef">
    <smartapi spec="run:program:gchii.tools.bindgen1a.GeneratorProgram?datasource=josdemo&amp;dest=${dest}/>
  </target>

Data source

The GeneratorProgram expects a datasource parameter. This parameter is the name of a jdbc:datasource: compatible data source.

The database: scheme enables the a program, such as BindGen, connect to a JDBC-compatible database using an external data source configuration file.

Example

As an example, the josdemo database is installed on a JDBC-compatible database service. In turn, this service is installed on a machine called jdbc.cjos.sourceforge.net. The jdbc.datasource.ini file contains a corresponding property: 
josdemo=jdbc:universaldriver://jdbc.cjos.sourceforge.net/josdemo
When you use datasource=josdemo, the generator program builds the following URI: 
  database:datasource:josdemo
This URI returns an instance of org.jos.database1a.Database, which enables this program to read the database metadata and generate the corresponding Java source code.
