Introduction

As a consequence of trying to find out what went wrong with JJOS 1i, I was able to understand something about making a virtual machine compatible with more than one class library.

By convention, the name of a built-in method in C/C++ resembles the corresponding native method in the Java programming language. To arrive at the name of a built-in method, replace dots (.) in the Java method name with an underscore (_).

The internBuiltInMethod() method of the decafJVM class enables decaf to bind a C/C++ function to a native method. Two or more native methods in the Java programming language may be bound to one C/C++ function.

How it works

If I write the internBuiltInMethods() method properly, I can enable decaf to bind to more than one version of Java source code. Java source code is compiled and linked into a jjos.zip archive.

The decaf virtual machine does not display an error message when a C/C++ function is bound to a class which does not exist. It shouldn't.

One build of a standard Java class library can be used in multiple virtual machines. Potentially, versions 1e, 1f, 1i and 2a of JJOS could reuse the jjos.zip archive from version 1e. Potentially, versions 1i and 2a could reuse the jjos.zip archive from version 1i. Bytecode can be maintained in a separate module and this indicates that it should be.
