DistributionGroup; DownloadJOS

Article contributed by GilbertHerschberger (2 August 1999).

Proposal

The whole point of building an operating system is so that

  1. many people can use it,
  2. some people can build upon it, and
  3. a few people can improve it.

We need to distribute products to our customers and other members of the JOS Project. There are three different kinds of distributions. Here is a (proposed) naming convention for distributions.

  • Blank is for...Runtime Edition

    This distribution is for many people. They want to download and use it. They don't care (that much) about the reasons behind the product. They just want to try it. The runtime distribution of any JOS project should be designated by using no suffix. This will let our customers (and beta testers) know that they can download an executable program. A runtime distribution includes executable code, and nothing but executable code. It has no source and no programming guide.

    With a JOS installer, runtime packages can be downloaded and "installed" on JOS. The base JOS product should be installed first. The base JOS product includes a JOS runtime executable. And then, any runtime package can be installed thereafter.

    Runtime distribution (and no suffix) should replace the use of binary on download pages.

  • DOC is for...Documentation Edition

    This distribution is for curious people. They want to have more documentation to understand how a product works. Both manual and automatic documentation should be written with HTML. This edition is where you might find HTML pages generated by JavaDoc and WebSource.

  • SRC is for...Source Edition

    This distribution is for some people. They are Java programmers. They want to download the source, to see how other programmers deal with techniques. Sometimes they want to customize the product to match their own environment. The source-code only distribution of any JOS product should be designated with an SRC suffix. This will let our customers (other Java programmers) know that they can download the source, use their own JDK to compile the source into an executable program. A source distribution includes source code, and nothing but source code.

    The SRC designation means lets everyone know that you're on your own to build a project make file and produce .class files or an archive. You have to deal with classpath issues.

    Source-only distribution (and SRC suffix) should replace the use of source on download pages.

  • TX is for...Technical Edition

    This distribution is for the few. These are JOS system programmers. They want to know how JOS works. They want to take JOS to the next level. The technical distribution of any JOS product should be designated with a TX suffix. This will give other programmers the technical documentation they need in order to use, extend and modify a product. This distribution should be self-contained. It includes the RX and SX versions.

    All of the technical documentation must be added to the distribution. This distribution should assume the customer has limited access to the JOS website at http://www.jos.org/ .

JOS Installer

We need a JOS-specific installer. It shouldn't be all that difficult to put an installer in the base distribution of JOS. After base JOS is installed, all other packages are installed using a JOS program.

When distribution packages are mapped to Java packages, the dependencies among packages should be automatically detected by the JOS installer. As I install the PlainTextBrowser, for example, the JOS installer should tell me that I must also install SmartAPI and SmartAPIStreamFactory.

You should be able to un-install any package, too. It should be as simple as using the delete command. The full featured un-installer should warn that PlainTextBrowser won't work if you un-install the SmartAPI or SmartAPIStreamFactory.

If we do it right, installing bytecode on JOS won't be as complicated as installing executables on traditional operating systems. What is it? We install a package file. When the base JOS distribution has been installed, there must be a default package directory. If the installer runs inside of JOS, it can determine the location of all package directories. (The list of package directories is stored in a registry.)

You can create a new package directory at any time. You can install any package in any package directory. Package directories are searched in the order in which they appear. You can install the same package in every package directory, only the first instance will be used.

This package search is just what you need if you're experimenting with new packages and don't want to lose your old one.

Dependencies

A installer-compatible package must provide a list of its dependencies. Of course, the list of dependencies will be stored in a registry-compatible file format. This list gives the JOS installer enough information to determine if dependencies have already been installed (by scanning the package path).
