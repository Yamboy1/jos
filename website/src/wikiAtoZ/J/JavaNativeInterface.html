Introduction

The Java Native Interface is an important part of the Java Virtual Machine. It provides both a mechanism to create a virtual machine and write native methods.

For more information, see Sun Microsystems' website for Java 1 Platform JNI for JDK 1.1 and Java 2 Platform JNI for JDK 1.2.

JOS and JNI

Both Sun Microsystems and the JOS Project are going to provide a JVM. These JVMs are similar in scope and function. Sun Microsystems' JVM is written as a shared library. The JOS JVM should be written as a shared library.

If Sun had done a better job, we could have built JOS directly from Sun's JVM. The Java Native Interface provides methods for managing multiple, concurrent JVMs within a single native process. Sun Microsystems JVM is written as a shared library. Unfortunately, it's flawed so that each native process can create one and only one JVM.

Does anyone know of a JVM for Microsoft Windows 95/98/NT that has been written properly, providing multiple, concurrent JVMs within a single native process?
Does anyone know if the Blackdown JVM for Linux has been written properly, providing multiple, concurrent JVMs within a single native process?

The multiple process JVM is necessary when each native process can create one and only one JVM. When a JVM is written correctly, according to the JNI, a multiple process JVM is unnecessary and redundent. The JOS JVM should be written as a shared library and should be written to create multiple, concurrent JVMs within a single native process.

At the lowest level, a kernel is a single native process. From the microprocessor's viewpoint, the kernel is the only process running. If the kernel can load a JVM as a shared library, and it can create multiple, concurrent JVMs from that shared library, there is no need to look for a solutions to running multiple processes within a JVM because you can create as many JVMs as you wish.

In order for JOS to be an implementation of Java, the JOS JVM must support the Java Native Interface (or JNI). The JNI might not have been a feasible option when the JOS Project started; but, it is worth revisiting now.

JOS runs on Windows

Java is a platform-independent platform. When we create a Java-based operating system, we create a platform-independent operating system. So, JOS runs on Windows.

We can use the Microsoft Windows 95/98/NT/2000 kernel to run all of our platform-independent code. The thin layer between our platform-independent operating system must be a native Win32 program. This is a hybrid program, part native and part Java.

  • Alternative 1

    We could reuse nothing. We could write everything from scratch. We could clone even open source and free license software. We could reinvent parts of Java and JVM that are already available from Sun, IBM, Borland, and Microsoft. We could have JOS finished in a few hundred years.

  • Alternative 2

    We could reuse everything. We could use the JDK and JRE from Sun, IBM, Borland, blackdown, Apple, and others. We could use whatever works, even if someone else wrote it. That doesn't cover everything we want. For some things, we could see what's out there and help them improve it. For a few, we could modify it to do what we want.

Hybrid applications

Hybrid applications are built with a little platform-dependent code and lots of platform-independent code. Every version of Java requires a little platform-dependent code, even JOS.

The Java Native Interface provides a powerful way to use an embedded JVM in a native program. This is how the java tool works. This is how the JOS kernel should work.

Platforms

Are you confused too? Sun Microsystems has introduced at least three Java platforms. They recently introduced "Java 2 Platform" as a new term. The Java 2 Platform is not equal to Sun's implementation of a JDK.

  • Java 0 Platform

    The reference implementation of the Java 0 Platform is Sun Microsystems' JDK 1.0.x. This is the version of Java that's consistently available in Java-enabled browsers. This version was shipped with Netscape Navigator 3.x.

  • Java 1 Platform

    The reference implementation of the Java 1 Platform is Sun Microsystems' JDK 1.1.x. This is the version of Java that's available on 22 platforms and few browsers.

  • Java 2 Platform

    The reference implementation of the Java 2 Platform is Sun Microsystems' JDK 1.2.x. This is the version strongly recommended by Sun Microsystems. Unfortunately, this version is only available today on a handful of platforms and no browsers.

Confusion

When Sun Microsystem's introduced the new platform, they left a lot of room for confusion in the Java community and in the press. The local bookstore has discounted JDK 1.2 books as if they are obsolete already, thinking that JDK 2 is the latest version when it's not. Books about JDK 1.2 should continue to put JDK 1.2 on the title page.

When you refer to the Java-compatible platform in general you should say Java 2 Platform. When you mean a JDK from Sun Microsystems, you should say JDK 1.2.
