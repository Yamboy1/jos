Article contributed by GilbertHerschberger (27 June 2000).

Introduction

A desktop written in Java could run on top of X alone, without a desktop written in machine code. A desktop enables you to launch other programs. When Linux is optimized for Java, there are both Linux and Java programs. Therefore, a Jannux(tm) desktop would enable you to run both Linux and Java programs.

Examples

A desktop is a classic Java application. For example, RyanHeise wrote SwingToolkit and distributes it as part of rheise.os.

I have been running Linux this way for more than two years. I wrote both Jose and UniversalBrowser. Personally, I find that the size of a desktop window is irrelevant. A desktop application written in Java can run on any foreign operating system in a window.

Dependency

A desktop depends on a process, either bytecode or machine code. When optimizing Linux for Java, a desktop application would run Linux programs and programs written in Java.

Model

When a GUI application written in Java runs within a desktop written in Java, there may be two virtual machines. The first virtual machine creates the frame that covers the virtual screen. The second virtual machine uses the first and its components.

When an application uses MPCL-compatible virtual machine, a GUI application written in Java also runs within a desktop written in Java; but, there is only one virtual machine.

Process API

One Java program can start another using the exec() method of java.lang.Process. An improved process API would enable one Java program to start another but with less complexity. This is one of the stated goals of the ERICProject.

  • The run:os: scheme is already available for running any native executable on Linux.

  • System properties may provide some clues to launch a new virtual machine without knowing exactly where it is.

  • Using the well-known interface to the java tool, it is possible to create a platform-independent API for launching a virtual machine from a virtual machine.

How does one virtual machine use the AWT from another? IBM has already been able to do this with Remote Abstract Windowing Toolkit (RAWT).

Command line approach

The process API could take a command line approach. Since the java tool is often invoked from the command line, a command line approach would make the process API familiar to a wide audience of existing programmers.

[java] -classpath <EM>cp</EM> <EM>cn</EM> [ <EM>arg1</EM> [ ... ] ]

The process API would provide properties to help determine where the original virtual machine has been installed on a particular Linux box. It could be as simple as storing a process API configuration file in a well-known Linux directory.

public class ExternalApplication {
  public ExternalApplication() {
  }
  public ExternalApplication( String v ) {
  }
  public String getCommand() {
	 return command;
  }
  public void setCommand( String v ) {
	 command = v;
  }
  public int exec() {
	 if ( command == null ) {
		return 255;
	 }

	 if ( !command.startsWith( "java " ) ) {
		return 255;
	 }

	 StringBuffer buffer = new StringBuffer();
	 buffer.append( "run:os:" );
	 buffer.append( separate( getExternalDirectory() ) );
	 buffer.append( command );

	 URI uri = new URI( buffer.toString() );
	 Program p = (Program) uri.getObject();
	 if ( p == null ) {
		return 255;
	 }
	 p.run();
	 return p.getProgramContext().getResultCode();
  }
  private String command;
}

Desktop browser

The desktop browser is part of UniversalBrowser. It has been launching programs for a foreign operating system for more than six months. From within a Java Virtual Machine, the desktop browser provides a point and click launch of any program, including the java tool.

When it is configured with the eric tool, you can use your favorite HTML browser can serve as your desktop. The eric tool is part of the ERICProject and supports shortcuts.
