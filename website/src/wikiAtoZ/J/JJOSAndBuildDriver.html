Introduction

A mechanism is needed to inform your Java program about the current build. To change the behavior of your Java program based on the current build, use the BuildTool. Sometimes your program needs to behave differently if it is running in host mode (on Linux or Microsoft Windows) than it does running in i386 mode on jJOS. This mechanism makes that possible.

The BuildTool class (decaf/build2a/BuildTool) uses the BuildDriver interface. It returns the appropriate build name. This class should be used; do not use a BasicBuildDriver class directly.

Build driver

The BuildDriver interface (jos/platform/BuildDriver) defines the following constants. 
  public static final int HOST_BUILD = 1;
  public static final int I386_BUILD = 2;
It defines the following method. This method should return the build ID, either HOST_BUILD or I386_BUILD. 
  public abstract int getBuildID();

Pure bytecode

This mechanism is implemented in pure bytecode; no native (or built-in) method is needed.

Unfortunately, it requires a different jjos.zip archive for each build.

Basic build driver

The BasicBuildDriver interface (jos/platform/driver/BasicBuildDriver) implements the BuildDriver interface. In the arch/host/bytecode directory tree, the driver implementation returns HOST_BUILD. In the arch/i386/bytecode directory tree, the driver implementation returns I386_BUILD.

Built-in method

When fast native calls are used, the native getBuildID() method of the BuildTool class might be used--instead of the BasicBuildDriver.

When complete, there will be no need for two different jjos.zip archives. All bytecode will be moved to the bytecode directory from common/bytecode.
