Introduction

The decaf passes a variety of parameters to its built-in methods. Built-in methods, or native methods, are written in machine code. The informal interface between decaf and its built-in methods might be called the decaf native interface.

Parameter list

Instead of passing separate parameters to a built-in method, a single object should be "passed". The object is the parameter list.

Here is a prototype of a typical built-in method today: 

jjBoolean _custom-method_ (
    java_thread *t,
    frame *f,
    resolved_method,
    const char *args[],
    int n_args
    );
Many built-in methods do not use all of these parameters. When a parameter is un-used, the g++ compiler displays a warning message. In this case, these messages are spurious, unnecessary.

Here is a prototype a typical built-in method using a parameter list object: 

jjBoolean _custom-method_ ( ParameterList *list );
The g++ compiler will not display warning messages if the built-in method does not use properties of the parameter list.

Within a built-in method, the relevant parameters can be retrieved from the parameter list object, like this: 

jjBoolean _custom-method_ ( ParameterList *list ) {
  list->getThread(); // replaces t
  list->getMethod(); // replaces rm
  list->getFrame(); // replaces f
  list->getCount(); // replaces n_args
  list->getParameter( n ); // replaces args[ n ]
}

Invoking a built-in method

''Note:'' It is possible for the calling method to efficiently re-use a parameter list object.

When every opcode method should call a common sub-routine when it invokes a built-in method. Duplicate code should not be written within the invokestatic(), invokevirtual(), invokespecial() and invokeinterface() opcode methods.

A built-in method might be invoked something like this: 

  ParameterList list;

  list.setThread( thread );
  list.setMethod( rm );
  list.setFrame( this );

  list.setCount( n_args );
  list.setParameter( 0, ... );
  list.setParameter( 1, ... );
  list.setParameter( 2, ... );
  list.setParameter( 3, ... );
  ''custom-method''( &list );
The setCount() method manages the storage for the argument array. If the args property has never been used before, a new args property is allocated by the ParameterList class. Otherwise, if the new count is greater than the old count, the old args property is free'd and a new args property is allocated. Otherwise, if the new count is less than or equal to the old count, the old args property can be re-used.

Future

The role of the parameter list can be expanded in the future without modifying the interface of existing built-in methods. For example, a push_jint() method could be implemented. This method might enable a built-in method to push a return value on the local variable stack. 
  list->push_jint( n ); // replaces getFrame()->stack->push_jint();
