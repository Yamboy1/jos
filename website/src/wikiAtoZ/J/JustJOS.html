Article contributed by GilbertHerschberger.

Introduction

Yes, I have been seriously considering writing applications to run only under the JOS. It may seem like counter to the rising tide of application programmers wanting to run in pure Java.

Unfortunately, there's a problem. The definition of pure Java changes from moment to moment. Sun Microsystems or any other giant software company doesn't have what I want. And I wonder if they will ever build what I want.

And what's wrong with building a better JVM? What could be wrong with providing things on JOS that are not available elsewhere?

Exclusive?

Sometimes I think that it isn't fair to provide something on JOS that is not available elsewhere. It troubles me the way some software is only available on insert-name-here to the exclusion of everyone that bought another operating system. Is the JOS organization going to be exclusive club? Are we going to require people run JOS and nothing but JOS?

I believe that JOS will become operating system number five. I already have four. JOS should be just another operating system that runs my computer. I hope to switch to JOS just as easily as I switch today from one operating system to another.

Why not?

I believe that I won't be able to write the kind of software I want to write (and my customer's want to use) if I follow Sun Microsystems wherever they go.

If they invent another bloated version of Java, and they will, I have to rewrite all that software whether I want to or not. After writing 125,000 lines of Java code, I want control over what I work on. I want to work on things on my time frame. I want to work on things at my own pace. I want to use the parts of Java that make sense to me. I want to learn about the parts of Java that I need to use today. Forget about the rest. I don't care.

Just JOS

If I build a class that requires something inside JOS to run, I want to let people know that it is exclusively JOS. It is made for JOS and nothing but JOS. I wonder if I should call it JustJOS.

For example, I want to build classic batch/console using an event-driven keyboard. These classes might be available only when JOS packages are a part of classpath.

How do I explain to my customers, for example, that one of my applications depends on JOS packages--but not the kernel--in order to run?

When written in Java, most of these classes will run flawlessly in any JVM.

Sun Microsystems marketing decided to call everything Java. It causes problems to use the same name for the processor AND language AND application program interface AND environment. They could have given all that better names.

Today, there are JVM-compatible compilers for COBOL and Pascal languages, too. They create class files, just like javac, which run inside a JVM. After you experience this, you can easily see that the JVM and even the JRE are misnamed.

Technically, almost all of the Java API is optional. While only a few classes in the API are special, most are not. The JVM doesn't care if they are there. If you write a program that does not use a long list of packages, and never will, why do you have to ship them? The software license says so.

  • I really don't want a new JVM. I'm happy with two from Sun and two from Borland.
  • I really, really don't want a new Java API. I personally couldn't rewrite something I already have when there is something more important that's missing: an OS.
  • I want to make some sense of things in Java that don't make sense, like
    • why the standard input stream assumes one line at a time
    • why exceptions allow you to get a message, but not set it
    • why an application has no interface, compared to applet and servlet
    • why an application always assumes command-line parameters even when there are none
    • why there are depretiated methods
    • why you can't write once

Architecture?

Is JOS an OS with a JVM? Can I also run a JVM from blackdown?
JVM JVM ... JVM
JOS OS/kernel

Or, is JOS a JVM that runs on an OS, along side other applications? This is what I already have with a JavaSoft JRE running on Microsoft Windows 95.
JOS JVM JOS JVM ... foreign app
foreign OS/kernel

Or, is JOS an OS and JVM?
pseudo-process psuedo-process ... pseudo-process
JOS kernel/JVM

Or, is JOS using Domains where others use Process?
app-1 app-2 app-3
Domain 1
app-4 app-5
Domain 2
app-6 app-7 app-8
Domain 3
JOS kernel

I want an OS designed from the ground up to be optimized to run Java code. Without change, this Java code runs on other JVM, too.

All utilities that can be written in Java should be written in Java. There should be so few lines of code written in C/C++ or whatever. The whole idea of reaching for this is to minimize the footprint of the non-Java code. Beyond the non-Java code, local storage for most of the Java class files must be optional. Provide a cache of most-recently used. Make class files available across the network.

I don't want a network computer. I don't want to boot my computer from a foreign network service. I want to boot my computer the way I always boot my computer: from a local partition on my hard drive or floppy.

Multiple what?

I can't figure out why everybody is talking about it. What's the point of having pseudo-processes running inside one JVM? What problem can only be solved this way?

It has unbearable technical problems.

  • What about complexity? It is probably the most complex way to proceed. There must be an easier way.
  • What about the static keyword? Semantics in the static keyword that cannot be resolved with multiple processes inside the JVM. There are well-known static methods, for things like System.exit() to stop everything in a process.
  • What about performance? Throwing all variables into one big giant list slows all of the applications down. Like existing JVMs, each process has its own (short) list of variables.
  • How is it going to be crash-proof? When pseudo-process goes bad, they all go bad. Process should mean independence.
  • How do you debug your programs? With existing JVMs, there is a way to pull the plug on the JVM even when the Java code gets into deep trouble.

This is different than intentionally running Java "applications" from a shell written in Java.

Why is this everyone's assumption? When looking at this pragmatically, there's semantics in the static keyword that cannot be resolved with multiple processes inside the JVM.

I really don't want multiple processes inside my JVM. I want multiple, independent JVMs. Why not? If JOS could launch a new instance of a JVM everytime a new process was loaded, it would have (1) it's own garbage collector thread, (2) it's own UI thread for handing modal dialog boxes, and (3) static would retain the semantics of one-per-JVM.

The kernel would then have JVMs and only JVMs to worry about. The kernel load one and only one native executable program: java.dll and reuses it for each instance of a JVM. The kernel could put a JVM into the foreground, give different JVMs different priority. JVMs would ask the kernel for more RAM, for virtual devices.

A JVM can launch another JVM. In fact, I've been doing that all along. (Yes, the JVM from Sun can launch another JVM.)

A JVM can launch another JVM and exit, effectively replacing itself. This is cleaner than rebooting the whole kernel.

I thought JOS would have a system program, written in Java, running in the master JVM. Of course the master JVM has different priveledges than all of the others, since there can be only one master.

The master JVM would have a separate thread for THE OS file subsystem. All other JVMs would request file access from the master JVM. From within the master JVM, it can install all of the real-time device drivers.
