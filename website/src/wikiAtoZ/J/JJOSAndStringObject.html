Article contributed by GilbertHerschberger (5 February 2002).

Introduction

What exactly is a string? This article introduces a string as it is defined in JJOS and compares it to the Java Virtual Machine Specification ( JavaVirtualMachineSpecification ).

According to the specification, this is a definition of a string:

Every string in JJOS should be an instance of java/lang/String.

Here are some of the implications of this premise.

  1. The java/lang/String class itself defines an array of chars to hold the individual characters in a string.
  2. Since java/lang/String extends java/lang/Object, an string is always spoken of as if it were allocated on the heap along with all other objects.
  3. Note: When a bytecode process is destroyed, so are all of its strings.
  4. An array of chars also extends java/lang/Object. Therefore it is always spoken of as if it too were allocated on the heap.
The implementation of a string in JJOS does not closely follow the specification.
  • An instance of java/lang/String on the object heap is never initialized. Its fields are ignored. When printing a string, for example, only internal C++ fields are used.
  • Most bytecode methods of java/lang/String do not work because its reference to an array of chars is always null.

Plan

Design a mechanism to upgrade the JJOS string to a true instance of java/lang/String.

Here are some of the prerequisites for this:

  • A method must be written in C++ to print an array of chars on the console. Any array of chars should be print-able. Since a console may not support 16-bit unicode characters, it may be necessary to convert them to 8-bit characters.
  • The constructor for an array of chars should be able to convert from a ASCIIZ string. An array of chars is no longer terminated with a NUL character. This constructor should be able to decode the UTF8 format so that 16-bit unicode characters might be encoded with 8-bit characters.
  • Every array of chars should be allocated on the heap for a bytecode process. When the bytecode process is destroyed, all of its arrays are destroyed.
  • A constructor for a java_string should accept an array of chars. The payload field of the java/lang/String should be populated so that all of the bytecode methods of a string work as expected.
  • A convenient method should be written in C++ to print an instance of java/lang/String. In fact, this method simply passes the payload field to the method that prints an array of chars.

More

Every 16-bit char in an array of chars should be stored on the object heap. Arrays are destoyed along with a bytecode process.
  • The native format of strings throughout the JJOS virtual machine should be array of chars, not constant_utf8.
  • In an array, 16-bit chars should be stored as 16-bits. They should not be stored in 32-bit, int-sized containers; otherwise, a virtual machine would run out of memory more quickly. They should not be stored in 8-bit, byte-sized containers; otherwise, the bytecode methods of java/lang/String do not work.
A program written in C++ should be able to create an array of chars something like this: 
  void program::example( BytecodeProcess *p ) {
    const char *text = "Hello, World!";
    jobject *o = p->CreateCharArray( text );
  }

A program written in C++ should be able to create an instance of java/lang/String something like this: 

  void program::example( BytecodeProcess *p ) {
    const char *text = "Hello, World!";
    jobject *o = p->CreateString( text );
  }
