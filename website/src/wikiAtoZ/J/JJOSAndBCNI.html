Article contributed by GilbertHerschberger (8 March 2001).

Introduction

The decaf.host and decaf.i386 packages contain platform-specific classes. The HostFactory class plugs into the BytecodeNativeInterface (BCNI) when the host build is running. The i386Factory class plugs into BCNI when the i386 build is running. Other builds are possible, including a win32 build.

BCNI enables us to write the platform-specific classes for many virtual machines into a single archive, using the power of namespace. Platform-independent classes can be written into a common package for all virtual machines to share.

JJOS has been selected as the baseline for additional work on BCNI. This should illustrate how to integrate a standard Java class library with a custom virtual machine.

First bug

This is the first bug to appear in the integration of decaf and BCNI. Apparently, an early version of decaf did not fully support the java/lang/String class. This bug was exterminated by finishing the string class in decaf.

For some reason, the HostFactory class is returning the wrong object when a request is made. It always returns the first object in the factory, without regard for the content of a request. It does not invoke the abort() function.

When additional tests were added to jos/platform/TestSuite, a number of conditions invoked the abort() function.

Example

The following example illustrates part of the if statement test. This is written in the Java programming language. 
  Object object = new String( "" );
  if ( object == object ) {
	 println( "OK" );
  }
  else {
	 println( "FAILURE" );
  }
In fact, an object is always equal to itself. This should be a very simple comparison. Which opcode is being used? Which opcode method is failing? To exercise each of the conditional opcodes, a test program must be written in Java assembler, using Jasmin.

Second bug

The invokestatic, invokespecial, invokevirtual and invokeinterface opcode methods were optionally compiled with the FAST_NATIVE METHOD flag. This mistakenly reverses the order of the ACC NATIVE and lookupBuiltInMethod() steps. While this might have been necessary for an early version of decaf, it is no longer useful.

The lookupBuiltInMethod() method should be invoked only when a bytecode method is ACC_NATIVE. It should always display a message when a built-in method is missing.

Third bug

A BCNI factory should not display a message when it does not support a given device. The getObject() factory returns null when a device is missing. The NullPointerException is thrown if the calling method does not check the return value from getObject().

The jos.system.KeyboardInterface interface should be supported by i386Factory. It should return an instance of the i386Keyboard class. 

    // *D! if ( v.equals( "jos.platform.KeyboardInterface" ) ) {
    if ( v.equals( "jos.system.KeyboardInterface" ) ) {
       if ( keyboardInterface == null ) {
         keyboardInterface = new i386Keyboard();
       }
       return keyboardInterface;
     }
