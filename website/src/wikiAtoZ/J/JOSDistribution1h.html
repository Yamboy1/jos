18 December 2000

The latest version of a utility should be available to a desktop through inheritence. A desktop file itself need not change. The latest version of BulkMake(tm), for example, is always available as gchii.tools.bulkmake.BulkMake. The actual application is found in gchii.tools.bulkmake2c. Here is a snippet of the relevant source code:

// gchii/tools/bulkmake/BulkMake.java
public class BulkMake
	 extends gchii.tools.bulkmake2c.BulkMake
}

When the next version of this utility is available, only the dispatching class is modified and recompiled. For more information, see also InheritCurrentImpl . -- GilbertHerschberger

18 October 2000

I am designing a Java servlet to help with package-oriented documentation. It enables a programmer to provide helpful information about each package. The functions are add a package, edit a package, view a package and list all packages. As a bonus, it will incorporate an BaseHttpProgram class, an extension of BaseConsoleProgram.

The Erica utility is working. It automatically downloads package files which have changed. It works across a network using FTP, HTTP or SMB. The Deric utility has been reworked to create package files interactively and automatically generate a Deric Index.

The BulkMake(tm) utility has been reworked to include a package-oriented cabinet tool. The cabinet tool creates unsigned cabinet (.cab) files.

Many utilities have been moved to Smart API Utilities from Smart API 1a, Smart API 1b and Smart API 2a. Some utilities have proven more difficult to move than others. For example, many utilities use the ColumnLayout from the Universal Browser. (Those utilities would not compile after removing the Universal Browser utility.) For another example, the ZipFinder package uses classes from the Find utility.

The ColumnLayout class has been moved to the org.jos.gchii.layout1a package.

The file filter and file processor classes from Package Archive 2a have been moved to the gchii.library.fileprocessor1a package. -- GilbertHerschberger
