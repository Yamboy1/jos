Article contributed by GilbertHerschberger (7 March 2001, updated 19 May 2001).

Introduction

Unlike version 1f, JJOS 1i provides a more concise directory tree. Instead of changing the directory tree all at once, the directory tree will be changed over the next few versions. Using evolution, this product can be built at each step along the way.

These changes may seem arbitrary at first; but, they're not. These changes make room for the future, such as a new build or new source platform. With the help of the BytecodeNativeInterface (BCNI), it is no longer necessary to define two or more classes with the same name.

About BCNI

With the integration of BCNI with JJOS, a custom jjos.zip file is no longer required for each build. This reduces the amount of effort required by a virtual machine designer to reuse the classes from the JOS platform. This is a big plus for the Java programmer who wants to build a custom application of JJOS.

One archive file can contain the code for all builds. This gives us a more easily understood Java namespace. The common/bytecode directory replaces the common/bytecode directory. The bytecode directory can be removed from arch/host. The bytecode directory from arch/i386 only contains jjos.zip, a symbolic link to ../../../bytecode/jjos.zip. This enables many builds to share jjos.zip, a single class library.

About CVS modules

The JJOS product requires both C++ and Java code. These programming languages use very different tools and might require different programmers. In JJOS 1f and 1i, C++ and Java source code are combined in a single CVS module. Unlike versions 1f and 1i, the next version will have two CVS modules, one for C++ and one for Java.

  • jjos1f - This module contains the C/C++ and Java source code for JJOS 1f. This version does not depend on BCNI.
  • jjos1i - This module contains the C/C++ and Java source code for JJOS 1i. This version depends on BCNI 1a, included in the jjos.zip file produced by the jjos1i-bc module.
  • jjos2a - This module contains the C/C++ and Java source code for JJOS 1f. This version does not depend on BCNI. With the host and i386 directories, a nativecode directory is used exclusively for compiling for Linux. Only in the host directory, a win32 directory is used exclusively for compiling for Microsoft Windows. The common/build directory is useful for building the Java source code.

Host build

When starting the jjos program on Linux, use one of the following commands: 
$ jjos jos.host.hostinit
or 
$ jjos jos.host.HostTestSuite

i386 build

When starting the jjos program with Etherboot or GRUB, use one of the following main classes: 
jos.i386.i386init
or 
jos.i386.i386TestSuite

Header files

The Java programming language illustrates how important it can be to keep interface and implementation apart. Likewise, public header files should be stored apart from C/C++ source code. During the creation of version 1i, public header files will be moved to the include directory. This opens up the potential of writing alternative implementations of private classes. It also opens up the development of JJOS to Microsoft Windows and other non-Linux platforms.
  • include/jjos/stdlib.h - For TARGET_UNIX, this header includes stdlib.h provided with your compiler/platform. Otherwise, this header defines all supported functions from the standard C library.
  • include/jjos/stdio.h - For TARGET_UNIX, this header includes stdio.h provided with your compiler/platform. Otherwise, this header defines all supported I/O functions from the standard C library.
