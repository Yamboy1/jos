Introduction

Boot-specific implementation could be moved to a "boot" framework. With this framework, it enables much of the start-up sequence to be common machine code. It might be possible to run other machine code programs besides a decaf virtual machine.

Model

The boot sequence can be divided into three (arbitrary) parts: init, draw and exit. The init part captures the command line from the boot method, such as host, GRUB or Etherboot. The draw part invokes a statically linked main program. The exit part is reserved for when the abort() or exit() function is invoked. 
class mc_System {
  public:
    /**
     * Boot-specific implementation.
     */
    virtual bool init() = 0;

    /**
     * Common routines.
     */
    virtual bool draw() = 0;

    /**
     * Exit event.
     */
    virtual void exit() = 0;

  protected:

  private:

};

Platform-independent class

A base class can contain platform-independent machine code. 
class BaseSystem : public mc_System {
  public:
    BaseSystem();
    virtual ~BaseSystem();
    virtual bool init();
    virtual bool draw();
    virtual void exit();

  protected:

  private:

};

Etherboot

An Etherboot-specific implementation extends the platform-independent class. 
class EtherbootSystem : public BaseSystem {
  public:
    EtherbootSystem();
    virtual EtherbootSystem();

    virtual bool init();
    virtual bool draw();
    virtual void exit();

  protected:

  private:

};

Startup

Only one system can be defined at a time. A system is statically linked to a boot image. It could be done like this: 
static mc_System theSystem;

mc_System *mc_getSystem() {
  if ( theSystem == NULL ) {
    #ifdef GRUB
      theSystem = new GRUBSystem();
    #else
      #ifdef ETHERBOOT
        theSystem = new EtherbootSystem();
      #else
        theSystem = new HostSystem();
      #endif
    #endif

    theSystem.init();
  }

  return theSystem;
}
