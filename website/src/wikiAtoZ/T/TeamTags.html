Team Tags

Members of a programming team must show respect for the source code and other programmers. In the spirit of cooperation and simple courtesy, we have been using team tags, or standard machine-readable characters placed in comments explaining what happened to the code. These tags are kinda, sorta like the abuse of comments that enables us to use JavaDoc.

When working on a program alone, deleting a line of code was a mistake because the "fix" didn't quite work. I don't want to delete the original code; I just want to comment it out. I'm often interrupted in the middle of writing a method (phone rings, people walk into my office). I could lose my place and forget to finish a method.

When working with a team of 60+ programmers during the winter of 1992-93, we were working on custom versions of a single, large, monolithic program. Team tags were extended a little to help coordinate common bug fixes and integration of custom code back into the core.

When all is said and done, these tags supplement the version control process. You don't have to go back to the version control system as frequently.

Syntax

In general the tags have the following syntax, so that they are human-readable and machine-readable. The name, date and description are optional.

  • tag name date description
where tag is one of the well known team tags;
where name is your name (or initials);
where date is modification date (usually today but not always);
where description is comments about the tag.

Further work in Java/C++ has refined the team tags into the following chart:
*A! A is for...Attention! This tag gets your attention. It should be used when you are interrupted in the middle of writing a method and you're not quite finished. Sure, the method might compile, but you need to know that it must not be shipped without another look. Remove the tag when you take a second look. In quality control, the application is ready to ship when there are no more attention tags. In the following example, imagine the code should be able to use the new version of the save method, but the new version hasn't been written yet. 
  int iMax = getBufferLength();
  for ( int i = 0; i < iMax; i++ ) {
	 if ( buf[ i ].isResponse() ) {
		buf[ i ].save();
	 }
	 // *A! buf[ i ].save( Document.RESPONSE );
  }
*D! D is for...deleted. Intentionally commenting out a line of code or a block of code prevents it from being compiled, but it is not a real comment. When programming for a team, never delete the code because everyone needs to know both the before and after version. The commented out code can be removed eventually when a "fix" has been tested and it works. Here is a trivial example of a single line: 
  // *D! int i = buf.length;
  int i = getBufferLength();
Here is a trivial example of a multiple line: 
  /* *D!
	 for ( int i = 0; i < buf.length; i++ ) {
		if ( buf[ i ].isResponse() ) {
		  buf[ i ].save();
		}
	 }
  */

  int iMax = getBufferLength();
  for ( int i = 0; i < iMax; i++ ) {
	 buf[ i ].save( Document.RESPONSE );
  }
*L! L is for...logic. There are times when you must use less-than-obvious expressions. When you know you're using convoluted logic for the sake of the compiler and language, you should prefix the convoluted version with an equivalent, long form. When debugging, sometimes it is easier to step through the long form. Here is a trivial example: 
  /* *L!
	 if ( source.isResponse() ) {
		return true;
	 }
	 else {
		return false;
	 }
  */
  return source.isResponse();
*T! T is for...Test. Don't delete your debugging code out of the source file. Use this tag to let yourself and others know that this code was used to debug a problem. 
  if ( buf == null ) {
	 // *T! System.out.println( "buf == null" );
	 return;
  }
*Z! Z is for...uh, well, documentation. Use this tag to let yourself and others know that more documentation is needed. When you're working on experimental code, it is inappropriate to use this tag (and inappropriate to create JavaDoc comments). It is only when it becomes stable and can be depended on by yourself and others that it should be committed to documentation. This tag says that this class or method is ready for documentation.
