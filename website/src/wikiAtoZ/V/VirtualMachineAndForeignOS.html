BytecodePages

Article contributed by GilbertHerschberger (14 August 2000).

Introduction

A virtual machine depends on a foreign operating system. What does a virtual machine depend on? By identifying these dependencies, we can discuss the similarity between UltimateJOS and TransitionalJOS.

System components

As a system, we can divide JOS into subsystems. It has a few major system components. Each major system component can be developed independently from the others. At runtime, these system components must be bound together. A Java-based operating system can be decomposed into five independently developed subsystems.

  1. boot loader
  2. kernel
  3. standard C/C++ library
  4. virtual machine
  5. standard Java class library

1. Boot loader

A boot loader runs after the BIOS. It creates an environment to load and run a kernel. A variety of boot loaders are already available, such as Etherboot and GRUB.

2. Kernel

A kernel is the first subsystem loaded and invoked. It must provide system calls for the standard C/C++ library. Here are a few system calls (in no particular order).

  allocate memory
  free memory
  create process
  destroy process
  invoke signal
  file open
  file read
  file write
  file close
  install interrupt
  install filesystem
  install device

These system calls must be invoked in the kernel address space, accessing and modifying internal kernel structures.

3. Standard C/C++ Library

A standard C/C++ library is required so that a virtual machine can be compiled and run within a kernel. Of the standard C library, we need to prioritize those functions already used by decaf and other virtual machines.

Here are some of the methods that could be implemented in a standard C library (in no particular order):

  memcpy();
  memset();
  alloc();
  free();

  strcat();
  strcpy();
  strlen();
  strstr();

  open();
  read();
  write();
  close();

  geteuid(); 
  seteuid();
  getuid();
  setuid();
  getgid();
  setgid();
  getpid();
  getppid();

  fork();
  exec();

  abort();
  kill();

  printf();

  setsignalhandler();

Plus, a standard C/C++ library would include a native interface, such as the JavaNativeInterface. It would include these functions (in no particular order):

  GetDefaultJVMEnvironment();
  CreateJavaVM();
  DefineClass();
  RegisterNative();
  GetStringUTF8();
  GetJavaVMs();
  GetObjectClass();
  InvokeNonVirtual<EM>type</EM>Method();
  InvokeNonVirtual<EM>type</EM>MethodA();
  InvokeNonVirtual<EM>type</EM>MethodV();
  InvokeStatic<EM>type</EM>Method();
  InvokeStatic<EM>type</EM>MethodA();
  InvokeStatic<EM>type</EM>MethodV();
  InvokeVirtual<EM>type</EM>Method();
  InvokeVirtual<EM>type</EM>MethodA();
  InvokeVirtual<EM>type</EM>MethodV();

4. Virtual Machine

A virtual machine loads after the kernel and uses the standard C/C++ library to invoke system calls. So far, the difference between Ultimate JOS and Transitional JOS is minimal. On Ultimate JOS, a virtual machine might be introduced before the system goes multi-user. On Transitional JOS, a virtual machine is introduced after the system goes multi-user. Thus, any virtual machine runs on both a native and foreign operating system. In other words, the JOS-specific kernel is just another foreign operating system.

When writing a virtual machine, a C/C++ programmer can use the standard C library. A virtual machine can be cross-compiled to run on both a native and foreign operating system.

5. Standard Java Class Library

When writing an application, a Java programmer can use the standard Java class library. A virtual machine needs to match a standard Java class library. A virtual machine developed for Kore won't work properly with Kaffe or classpath.

When writing the machine code part of a native method, a C/C++ programmer can use the standard C/C++ library.
