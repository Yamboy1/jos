ArchitectureGroup

Article contributed by GilbertHerschberger (9 February 2000).

Introduction

The ArchitectureGroup explores different architectures. Out of many possible architectures for JOS, this group should help guide our members to the architecture with the greatest benefits.

The architecture must not operate in a vacuum. It gathers more information. It provides feedback to other groups. It works closely with the KernelGroup to see how a virtual machine, application program and device driver fit together into a whole. It proposes changes to the kernel and virtual machine.

Aspects

Our operating system has four architectural aspects:
  • Porting
  • Native interface
  • Device driver
  • Application

Porting

We need a strategy to address the porting aspect of our operating system. This strategy must address how JOS can be ported to different processors and all kinds of peripherals. Parts of JOS are platform-independent, working across all processors and peripherals. Parts of JOS are platform-specific, working only on one processor or only on one peripheral. An architecture must include the strategy of writing most of the operating system in a platform-independent way.

While I am sure that most of the strategy has been discovered and discussed, it must be written down. We must be able to share our strategy with others.

Native interface

We need a strategy to address the native interface aspect of our operating system. This strategy must address where machine code ends and bytecode starts. Part of JOS is written in bytecode (as much as possible). Part of JOS is written in machine code. A bytecode-based operating system must provide an architectural overview of exactly how this is appropriate. What should be written in bytecode? What should not? How does a programmer decide?

What are we looking for in a native interface? If we document what we have and what we are looking for, it would bring clarity to the programming effort. The JavaNativeInterface, BytecodeNativeInterface and decaf native interface have been proposed.

Device driver

We need a strategy to address the device driver aspect of our operating system. This strategy must explain to a programmer and others how to build a device driver. Programmers need a device driver development kit (D3K). What services does the kernel provide to a driver? How does a driver plug into the kernel? What is the life cycle of a driver? Some device drivers might need to use native code. This strategy must address these issues as well.

Other programmers must build, test and distribute JOS-compatible device drivers. We must give these programmers an opportunity to write a device driver others can use.

Application

We need a strategy for the application aspect of our operating system. This strategy must explain to a programmer and others how to build an application program. Programmers need an application development kit. What services does JOS provide to an application? How does an application plug into the kernel? What is the life cycle of an application? Some applications might need to use native code. This strategy must explain how this is done or forbit it.

This might be the easiest strategy to document. We anticipate compatibility with Sun Microsystems JDK. Any application written for the JDK runs -- unchanged -- on JOS. Or, does it?

Nowhere does an architecture include the details of how bits are painted onto VGA hardware. It does not include the file subsystem, except by reference. It must be a high level document. It must be reviewed by members of this project, especially members of the architecture group.

The ArchitectureGroup has a lot of work ahead. If the work is done well, many volunteers can write the code that makes JOS work.

Criticism

The architecture group has been consistently criticized for failing to document the architecture of JOS. Much of this criticism is warranted. There is no concise document we can point to that explains all four aspects.

Members of the ArchitectureGroup should volunteer to briefly document each aspect. Members must work together so two or more members can document parts of the same aspect at the same time. We must work toward a comprehensive, exhaustive documentation of these important aspects.
