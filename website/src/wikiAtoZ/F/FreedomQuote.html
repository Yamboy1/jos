ArchitectureGroup

Article contributed by GilbertHerschberger (20 June 2000).

Bumper sticker

Driving home from work the other day, I read a bumper sticker. It made me stop and think. It said something like this:

None are free when anyone is oppressed.

Good freedom is different from bad freedom. Good freedom comes from rules that apply equally to everyone. Good freedom includes everyone, excludes none.

On the other hand, bad freedom comes from an elite group where rules apply to everyone else. Bad freedom is based upon exclusion and oppression.

Technical

I am a technician. I am an architect. I do not intend to speak socially about government and politics. Speaking only as a technician, I embrace each architecture that creates one set of rules for everybody. I reject each architecture with a foundation of two different sets of rules. What is good for the customer is good for the vendor.

One product built upon good architecture is more effective than another built upon bad architecture. Good architecture is the key to effective products.

A good architecture enables everyone to use it. There is a place for everything and everything has a place. Good architecture is based on inclusion and rules that apply to everything.

For example, it is unthinkable for Sun Microsystems to reserve the right to (re)create all application programming interfaces (APIs). Sun Microsystems wishes to oppress other programmers. Other programmers might understand the potential of Java more than they do. Other programmers might understand how to build a Java-based operating system better than they do.

It is unthinkable for Sun Microsystems to support only the latest and greatest versions of Java. They act irresponsibly, as if previous versions do not exist.

In the end, I must reject this product of Sun Microsystems. Java Technology only works if you work at Sun Microsystems.

Wish list

For the next technology, here is what I would like to use. I want a simple, straight-forward, object-oriented virtual machine (much like JDK 1.02). I would like virtual machine technology to keep separate, once and for all time, from the standard class libraries.

I would like to work with a product from a vendor that truely understands object-oriented programming, not just some kludge. I want perfect backward-compatibility. I want uniform resource technology from the beginning.

I want a clear separation of core and extensions. Anyone should be able to design, market and succeed with a unique extension. The vendor of this technology must not compete with its own customers.

And then, I want an operating system to complete the picture. I want an operating system built upon a virtual machine, so that I can use one operating system on many hardware platforms.

I want a system programming language. I want a language that will embrace porting from one virtual machine to another. I do not want it to be blind to porting like the Java programming language.

I want an abstract windowing toolkit where events are not tightly coupled to a base class.

I want a vendors to create a virtual machine in an afternoon, something like VMKit. I want many different virtual machines with reliability and compatibility for all.

About JOS

I want a virtual machine that makes it easy to build an operating system. I want a subprogram API that makes it easy to embrace and extend existing programs. I want an open source virtual machine with a well-known pure reflection subsystem and well-known verification subsystem. Why re-invent the wheel? I want no restriction on the development of custom APIs and free exchange of compatible packages.

With the exclusionary architecture created by Sun Microsystems, it is difficult -- if not impossible -- to build a complete operating system. If we were to follow the JavaVirtualMachineSpecification literally, we are not allowed to build an operating system with Java. If we were to follow the license agreement, we are not allowed to build an operating system with Java from Sun Microsystems.

We are underserved by Sun Microsystems. Why do they require us to build a clean-room version of the Java standard class libraries and other standard extensions? Why aren't the Java standard class libraries from JDK 1.02 published as open source?

Why didn't Sun Microsystems add the JavaNativeInterface to JDK 1.02, giving us JDK 1.03? Their architecture excludes all those working on embedded and consumer devices. Sun Microsystems strongly encourages others to build incompatible virtual machines. Their technology leadership is inadequate. And yet, they won't endorse anyone else as technology leader.

Not everyone

Not everyone wants this kind of freedom. Freedom has been rejected by oppressors. For anyone that does not wish to play by the rules of inclusion, they will be unable to find freedom. In the end, their freedom will be taken away.

Summary

With virtual machine technology, someone will be able to build the kind of software platform that ultra-conservative corporate customers(tm) really want. It is the software platform that I want.

Sun Microsystems oppress their customers to gain their own freedom, just like Microsoft.

While persistence through serialization looked good at the time, it has proven to be the disgrace of JDK 1.1 and 1.2. Persistence is not and should have never been based upon Sun Microsystems' favorite approach. Persistence is and should have been an open-ended plug-in, a persistence API. It should never affect every class in the entire standard class library. With a call for XML-based serialization, Sun Microsystems is stuck between a rock and a hard place. Their architecture excluded other competent forms of persistence.
