Introduction

The primary focus of object-oriented programming is using custom objects. To use a custom object, we need a mechanism to create it at runtime. What is a factory? When should it be used?

For an example of a factory for Versioning, see also FactoryExample.

For an example of a factory with URL support, see also FactoryExample2.

For an example of a common factory with URL support, see also FactoryExample3.

Creating an object

There are two patterns used to create an object: constructor and factory.

A constructor is the most highly recommended mechanism. It is a classic pattern. It is direct and easy to use. A constructor is a static method attached to a custom class. Custom objects are created by passing zero or more parameters to a constructor.

A default constructor is by far the easiest way to create a custom object. A default constructor has zero parameters. It is used inside Class.newInstance().

Some custom objects are more complex. So complex they can't be created with a default constructor. This is when other constructors are written using additional parameters.

Constructors can't be used in every case. There are a few problems. Since the constructor is attached to a class, the exact class of a custom object must be known in advance. A constructor won't work if the exact class is unknown. This is a requirement in the use of constructors.

Working with unknown custom objects is supported by dynamic class loading. This works with a default constructor. Or, this works with the introspection API. Dynamic class loading makes a requestor do all of the hard work. There has got to be a better way.

Factory pattern

A factory pattern is a better way. Factory is a buzzword, taken from the real world, meaning a place where things are made. It is given a new meaning in software. A factory is a place where custom objects are made.

By convention, factories are always named the same way. A customFactory is a specialized class with factory methods designed to create custom objects. The name of the custom factory class is known in advance. Loading the factory itself can be performed in the usual way.

Custom objects are created frequently. They are created most frequently using a constructor. Every constructor is also a factory method because it is also a place where custom objects are made. The amount of complexity required inside a constructor varies from one class to the next. What should we do when a constructor gets to be too complex? Move it to a factory. We can move a constructor away from a custom class to a custom factory class.

A factory really shines when working with a set of custom classes, where the specific class to use must be determined at runtime. Step 1: create a factory. Step 2: use the factory to create a custom object.

A requestor can help the factory create a custom class by passing parameters to a factory method or configuring the factory before calling a factory method. Passing parameters to a factory method is almost like passing parameters to a constructor, but, the factory can determine which custom class to use.

A custom factory is also used to create a custom object and all of the custom objects it contains.

A custom factory helps with robust persistence. When loading a custom object, a requestor calls a factory method with a URL. A factory could use a URL, including protocol, host and object name, to load an object from an object store, file and/or database.

A custom factory helps with distributed computing. Since requestors typically don't care if the object is local or remote, a requestor calls a clever factory method. A properly configured factory determines if a custom object should be local or remote. If remote, a factory searches for a custom object across the network and creates a custom proxy. Either way, the requestor gets an object as requested.

The factory pattern is used to expand the scope of what can be done on behalf of a requestor when a requestor wants a custom object. When designing an extendable API, remember the power of a factory.
