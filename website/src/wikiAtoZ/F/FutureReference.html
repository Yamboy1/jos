Introduction

Que çera, çera

Working with a future reference makes me think of these words: Que çera, çera. This is a popular saying in Italian that means what will be, will be. It was used in a song made famous by Dorris Day. If it's spelled wrong, I'm sorry. I don't spell in Italian.
About our world

As human beings, we want to make sense of everything. We have many questions. We want our questions answered. We have invented many devices over the centuries, including computers, to help us understand our world. We want to understand more about our world. Historically, a large part of the funding to build a computing machine comes from people seeking to understand our world.
Prediction

As human beings, we have questions about the future. We want to "know" more about the future. Will I be pretty? Will I be rich? Answering questions like these is difficult, if not impossible. There's no way to say with certainty how things will turn out. Accurate prediction is just beyond our reach.

On the other hand, the computing machine theory is simple:

  • If only we could work on something that we know for sure is never going to change...
  • If only we could collect enough data about its past and analyse it fast enough...
  • If only we could find the right algorythm and process...
  • If only we could find the right model...
  • ...we could predict its future.

Where is it going to be?

It is the same when building an API. We want to "know" how future classes are going to be implemented.

An interface is a future reference. When an interface is created, there are no classes that implement that interface yet. An interface is focused on the future. An interface is created to let a program know where the functionality is going to be. With an interface, the functionality isn't there yet. It is planned. It is promised. It is yet to come.
Divide and conquer

An interface is used to reference many classes, in theory, that haven't been built yet. We use future references to divide a complex puzzle into solvable problems. Imagine that! You can begin to describe part of a class without understanding it in its entirety.

There are rare cases. I have found a few rare cases where there must be only one implementation of a critical class. There is no point in building an interface when there is only one class. It would be a waste of time.
Birth of an interface

Where does an interface come from? Interfaces can come from studying a few existing classes. By far, the most frequent case is building an interface to clean up a collection of existing classes. After having built a few classes, we can see clearly that they have a few things in common we could leverage. When moving from classes to a common interface, we are backing into an interface.

Interfaces can come from a wish list. We can work with the way the classes should work in a perfect world. An interface may be designed with a perfect set of methods, and then, we set out to figure out how to write the methods. When moving from a common interface to custom classes, we are using future reference.

We really want the method to have this name, these parameters and this return value. The method prototype written into an interface doesn't guarantee the method will do what it is supposed to. The purpose of the interface is to open the window of opportunity. At least with an interface, we can get the syntax right. At least, we have a fighting chance.
Modeling helps

Is this going to be the perfect API? Answering a question like this is difficult. Nobody wants to make a world famous API for dead-end software. Modeling helps. We can manage the risk of implementing a dead-end framework by modeling with interfaces.

A future reference helps save time. When modeling an API, we can delay full implementation until we are more comfortable with the interface. We can write sample code against an interface, to see how (and if) it it is going to work. We can use the interface in a snippet of code. By delaying implementation, we reduce the risk of fully implementing some design that's going to be thrown away. When modeling an API, the interface is a great tool because lots of design issues can be worked through without wasting time fully implementing every design.
Like a promise

An interface is something like a promise. There are hints and clues to programmers using the interface about what any future implementation should do. Requestors must call methods of an interface, that's true. But the implementation of an interface can do anything, anything it wants. While requestors must keep their promise, the responder doesn't have to. And this is a good thing. It allows a design to remain flexible and open ended.

Many people say that the Java interface has something to do with multiple inheritence. I disagree, emphatically. It has everything to do with future reference. In practice, COBOL copybooks and C/C++ header files are distant cousins of the Java interface. COBOL copybooks contain hints and clues to programmers using data storage and subprograms. C/C++ header files contain hints and clues to programmers using data structures, modules and libraries. The function prototype defines the signature of a function in the same way an interface defines the signature of a method.

One day in the future, we hope we'll be able to buy high quality software. Inside Realign Software, we believe high quality software must be anchored to an immovable interface. Once an interface has been developed, we must not allow anyone to change it. To keep our promise, we must stubbornly refuse to change an old interface, even under extreme political and technical pressure.

We can always build a new interface, using a new namespace.
