Introduction

The more I think about it, the more it seems to makes sense. Why not build a JVM to run as a service? The best JVM would be able to optimize in a multiple process environment. It would share everything that could be shared without breaking the one-process-one-JVM standard.

Ultimately, such a JVM would separate the system class manager into a shared object (.so or .dll). As a shared object, all instances of the JVM shared one instance of system class manager. This alone would eliminate the redundant loading of system classes for all instances of the JVM, right?

Meanwhile, a JVM service could be written for an existing, non-optimized JVM. It would be able to put a wrapper around a JVM on a foreign operating system.

For example, a JVM service would run inside any JVM, including the jrew.exe program from JavaSoft.

At the other end, a request for a new Java application could turn around and exec any other JVM (from Sun, Borland, IBM, etc, etc.) If the JVM service can exec all kinds of JVMs, it could surely start a JOS JVM, too.

Procedure

The following procedure illustrates how to use a JVM wrapper on a foreign operating system.

  1. Start the JVM wrapper service.

    The JVM wrapper service itself is a quiet service. It runs inside the javaw/jrew, rather than java/jre. The service must be running on a machine in order to make connections to the service and start Java applications.

  2. Start telnet.

    Use your favorite telnet application to open a connection to the JVM wrapper service. Within the JVM wrapper service, each connection starts a JVM wrapper, running inside its own thread. When you make a connection, the JVM wrapper service sends a welcome message to telnet and waits for keystrokes.

  3. Type logout and press the Enter key.

    Each keystoke is sent from telnet to the JVM wrapper service. Each keystroke is turned into a KeystokeEvent and put an event queue. On each enter key, the JVM wrapper runs the given command (alias or URL).

    Application design

    The JVM wrapper provides a special environment for applications. There may be many applications running inside each JVM wrapper. While the JVM wrapper service itself is quiet, applications running within a wrapper have no problem opening windows and running native OS programs.

    Shell written in Java

    This project explores the idea of reading bytes from a socket, turning them into individual KeystrokeEvents, and pushing events to a Java application. It provides an immediate environment for testing part of the JOS application environment. Since the JVM wrapper service is a single JVM running on a foreign OS, the classes written inside this environment provides a way to start many applications within a single JVM.

    An application might open a frame and return. The frame runs in another thread, separate from the JVM wrapper. As soon as an application returns to the JVM wrapper, the JVM wrapper issues a prompt.

    Additional sessions may be started by making additional connections to the JVM wrapper service. When using telnet to start an application, the application is not killed when a connection is lost. Since there are many JVM wrappers running, there should be a mechanism from the telnet side of a connection to switch from one JVM wrapper to another.

    Automatic telnet

    It is possible for the JVM wrapper service to (1) start listening on a port, and then, (2) automatically start your favorite telnet application. As soon as we have a JOS telnet, written in 100% Java, we should be able to run multiple copies of telnet from within the JVM wrapper.

    Configuration

    The JVM wrapper needs its own configuration. It must be able to keep the TCP/IP port (to listen to) and maximum number of concurrent JVM wrappers. In the future, the JVM wrapper service stores a URL in its configuration used to start your favorite telnet application.

    Log

    The JVM wrapper needs its own log. It must be able to keep a record of every command that has been run (plus who and when).

    Shortcut

    A simple Java program (or C/C++ for that matter) should be written to start any application on the JVM wrapper service. In a single command, the shortcut should connect to the JVM wrapper service, send a command with the enter key, and then disconnect.

    GUI

    A GUI program, written in Java, should maintain a long-term connection to the JVM service. It sends any command entered in a text box or a command behind an icon.

    Multiple JVM wrapper services

    Since each JVM wrapper service listens to a single port, it is possible to have multiple JVM wrapper services installed and running on one host. This makes it possible to simulate interaction between multiple domains on existing platforms.

    About System.exit()

    Just like every JVM, a rogue application could call System.exit() and shut down the entire JVM wrapper service. A well behaved security manager should be designed to make it more difficult for a rogue Java application to shut down the JVM wrapper service. And yet, there must be a shutdown command available through telnet to shut down the service.
