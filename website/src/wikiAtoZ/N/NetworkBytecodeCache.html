Introduction

Sometimes I think of the bytecode cache in a client/server model, instead of a statically linked subsystem. Here are some plans for a networked bytecode cache service.

Service

On the server side of the client/server model, a bytecode cache service decodes bytecode on behalf of a client. The service is responsible for the following features:
  • The service loads class files and archives (.cab, .jar and .zip) from its file subsystem. The service deals with codebase, directories and files.
  • The service decodes the class file into all of its component parts. Only a small part of a class file can be retrieved by the client in a single request.
  • The service automatically intern's all class files. It keeps exactly one copy of a class file in memory, regardless of how many copies are stored in a file subsystem. It does not associate a class file with a specific directory.
  • When a client connection dies, the service automatically free's all of the resources used by that client.

Client

Here is how a client works with this service.

The client opens a connection. For simplicity, imagine that a client is reliably connected to the service.

The client uses an open connection to establish one or more class loaders. The service returns a class loader handle for each request to open a class loader. Subsequent use of a class loader includes this server-assigned handle.

The client uses an open class loader handle to add a directory to the class loader. Each directory used by a class loader must be added in a separate request. The service returns an error code if the directory does not exist. The service translates the directory name into its own storage system. Directory names used by the client are symbolic.

The client uses an open class loader handle to establish one or more classes. The service returns a class handle for each request to open a class. Subsequent use of a class includes this server-assigned handle.

The client uses an open class handle determine the following properties of a class.

  • major version
  • minor version
  • constant count
  • constant
  • constant type
  • class access
  • this class spec CP
  • this class spec
  • this class name
  • super class spec CP
  • super class spec
  • super class name
  • interface count
  • interface spec CP
  • interface spec
  • interface name
  • field count
  • field access
  • field name CP
  • field attribute count
  • field attribute
  • field constant attribute
  • method count
  • method access
  • method name CP
  • method name
  • method descriptor CP
  • method descriptor
  • method attribute count
  • method attribute
  • method exceptions attribute
  • method code attribute
  • class attribute
  • class sourcefile attribute
  • and more.
The client is not required to store any of the low-level properties in its own memory address space. Rather, the client stores 32-bit handles for classes and class loaders.

Convenently, zero (0) is an invalid handle. A request for a class or class loader has failed if the service returns a handle equal to zero.
