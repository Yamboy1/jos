CjOSPages

Article contributed by GilbertHerschberger (10 February 2001).

Introduction

For the CjOS host build, the cjvm program is intended to run on Linux and Windows.

Commands

Start the cjvm tools with one of the cjvm.. commands. There will be more than one command to support more than one start-up sequence. The cjvm.. commands will start a common virtual machine. If possible, a virtual machine will be dynamically linked with the cjvm tools.

$ cjvm

Options

The first option on the command line is the name of a custom application class. This will be known as a main class. A main class is the primary class of a custom application.

$ cjvm [ <EM>location</EM> [...] ]

This must not be confused with the main class of the operating system. The main class of the operating system is always invoked regardless of what appears on the command line. The main class of your custom application is passed to the main class of the operating system. The operating system invokes a custom main class after it has established all standard subsystems.

Main OS classes

The main class for the host build will be cjos.arch.host.init and the main operating system class for the i386 build will be cjos.arch.i386.init. The name of the main class is always hard-coded into this virtual machine. It cannot be changed at the command line. In order to change the main class of the operating system, the kernel must be re-compiled.

This virtual machine uses BytecodeNativeInterface to select platform-specific classes. The main operating system class installs a custom BCNI-compatible factory and then always invokes the main method of the init class. The name of an application's main class is passed to the main method of the init class.

Main application class

More than one application can be encoded on a cjvm command line. Each application is invoked within its own bytecode process. Each application gets its own environment, its own primordial class loader, its own System and System properties, its own standard input, output and error streams, its own heap, its own threads (main, ui, finalization and gc).

File system interface

If a custom file system is installed, all applications share a single subsystem. A custom application locates the real file system through the SmartAPI. The standard Java class libraries use the common file subsystem. The file subsystem runs as if in the primary domain, process 0. This is owned exclusively by the main operating system class.

Graphical user interface

If installed, all applications share a single subsystem for their graphical user interface. The main operating system class will install a real GUI. The real GUI will provide a virtual GUI for all other applications. A virtual GUI locates the real GUI through SmartAPI. The real GUI runs as if in the primary domain, process 0. This is owned exclusively by the main operating system class.
