Article contributed by GilbertHerschberger (16 November 2000).

Introduction

There is a difference between classpath and codebase. What is it? Both are a list of directories/archives where .class files can be loaded. Both use a platform-dependent directory separator.

Applet

A codebase is most often heard when speaking of Java applets, not applications. The early browsers required applets to keep codebase completely separate from classpath. While codebase is an applet concept, I wonder what would happen if it were standard in Java applications.

Servlet

A codebase is important to Java servlets as well. A separate class loader is expected to load each servlet so that servlets can be unloaded without taking down the entire HTTP service.

Application

An application would benefit from codebase, too. The classpath property should be reserved. It should only contain the boot packages. All other packages could be loaded with a custom class loader, and therefore, a unique codebase.

With the introduction of the Java 1 platform, it has been possible for a custom class loader to load class files directly from an archive (.jar or .zip). Standard extensions, such as Swing, can be loaded through codebase, and should be.

java tool

I dislike the java tool. Instead of loading an application through a primordial class loader, the java tool should always load an application through a custom class loader. The codebase property should be passed to the custom class loader through the java tool. If this is used consistently, classpath never needs to be modified.

java -codebase=myapp/classes myapp

When an application launches another, it can create a second class loader and therefore a second codebase. Two applications running in the same virtual machine can have different codebase properties.

In bytecode

Many of these problems can be solved in bytecode. The codebase property is useful to Java applications. The introduction of the archive (.jar and .zip) did not require a machine code solution after all.
