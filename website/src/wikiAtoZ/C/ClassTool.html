Article contributed by GilbertHerschberger (9 December 1999).

Introduction

The instanceof operator is very useful when your program is working with objects. Many programs, like Controllers from the SmartAPI work with classes, not objects. In a single request, an old-style controller might create an instance of a class and then throw it away because it is not an instance of the target interface.

A new-style controller can predict an object will be an instance of the target interface by using a class tool.

Class tool

The class tool is a separate, isolated tool, with no unnecessary connections to the java.lang.Class class. It is easy to use in your program when working with classes. 

public interface ClassTool {
  public abstract boolean instanceOf( Class c, String[] list );
}

The instanceOf method recursively scans a class, all of its interfaces and super classes. It returns true if any interface/class name in list is implemented. It is not used to determine which interface/class name is implemented.

Basic class tool

Since a class tool might be implemented as a JVM-specific class, I've separated a basic class tool from its class tool interface.

public class BasicClassTool 
	 implements ClassTool {
  private static final String DUMMY_VALUE = "";
  public BasicClassTool() {
  }
  public boolean instanceOf( Class c, String[] list ) {
	 Hashtable cache = new Hashtable();
	 int iMax = list.length;
	 for ( int i = 0; i < iMax; i++ ) {
		cache.put( list[ i ], DUMMY_VALUE );
	 }
	 scanClass( c, cache );
  }
  protected boolean scanClass( Class c, Hashtable v ) {
	 if ( c == null ) {
		return false;
	 }
	 if ( v.get( c.getName() ) != null ) {
		return true;
	 }
	 Class sc = c.getSuperclass();
	 if ( sc != null ) {
		if ( scanClass( sc ) ) {
		  return true;
		}
	 }
	 return scanInterfaces( c, v );
  }
  protected boolean scanInterfaces( Class c, Hashtable v ) {
	 Class[] list = c.getInterfaces();
	 int iMax = list.length;
	 for ( int i = 0; i < iMax; i++ ) {
		if ( scanClass( list[ i ], v ) ) {
		  return true;
		}
	 }
	 return false;
  }
}

Native JVM-specific class

When a process requests an implementation of a ClassTool, a JVM might return a JVM-specific class, something like this:

public class NativeClassTool 
	 implements ClassTool {
  private native static boolean scanClass( Class c, String[] list );
  public NativeClassTool() {
  }
  public boolean instanceOf( Class c, String[] list ) {
	 return scanClass( c, list );
  }
}

It is important to note that a well designed interface does not presume a method is native. Neither does it presume that a instance method is native. A instance method can be passed to a static method, as shown in this example.

Getting a class tool

Your application can get a copy of a class tool throught the BytecodeNativeInterface. The following example method prints out Result: true when invoked: 

  public void example() {
	 URI u1 = new URI( "class:java.awt.Button" );
	 Class c = (Class) u1.getObject();

	 URI u2 = new URI( "bcni:ClassTool" );
	 ClassTool tool = (ClassTool) u2.getObject();

	 String[] list = {
		"java.awt.Component"
		};

	 System.out.println( "Result: " + tool.instanceOf( c, list ) );
  }

Integration of class tool

Since this is a recurring puzzle throughout all controllers and custom applications, it makes sense to expand the scope of the smart class controller to automatically perform this service. A simple way to do this is illustrated in the following example: 

  public void example() {
	 URI uri = new URI( "class:java.awt.Button" + 
		  "?instanceof=java.awt.Component" );
	 Class c = (Class) uri.getObject();

	 System.out.println( "Result: " + ( c == null ) );
  }

About pure reflection

The class tool uses a class loader to predict when one class is an instance of another. It does not use pure reflection. Pure reflection cannot accurately predict the behavior of a class loader.

Improvement

All of the programs already written to use the Smart API will automatically benefit from new-style controllers -- without modification. The semantics of the Smart API subsystem are improved. Using a class tool, new style controllers will eliminate a lot of extra work. An instance of a class will be delayed until a class has been checked to match the required interface.

Wow! That didn't take long at all. I have written and tested a class tool. I hope to make it available soon, along with a new smart class factory.

Here is a long-hand example of using a class tool alone. (This example works with out installing the new smart class factory.)

  public Button getButton( String cn ) {
	 URI uri = new URI( "class:" + cn );
	 Class c = (Class) uri.getObject();
	 if ( c == null ) {
		return null; // class is missing
	 }

	 String[] list = {
		"java.awt.Button"
		};

	 ClassTool tool = new BasicClassTool();
	 if ( !tool.instanceOf( c, list ) ) {
		return null;
	 }

	 return new URI( "object:class:" + cn ).getObject();
  }

Here is a short-hand example of the same thing, after your SmartAPI is configured with a new-style smart class loader:

  public Button getButton( String cn ) {
	 URI uri = new URI(
		  "object:class:" + cn + "?instanceof=java.awt.Button" );
	 return (Button) uri.getObject();
  }

The new-style smart class loader depends on the class tool. I am writing the class tool into the smart class loader package. In this example, it returns a button class if cn is the name of a class that extends java.awt.Button. -- GilbertHerschberger (10 December 1999).
