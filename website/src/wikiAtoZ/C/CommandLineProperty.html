Introduction

Imagine how easy it might be if the java tool were written properly. What else would it do? This article discusses the proposed java.command system property.

Command line

A command line is always used to invoke the java tool. The exec() function in the standard C library accepts a command line string. A command line is passed to the kernel in Etherboot and GRUB.

Why needed

One virtual machine can invoke another. The exec() method of the java.lang.Runtime class enables a virtual machine to invoke other machine code programs, including another virtual machine. Unfortunately, a virtual machine doesn't yet provide your Java method with enough information to invoke another virtual machine using the same command line options. What is the solution?

A java tool should pass the entire command line to the virtual machine as a system property, giving your Java application enough information to launch another virtual machine. Any method could examine the system property, like this:

  public String getCommandLine() {
	 Properties p = System.getProperties();
	 String n = "java.command";
	 return p.getProperty( n );
  }

If desired, an application could launch another instance of itself, using exactly the same command line, like this:

  public void example() {
	 URI uri = new URI( "run:os:" + getCommandLine() );
	 Program p = (Program) uri.getObject();
	 p.run();
  }

Old java tool

To pass a command line with an old java tool, the command line must be encoded. As a quoted string, the -D argument could be used to set an arbitrary system property, such as java.command.

The virtual machine is unaware of the original command line:

java myApp

To make it aware, pass the command line as a system property:

java -Djava.command="java myApp" myApp

Inside the virtual machine, the command line system property is the original, excluding the -Dcommandline= option.

java myApp

Off-the-shelf virtual machine

When one virtual machine invokes another, it should always pass the command line through the -Dcommandline= option.

Multiple bytecode process

Inside an MPCL-compatible virtual machine, the command line property should be automatically passed from one virtual machine to another.

The eric tool

When a virtual machine is created by the eric tool, it will automatically pass the original command line to the virtual machine.

eric location=run:program:org.jos.demo.DemoProgram

Inside the virtual machine, the eric.command system property matches the command line exactly:

eric location=run:program:org.jos.demo.DemoProgram

Shortcut for java tool

Using java -option -Dname=value main parameter as a guide, the following class illustrates the properties of a shortcut for the java tool.

public interface JavaShortcut {
  // about options
  public int getOptions();
  public String getOption( int index );
  public void setOption( int index, String v );

  // about system properties
  public int getSystemPropertyCount();
  public String getSystemProperty( int index );
  public String getSystemProperty( String n );
  public void setSystemProperty( String n, String v );

  // about java.class.path
  public String getClassPath();
  public int getClassDirectoryCount();
  public String getClassDirectory( int index );
  public void addClassDirectory( String v );

  // about main class
  public String getMainClass();
  public void setMainClass( String cn );

  // about parameters to main()
  public int getParameterCount();
  public String getParameter( int index );
  public void setParameter( int index, String v );
}

When the command line system property is available, an instance of JavaTool can be created from it, possibly modified and passed to Runtime.exec(). The eric tool has a corresponding class called EricShortcut.
