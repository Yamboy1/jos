Introduction

After a review of the relationship between CODEBASE and CLASSPATH, I started playing with the design of a codebase tool. In this article, I'll share a little of what I've been thinking.

By command line

The java tool manages CLASSPATH by the command line. A classpath property is passed to a virtual machine through the java tool. In practice, this requires each application to configure a command line to pass the right value of the classpath property.

By environment variable

Another approach is possible. Instead of passing CLASSPATH on the command line, it is possible to set CLASSPATH as an environment variable.

By configuration

I suggest that it would be far easier to pass CLASSPATH and CODEBASE properties through configuration file(s). Here is how it might work if the java tool were changed.

  1. The tool should read its default CLASSPATH property from a configuration file. Each java tool looks in its own program directory for the classpath.conf file. Typically, the file is called java/bin/classpath.conf.

    If the classpath.conf file exists in the current directory, it must be ignored.

  2. The tool should always invoke a well-known method inside the Java standard class library. It should never directly invoke a method of an arbitary class. Rather, the main class passed on the command line should be passed to a well-known method inside the standard class library. The standard class library must always be available on CLASSPATH.

  3. This well-known method would read its CODEBASE property from a file. It would look first to the current directory for a codebase.conf file. If a file does not exist in the current directory, it should look elsewhere. The CODEBASE property is application-specific. A custom class loader is always used to load classes on CODEBASE.

    Since the custom class loader is written in the Java programming language, it is capable of reading classes across an FTP or HTTP connection.

    Maintaining codebase property

    The codebase property is stored in codebase.conf. The codebase tool is an interface to the codebase.conf file, thus maintaining the codebase property. For example, it does not allow duplicate directories. It validates the codebase.conf file to make sure every directory and every archive actually exists.

    • List

      To list the contents of the codebase.conf file, use the following command:

      codebase

      When invoked, this command will provide something like the following list:

      codebase program (1a)
Type		Status  Name		  
--------- ------- -----------
directory OK		/usr/local/share/classes
directory MISSING /usr/local/missing
archive	OK		/usr/local/lib/some.jar
directory NETWORK http://www.jos.org/redist/classes
archive	NETWORK http://www.jos.org/redist/lib/some.jar
package	OK		/usr/local/jpkg
package	NETWORK http://www.jos.org/redist/jpkg

    • Add directory

      To add a directory, use the following command:

      codebase d=<EM>some-directory</EM>

      where some-directory is the name of a directory on the local machine, or some-directory is the uniform resource location of a directory across the network.

    • Add archive

      To add an archive, use the following command:

      codebase a=<EM>some-archive</EM>

      where some-archive is the name of an archive on the local machine, or some-archive is the uniform resource location of an archive across the network.

    codebase.conf example

    The configuration file is a data registry-compatible file. It is a plain-text file, edited by hand with your favorite text editor. Here is an example of its contents:

    s0=directory,/usr/local/share/classes
s1=directory,/usr/local/missing
s2=archive,/usr/local/lib/some.jar
s3=directory,http://www.jos.org/redist/classes
s4=archive,http://www.jos.org/redist/lib/some.jar
s5=package,/usr/local/jpkg
s6=package,http://www.jos.org/redist/jpkg

    classpath.conf example

    The classpath.conf is also a data registry-compatible file. Its format is identical to codebase.conf.

    Custom class loader

    A custom class loader reads classes using a custom configuration. The custom configuration is passed to it. For each source, the custom class loader creates a class or package provider. While there is only one custom class loader, it can efficiently load classes from multiple sources and formats.

    Full circle

    The well-known method invoked by the new java tool is just another method. It can be invoked by your Java program to launch another program with a separate codebase. If by some mechanism CLASSPATH were mostly empty, static fields of system classes would be unique for each custom class loader.

    • java.lang.Object
    • java.lang.Class
    • java.lang.ClassLoader
    • java.lang.String
    • altlang1a.CustomClassLoader

    This would result in nearly an MPCL-compatible virtual machine.
