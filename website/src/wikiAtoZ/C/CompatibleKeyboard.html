Introduction

What we would like to have is a keyboard driver generate scancode events directly. The kernel would pass scancode events on to the current process.

KeyboardDriver -> kernel -> current process

We would like StandardInputStream to convert scancode events back to an InputStream for standard Java applications.

KeyboardDriver -> kernel -> StandardInputStream -> app

Meanwhile...

What are we going to do now, while the KeyboardDriver and JOS kernel aren't finished? We don't have to wait. We can experiment now with event driven keyboard driver. Here's how:

A JVM-compatible keyboard is the first link in the chain. CompatibleKeyboard reads an InputStream, like System.in, and generates scancode events.

System.in -> CompatibleKeyboard -> ScanCodeListener

We can test StandardOutputStream, too. StandardInputStream converts scancode events back to an InputStream. Each keystoke takes a round trip.

System.in -> CompatibleKeyboard -> StandardInputStream -> app

Starting with the source code below, you should be able to build a class that reads System.in and does something with every keystroke.

Listener

Build a ScanCodeListener, like this:

public interface ScanCodeListener {
  public void onScanCode( int v );
}

Test class

Run a test class, like this, to demonstrate scancode events directly from the keyboard. As an alternative, the compatible keyboard can read from any input file. You could create a file containing all of the possible keystrokes. CompatibleKeyboard should generate appropriate scancodes.

public class CompatibleKeyboard_Test
	 extends ScanCodeListener {
  public static void main( String[] args ) {
	 CompatibleKeyboard_Test app =
		  new CompatibleKeyboard_Test();
	 app.run();
  }

  public void run() {
	 System.out.println( "Press the Escape key to exit." );
	 CompatibleKeyboard keyboard =
		  new CompatibleKeyboard( System.in, this );
	 keyboard.run();
  }

  public int onScanCode( int v ) {
	 System.out.print( "scancode=" + v );

	 if ( v == 27 ) {
		System.exit();
	 }
  }
}
