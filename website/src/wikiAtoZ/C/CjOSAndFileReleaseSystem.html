About this document

This document provides more information about moving our distributable files to the File Release System of sourceforge.net. It provides an overview of our current plan to reduce the amount of storage space required by our project website.

To upload

The following procedure illustrates how to upload a file to the File Release System.
  1. FTP to upload.sourceforge.net
  2. Login as "anonymous"
  3. Use your e-mail address as the password for this login
  4. Set your client to binary mode ("bin" on command-line clients)
  5. Change your current directory to /incoming ("cd /incoming")
  6. Upload the desired files for the release ("put filename")

Website

Introduction

This section provides more information about our project website.

Purpose

Our project website at http://cjos.sourceforge.net has served more than one purpose. Here is a clarification of its purpose.
  • Our project website is intended for the general public. It provides browse-able information about our project. It does not provide detailed or members-only information that is more suitable to JOS Wiki.
  • Our website contains important publically available documents, such as project constitution. It contains important historical documents, such as mailing list archives. These documents are maintained through a project webmaster. They are valuable and not available elsewhere.

Non-purpose

Our website has also provided other files. The download directory contained related files provided by others and other projects. The mirror directory contained files created by our project. Here is a clarification.
  • Our website should not contain non-browsable files. It should contain neither binary distributions nor source distributions of our products.
  • Our website should not include related files provided by others and other projects. Rather, it should provide links to these files.

Plan

Here is our plan for moving from a large (640MB) to a small (100MB) website.
  1. Delete the download directory. (Completed.)
  2. Delete the mirror directories. (Completed.)
  3. Delete the download-able CD-ROM image of our website. (Completed.)
  4. Create an archives for each mirror directory. The content of the mirror/jos1m directory, for example, becomes a joste1m.zip archive.
  5. Use the File Release System of sourceforge.net to re-release an mirror archive. The package name is mirror.
  6. Use the File Release System to re-release the download-able CD-ROM image of the JOS Technical Edition.
  7. Fix our website. Download pages should include links to the File Release System.
  8. Fix articles in JOS Wiki. Download pages should include links to the File Release System.

Summary

By clarifying the purpose of our website, we expect to reduce its size and keep its size manageable in the furture.

Packages

Introduction

This section provides more information about the packaging of our Project.

Design

Packaging is an important part of our project. We have been working on a new package design. We have decided to take this opportunity to re-think and re-work our package design before we commit to packaging inside the File Release System.
  • Build with Apache Ant.
  • Reduce the number of target jars.

Ant

The CVS archives reflect most of our new packaging. For example, our Ant scripts are maintained in CVS.

Target jars

To reduce the number of target jars, certain jars have been combined. A new naming convention has been designed. Here are a few examples.
  • The cjos-khimera.jar is a combination of cjos-khimera1a.jar, cjos-khimera1b.jar, khimera1c.jar, khimera1d.jar.
  • The cjos-smartapi.jar is a combination of cjos-smartapi1a.jar, cjos-smartapi1b.jar, cjos-smartapi2a.jar and cjos-smartapi4a.jar.
Jars that contain multiple versions of a product or API will have a version-less name.
  • The cjos-jose.jar is a combination of versions 1a, 1b, 1c and 1d. It replaces cjos-jose1d.jar, which looks as if it were only version 1d but is not.

Target wars

To reduce the number of target wars, a war file should contain a jar with multiple versions of a servlet. The distributed web.xml file should use the recommended version by default. To switch to another version, a customer can (easily) modify the web.xml file.

Release

A release of a product (or API) should include both a binary and source distribution.
  • Each release should have a unique release number.
  • This can be partially automated. An Ant script should be used to create the binary and source distribution files from the corresponding CVS.
