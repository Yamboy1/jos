API

Article contributed by GilbertHerschberger (10 June 1999).

Introduction

I have been using a few other techniques to find the current implementation. One of these other techniques is editing and recompiling a program.

Very popular

This technique is the most popular technique for selecting the current implementation of an interface. This technique will continue to be used by many programmers, and should be. It will not be replaced.

While you are maintaining a program, there are opportunities to use new classes from new packages that didn't exist before. You may have learned about new built-in classes. You may have downloaded class from others in the JOS project. You may have purchased a new package. You may have upgraded an old package. Each of these events is an opportunity to make a program better.

You can decide that your program should use classes from a different package this time. It's alright. That's what you're supposed to do.

Switching APIs

A rewrite is primarily used to switch APIs. There comes a time in every programs life when it needs to switch to the latest API, so that it doesn't need a bridge. Bridges and run-time compatible APIs help us ease the transition from one API to another. They are a long term approach, not a long term solution.

Smart API

The demand for high quality software far exceeds the supply. There are a lot of new programs to write. Programs must go where no program has gone before. Who has time to rewriting all those old programs? We cannot afford to rewrite every program with each new invention, with each new technique, with each new technology.

A good API lets you rewrite old programs less often. A great API lets you rewrite your old programs as least often as possible.

Tomorrow

The future holds more refinement. We can build more reliable, robust and stable implementations. A smart API changes the way we think about software. We can finish our program now, with a smart API and functional implementations. We expect to writing a better implementation sometime in the future.
