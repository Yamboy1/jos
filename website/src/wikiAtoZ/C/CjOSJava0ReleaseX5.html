Introduction

This is part of the effort to migrate packages away from the com.realignsoft namespace toward the gchii and org.jos namespaces.
  • Note: A stub jar (stub-mc.jar) is required to compile the gchii.si.mastercompiler2a.bridge package.

Master Compiler 2a

The latest version of the Master Compiler is G-2a. There are nine (9) versions of the Master Compiler. Of these, two are not bridge-able.
  • R-1a (com.realignsoft.jose.mastercompiler1a)
  • G-1a (gchii.si.mastercompiler1a)
Seven (7) are.
  • R-1b (com.realignsoft.jose.mastercompiler1b)
  • R-1c (com.realignsoft.jose.mastercompiler1c)
  • R-1d (com.realignsoft.jose.mastercompiler1d)
  • G-1b (gchii.si.mastercompiler1b)
  • G-1c (gchii.si.mastercompiler1c)
  • G-1d (gchii.si.mastercompiler1d)
  • G-2a (gchii.si.mastercompiler2a)

Bridge

A bridge class provides a bridge between a well-known class and an optional, process-specific class. The bridge class typically implements a well-known class and uses the optional class. Sometimes a bridge class is an instance of the both the well-known and optional classes.

A bridge class is expected to depend on the classes for which it provides a bridge. In other words, a bridge class should not load if an optional class does not load.

The bridge classes in the gchii.si.mastercompiler2a.bridge package did not yet implement this behavior as expected. For example, the bridge for version R-1d (gchii.si.mastercompiler2a.bridge.CompilerR1d-Holder) would mistakenly load even when the base compiler class (com.realignsoft.mastercompiler1d.BaseCompiler) could not be loaded.

Compiler G-1d

The bridge class for version G-1d (gchii.si.mastercompiler2a.bridge.CompilerG1d-Holder) should depend on both gchii.si.jose1d.Console and gchii.si.mastercompiler1d.BaseCompiler.

With release x5, it is a little better. This bridge assumes that if the console class (gchii.si.jose1d.Console) exists, so does the base compiler class (gchii.si.mastercompiler1d.BaseCompiler).

The bridge classes in gchii.si.mastercompiler2a.bridge did not implement this behavior as expected. Rather, the bridge for com.realignsoft.mastercompiler1d would mistakenly load even when com.realignsoft.mastercompiler1d.BaseCompiler could not be loaded.

With this release, it is only a little better. This bridge assumes that if the console class (com.realignsoft.jose1d.Console) exists, so does the base compiler (com.realignsoft.mastercompiler1d.BaseCompiler) class.

Summary

There is more work to do here. Is the base compiler class abstract? If so, it cannot be instantiated directly. Something within a holder class should depend directly on the corresponding base compiler class.
