Article contributed by GilbertHerschberger (1 March 2001).

Introduction

The choice of a programming language has greatest implications for a software product. The CJOS project is expected to use C, C++ and Java. How does a programmer choose where to implement some functionality? This article describes where C, C++ and Java can be used successfully to cover the entire scope of an operating system.

C

The C programming language should be used between subsystems written in machine code. This language lends itself well to easily enabling one subsystem to using functionality and objects encapsulated in another subsystem. Subsystems are correctly coupled through C functions. Structures and unions are rarely exposed. Header files contain function prototypes, which define a formal interface between subsystems.

Factory methods should be written as C functions and exposed in a header file. A factory method should return a C++ object, or better, a handle to a C++ object.

The C programming language is used to implement a standard C library in machine code. This includes well-known functions such as open() and close(), read() and write(). It includes the C string functions (strcmp(), strcpy(), strlen()) and C memory functions (memcmp(), memcpy(), memset()).

C++

The C++ programming language should be used within a subsystem written in machine code. These subsystems further break down the functionality of a kernel and virtual machine. This language lends itself well to tightly-coupled, object-oriented subsystems. Methods (functions) should be attached to a class (structure) in the style of object-oriented thinking.

C++-style exceptions should not be used at this time in the i386 build. Creating the corresponding support for exceptions is beyond the scope of the current project.

Standard class library, templates and a standard template library should not be used in the i386 build. Standard C++ class library is replaced by standard Java class library.

Java

The Java programming language should be used for standard Java class library, high-level device drivers, and applications. This language should be used as much as possible, without sacrifice of common sense and clean design. Low-level device drivers will be written in C++ and exposed to Java through C functions.

Bytecode will be developed in parallel with machine code. For example, bytecode is a separate CVS module. The bytecode will be compatible with the Bytecode Native Interface (BCNI).

Summary

The C, C++ and Java programming languages can be appropriately combined to give this project the strengths of each. C++ provides the best encapsulation of machine code objects. C reduces the coupling between C++ classes, especially when classes are part of separate subsystems. Java bytecode provides platform independence and a foundation of extreme reuse and perfect backward compatibility.
