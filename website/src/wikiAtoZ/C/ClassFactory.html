Introduction

After using a factory pattern for editors and other objects, it begins to make sense to use a factory pattern for Java classes themselves.

This article illustrates the class factory version of the java.lang.ClassLoader. While you should be aware of the java.lang.ClassLoader and class factory, a BytecodeFactory is the preferred mechanism.

A class factory creates classes. A class factory is an interface, like this: 
public interface ClassFactory {
  public boolean acceptsURL( String url );
  public Class getClass( String url );
}

Example 1

The following example demonstrates how the class factory is used. 
  public void example() {
  String cn = "org.jos.editor.PlainTextEditor";
  String prefix;
  Class c;

  MasterClassFactory factory =
		new MasterClassFactory();
  factory.addFactory( new SystemClassFactory() );
  factory.addFactory( new HTTPClassFactory() );
  factory.addFactory( new ApplicationServiceClassFactory() );

  [1]
  prefix = "x-class:system:";
  c = factory.getClass( prefix + cn );

  [2]
  prefix = "x-class:http://host.domain.com/";
  c = factory.getClass( prefix + cn );

  [3]
  prefix = "x-class:app://host.domain.com/";
  c = factory.getClass( prefix + cn );
}
  1. The system subprotocol is used by a system class factory to load a class from the local drive.
  2. The http subprotocol would use a (mythical) HTTP class factory to load a class from an HTTP service.
  3. The app subprotocol would use a (mythical) class factory that downloads classes from an application service. An application service is a service optimized for the storage of Java bytecode.

A system class factory implements the ClassFactory interface and finds classes on a local drive, like this: 
public interface SystemClassFactory {
  public static String protocolPrefix = "x-class:system:";
  public SystemClassFactory() {
  }
  public boolean acceptsURL( String url ) {
	 if ( url == null ) {
		return false; 
	 }
	 if ( !url.startsWith( protocolPrefix ) ) {
		return false;
	 }
	 int pos url.indexOf( "//" );
	 if ( pos >= 0 ) {
		return false;
	 }
	 return true;
  }
  public Class getClass( String url ) {
	 if ( !acceptsURL( url ) ) {
		return null;
	 }
	 String cn = 
		  url.substring( protocolPrefix.length();
	 return loader.findClass( cn );
  }
  private PrimordialClassLoader loader =
		new PrimordialClassLoader();
}

A master class factory implements the ClassFactory interface and dispatches requests for a class to any registered class factory. Programmers extend this design with custom class factories. 
public interface MasterClassFactory
	 implements ClassFactory {
  public MasterClassFactory() {
  }
  public boolean acceptsURL( String url ) {
	 int iMax = manager.getCount();
	 for ( int i = 0; i < iMax; i++ ) {
		ClassFactory item = manager.getItem( i );
		if ( item.acceptsURL( url ) ) {
		  return true;
		}
	 }
	 return false;
  }
  public Class getClass( String url ) {
	 int iMax = manager.getCount();
	 for ( int i = 0; i < iMax; i++ ) {
		ClassFactory item = manager.getItem( i );
		Class c = item.getClass( url );
		if ( c != null ) {
		  return c;
		}
	 }
	 return null;
  }
  public void addFactoryByName( String v ) {
	 Class c = getClass( "x-class:system:" + v );
	 if ( c == null ) {
		return;
	 }
	 Object o = c.newInstance();
	 if ( !( o instanceof ClassFactory ) ) {
		return;
	 }
	 addFactory( (ClassFactory) o );
  }
  public void addFactory( ClassFactory v ) {
	 removeFactory( v );
	 int index = manager.getCount();
	 manager.setItem( index, v );
  }
  public void removeFactory( ClassFactory v ) {
	 int iMax = manager.getCount();
	 for ( int i = 0; i < iMax; i++ ) {
		ClassFactory item = manager.getItem( i );
		if ( item == v ) {
		  manager.setItem( i, null );
		}
	 }
  }
  private ClassFactoryManager manager =
		new ClassFactoryManager();
}

Problems and more problems

In this design, there are severe and insurmountable problems.

  • This design does not consider the implementation of a class loader. It leaves the class loader up to the individual class factory. This becomes a problem when classes from one source are mixed with classes from another.
  • A class factory implies that there are different ways of creating a class, when there's not. In fact, there is only one way to create a class--through a class loader.

Bytecode factory

Classes are created from bytecode. Bytecode has to be stored somewhere. What we're looking for is a bytecode factory, not a class factory. Using the same pattern, we can create custom bytecode factories that retrieve bytecode stored in lots of different containers. We can build one class loader that uses a bytecode factory to create classes.
