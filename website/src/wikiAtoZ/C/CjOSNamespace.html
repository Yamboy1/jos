CjOSPages

Article contributed by GilbertHerschberger (4 January 2001).

Introduction

The latest issue of Dr. Dobb's Journal is special. It focuses on Java technologies. It also describes how Java has failed to meet the needs of the software industry.

Someone else has already proven that an alternative namespace is feasible. The Waba project was features in this journal. The Waba project has thrown out the convoluted, confused and irreparibly damaged java namespace and created its own. Like the AltPackageProposal, it suggests that a Java virtual machine depends on the classfile format more than anything inside the standard Java class libraries.

oak namespace

Remember when "Java" was called "Oak"? Before Java was called "Java", it was called "Oak". The classfile format was defined. Opcodes were defined.

"Oak" was upgraded to "Java". The classfile format remained the same. Opcode definitions remained the same. The only thing that changed from "Oak" to "Java" was moving classes from the oak namespace to the java namespace.

The "Oak" compiler was written in machine code. Later, the "Java" compiler was re-written in bytecode so that it too could be "platform" independent.

java namespace

At one time, it might have been possible to salvage the java namespace. With the release of JDK 1.3, this is no longer possible. Sun Microsystems has increased the complexity to the point where perfect compatibility is no longer feasible.

waba namespace

The waba namespace is simpler than JDK 1.02. All optional packages are not part of the waba namespace, making it possible to maintain perfect compatibility and provide applications with all of the latest and greatest features.

cjos namespace

Let's start with a working virtual machine, such as KISSME or TEASEME. If it were modified so that it depends on the cjos namespace, we could maintain perfect backward compatibility across all versions of CjOS. It would abandon the chaos, confusion and complexity that Sun Microsystems introduced in the java namespace.

Like Java and Waba, it would have a language package called cjos.lang. Like Waba, it would have a system package called cjos.sys.

Bottom up

Within a working virtual machine, every reference to java/lang/Object should be changed to cjos/lang/Object. Every reference to java/lang/Class should be changed to cjos/lang/Class.

At first, the new virtual machine might not support a class loader in bytecode. It might only support only a primordial class loader for each bytecode process.

Common tools

Like Waba, CjOS uses the classfile format from Java. Like Waba, CjOS uses the opcodes defined by Java. Like Waba, CjOS defines an alternative namespace, and yet, remains compatible with an off-the-shelf Java compiler. Like Waba, CjOS applications run with either an off-the-shelf Java virtual machine or a CjOS virtual machine.
