Article contributed by GilbertHerschberger (31 May 2002).

Introduction

After installing Caldera OpenLinux incorrectly for the third time, I started thinking. We can safely assume that our customers already have a PC-compatible operating system and a Java Runtime Environment (JRE). Why not take full advantage of that? Then it becomes possible to write an installation program for Linux and ReactOS in 100% Pure Java.

Here is how it might work. The installation program must collect information about the computer and about the end user's choice of applications, services and drivers. Suppose you could build a program in Java to collect this information. It could make a good impression. It would not be limited to real memory. The sky is the limit.

Once the Java program has collected all of the relevant information, it might store the target system configuration to disk. It could be stored as a single XML-compatible file. In turn, a bootable installation program could be written in the C/C++ programming language. It follows the instructions generated by the Java program to install an operating system.

Drivers

In the Java program, plug-ins can be written for each driver. A plug-in would present a dialog box to collect its configuration information. Here is where reuse is possible. After the operating system has been installed, the same dialog box can be reused to re-configure the driver. When running in the installation environment, the plug-in writes its information to a target system configuration. When running in the target environment, the plug-in writes its information to current system configuration.

Anyone can write a plug-in by using a well-known interface provided by the author of the Java installation program.

By writing all of these programs in Java, they are platform independent. Configuring an operating system works identically regardless of whether the end user runs Linux, Mac OS/X or Microsoft Windows.

Two steps

Operating system installation should always be performed in two steps.
  1. This step should include collecting preference and information about the target system configuration. This is where the end user should browse a list of applications that could be installed. Here is where they determine how a machine will be used. Should it have X? Should it have KDE? Should it have an HTTP service? FTP? BOOTP?
  2. This step should implement the target system configuration on a machine. Some of the hardware-specific choices must be determined at this point and not before. Hardward-specific choices include EIDE vs. SCSI, hard drive and video card. The choice of which applications should be installed and how the machine is going to be used should not be decided here.
