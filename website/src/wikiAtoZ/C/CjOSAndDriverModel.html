Article contributed by GilbertHerschberger (31 May 2002).

Introduction

According to what I have read about JINI, every driver must provide its own dialog box because every driver is different. Since we're talking about Java's technology, it becomes possible for the author of a driver to provide a platform-independent dialog box class for users to configure the driver.

What else do all drivers have in common? Very little. That is why it has been such a challenge for people to write a popular driver framework. There are all kinds of drivers. It is difficult to classify and catagorize all of them.

Here is what I have found so far. There is little difference between a driver and an application program and a service. While we think of them differently, the computer does not.

Imagine you have a word processor written in the Java programming language. It might support plug-ins to handle additional word processing tasks, such as a spell checker and mail merge plug-in. These plug-ins are drivers in the sense that they were unknown by the author of the word processor and they work to extend the word processor in specific ways.

Now we can apply that to an operating system kernel. A kernel is a lot like a word processor with plug-ins. A plug-in for a kernel extends the kernel in specific ways. In a kernel, we call them drivers. They are programs designed to run in a specific environment. Plug-ins for a kernel should implement a well-known driver interface so that the kernels driver manager and driver environment can load an arbitrary driver by name.

Most drivers must be loaded in a specific order. To load drivers in the right order, the kernel should invoke a well-known program. This program can invoke other programs. These programs are invoked in a primative environment and are limited to certain classes and packages.

Stream-based

Many, but not all, drivers are stream-based. Every stream-based driver should implement a well-known "stream-based-driver" interface. This is one catagory of driver. Additional catagories are possible, but the kernel may not be required to use additional classifications.

Driver manager

The kernel provides a driver manager. It might be beneficial for the driver manager in kernel-space to be browse-able so that existing drivers can be identified by applications in user-space. From user-space, an end-user can invoke a program which loads an arbitrary driver into the kernel's driver manager.

Threads

Drivers should depend on kernel thread(s). The driver manager is a passive object; it does not invoke methods of a driver. The kernel invokes methods of the driver manager and drivers.

Dynamic install

When an end-user invokes a program, it can load an arbitrary driver into the kernel's driver manager. The program must provide a copy of the driver's class and, if necessary, additional classes it depends on. It is very likely that the kernel's java.class.path is different than the user's java.class.path. The kernel's java.class.path must includes all the driver classes which must be loaded at boot-time. Drivers depend on the driver environment provided by a kernel. It is not likely that a driver class could be successfully invoked in user-space.
