SimpleJOS; JVMSpecificClasses

Article contributed by GilbertHerschberger (23 June 1999).

Introduction

When Sun Microsystems wrote their own implementation of the Java API, they went directly from the Java API to native methods. If only they had used the power of interfaces and factories to create a Java API and a JVM API. Vendors could have implemented JVMSpecificClasses and simply re-distributed the Java API. That's exactly what we want to do for JOS.

On the web

The GNU Classpath project can be found on the web at http://www.classpath.org/.

Check it out. There is an ongoing open source project underway to create a cleanroom version of the java.* and sun.* packages. The GNU Classpath project has already decided to distribute their code under the LGPL license. They explain their reasons for signing over the copyright of their source code to the Free Software Foundation (FSF).

According to their website, the classpath project is needed by the open source community. Sun Microsystems has not opened all the source to rt.jar, and therefore, cannot be redistributed with open source software. Sun's licensing is complicated for anyone trying to redistribute Java virtual machine.

The classpath project is creating java.* packages as a replacement for the java.* packages from Sun. All of the source code for classpath will be created in a cleanroom environment, without looking at the source code from Sun.

The classpath project is creating gnu.java.* packages as a replacement for sun.* packages. These "internal" classes are needed to support java.* native methods.

At some point in the future, when the classpath project produces a cleanroom implementation of core packages, we should be able to redistribute them with JOS.
