SimpleJOS

Article contributed by GilbertHerschberger (28 July 1999).

Introduction

Programming in-the-large will never be the same. Java has changed all of the rules with a very large namespace. And yet, there are very important things about programming in-the-large that, well, Java has been a little slow to support.

You can write a large number of unique classes in Java. There is so much namespace, you'll never have to use the same absolute class name twice. With so many classes, it was important to find a way to keep classes organized. The Java package is the solution.

Packages

Packages are a natural container for multiple shared objects (or dynamic link libraries). All of the classes are created inside this container because they work well together.

One class at a time

When programming in-the-large, you can't afford to keep on thinking about one class at a time. You have to start thinking about packages at the package level.

  • The smallest thing you should compile is the class, right? The javac tool expects you to think about one class at a time. It is based on programming in-the-small. The ultimate javac tool should compile one package at a time. It should automatically detect all of the Java source code files in a package--without make--and compile the bytecode that needs to be compiled.
  • The smallest unit of distribution for Java is the package. There is no point in distributing one class file at a time. Where are the PackageFiles?
  • Packages are useful to other packages. This is where extreme reuse begins. You can use the code out of other packages. Based on a bottom up approach, you can use any bytecode that has already been compiled.

Package statement

The package statement in source code is not necesarily the package you want in your bytecode. The package source code should provide the mechanism to change packages without rewriting the source.

The source code for my packages start out with a package statement. It looks like this: 

<STRONG>package</STRONG> com.realignsoft.demo;

What happens when I want to ship the identical mechanism in bytecode with another package, say org.jos.demo?

Package source code

With a package source code file, I can describe the relationship between packages. One package can inherit the behavior of other packages.

<STRONG>public</STRONG> <STRONG>package</STRONG> org.jos.demo
	 <STRONG>extends</STRONG> com.realignsoft.demo {
}

Result

When I pass the name of the package source code (.pjava) file to the josc tool, it should compile the package, the whole package, using the given instructions.

By inheriting the behavior of another package, there's a bunch of code that I do not have to rewrite.

About JOS

I propose that we develop these packages with real source code:

  • java10.*
  • java11.*
  • java12.*
  • java13.*

and use package source code to compile this source code into the java.* packages.

How did Sun do it? Sun Microsystems developed the java. packages in the javax. namespace. Sun did not develop the java. packages in place. They used the .9 version of Java to compile version 1.0. The javac tool was able to compile the class inside the java.lang. package because it was really the java.lang. package; it was javax.lang..

As you know, I don't like the javax.* packages.
