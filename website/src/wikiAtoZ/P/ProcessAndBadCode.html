ProcessPages

Article contributed by GilbertHerschberger (10 April 2000).

Introduction

It's all about control. I want to control the lifetime of any program running on my machine. All programs should work the same way. The mechanism should be program-independent. When I want a program to end, I depend on my operating system to end it.

I like the process model. The process model is critical to maximizing the up-time of an operating system. A process model is useful for both good and bad code. I would like to extend the process model into the bytecode of a virtual machine.

Good code

An operating system should control a correct program. A correct program may start and require the intervention of the operating system to stop. A daemon is often designed this way.

An HTTP daemon, for example, should run until the operating system kills it. There is no need for a separate program-specific mechanism.

public class Main {
  public static void main( String[] args ) {
	 new HttpDaemon().run();
  }
}

Bad code

In order to understand bad code, we have to figure out where bad code comes from. We must separate the intention of a programmer from the reality of the programming art.

Most programmers write good code, and always intend to. Bad code comes from a programmer. Bad code is a by-product of inexperience. Most programmers are inexperienced. A novice programmer writes bad code because they do not know better. An intermediate programmer writes bad code and knows how to fix it. Even the brightest, most gifted programmer writes bad code occasionally.

Bad code is not equal to malicious code. A malicious programmer writes malicious code, and intends to.

I want an operating system to enable me to run good code. Nothing should prevent me from running my own code on my machine.

An operating system might protect me from bad code. It should help me identify an author of malicious code.

Endless program

The following program is valid. It will compile. It will run within a virtual machine. How do I control its lifetime?

package org.jos.nondemo;
public class EndlessForProgram {
  public static void main( String[] args ) {
	 for (;;) ;
  }
}

This is an equivalent program:

package org.jos.nondemo;
public class EndlessWhileProgram {
  public static void main( String[] args ) {
	 while (true) ;
  }
}

Under construction

An operating system should not crash just because one program is under construction. When a program is under construction, it isn't finished. It has holes. It behaves badly. It is filled with bad code.

My operating system runs programs in all phases of development, from experiemental to production systems. My operating system runs many programs that are still under construction. I want a new operating because the one I have crashes far too often. It takes precious time away from my programming when I have to reboot.
