Article contributed by GilbertHerschberger.

Alternative 1 The first design was the idea that there would be one hash table for packages and one hash table for classes in each package. Unfortunately, this closely associates a class with Package instead of PackageManager.

  • PackageManager extends ClassLoader
    • Hashtable<Package>
  • Package extends ClassLoader
    • Hashtable<Class>

The strength of this design comes from both search speed and garbage-collection. When a hash table contains classes for one and only one package, the search is efficient. When a package needs to be removed, the Package object is removed from the hash table of packages. It just goes away by itself.

Alternative 2 After some trouble with the first alternative, the second was born. The second design was the idea that there would be one hash table for all classes, and when a new package appeared, the package would be imported.

  • ClassManager extends ClassLoader
    • Hashtable<Class>
  • PackageLoader( String packagepath )

  • While a ClassManager creates Class objects, the PackageLoader does not. PackageLoader only deals with byte[].

  • Any requestor with a class name and bytecode can call ClassManager.addClass( String, byte[] ). If a class has already been added to a ClassManager, the request is ignored. No exception is thrown.

  • A requestor calls ClassManager.findClass(). When a Class has been added, a Class is returned from the Hashtable<Class>. When a ClassManager detects the use of a new package, it calls PackageLoader.loadPackage( this, packageName ).

  • Inside loadPackage(..), a PackageLoader calls ClassManager.addClass( String, byte[] ) for each class in a package.

  • A PackageLoader could be extended to load packages from new file formats or across the network.

  • customPackageLoaders are added to ClassManager with ClassManager.addLoader( PackageLoader ).

  • PackageLoader.loadPackage throws a PackageNotFoundException. If there are many PackageLoaders registered with ClassManager, each loader is called until a package is found or there are no more loaders.

If a Process uses a default ClassManager, there can be a different kind of ClassManager for each kind of Process. - GilbertHerschberger 14 APR 1999

  • jos.standard.StandardProcess
    • implements jos.Process
    • uses jos.standard.ClassManager

  • jos.compatible.CompatibleProcess
    • implements jos.Process, and
    • uses jos.compatible.ClassManager

  • jos.standard.ClassManager
    • extends java.lang.ClassLoader
    • reads one package at a time
    • reads from standard JOS package files

  • jos.compatible.ClassManager
    • extends java.lang.ClassLoader
    • reads one class at a time
    • reads from standard Java class files
    • reads from standard JAR files

Question: How can a any JOS ClassManager remain compatible with java.lang.ClassLoader when compatible ClassLoaders will depend on ClassManager? Classes derived from java.lang.ClassLoader expect findSystemClass() method to find a system class. Finding a system class is performed by ClassManager.

In order to reduce waste, there must be a system class manager to manage all of the system classes across all processes. A reference for a Class in the bytecode interpreter is going to be external to a Process. Therefore, the ClassLoader associated with a system class is going to be external to a Process. When an object is asked to load another class with its ClassLoader, system classes will use their own SystemClassLoader.

What will custom classes use? Custom classes will use a custom class loader, derived from ClassLoader. The base ClassLoader must be able to implement the findSystemClass() method.

Surely, a Class itself should not be garbage-collected by the Process. In a standard JVM from Sun, all Classes are garbage-collected when a process is killed. Inside JOS, Class cannot be garbage-collected in the usualy way. When garbage-collected, the entire package must be garbage-collected together, like removePackage(). The package cannot be garbage-collected until all objects depending on the class (in all Processes) are garbage collected.
