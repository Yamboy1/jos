Article contributed by GilbertHerschberger (09 November 1999, updated 17 August 2003).

Introduction

Your invited to think more about package oriented tools. Nothing smaller than a package is a concern of package oriented tools. A package is stored in a package file. A package copier is a tool to copy the source code for a whole package. A package compiler is a tool to compile a whole package. A package archive is a tool to archive a whole package.

Size of Unit

When you find an appropriate size for a data processing unit, it adds greatest efficiency. Select something too small and you waste your time keeping up with the overhead of the development process. Select a unit too large and you waste your time processing lots of data that does not really need to be processed.

A package is the smallest unit of distribution for Java bytecode. Could it be the smallest possible unit for copying, compilation and archiving? If it is, it puts an efficiency on programming in-the-large. You can forget about individual classes and concentrate on bytecode at a higher conceptual level: the package.

Process

When you have your original source code in a directory, you can make a copy of it quickly with the package copier.
  • The package copier makes a separate copy of the source code for the express purpose of distributing it. In addition, the copy of the source is used for JavaDoc, SourceDoc and compiling it for a binary distribution. The smallest unit of copying is the package.
  • The package compiler only compiles a package when class files are out-of-sync with their source files. The package compiler uses the source code created by the package copier. The smallest unit of compilation is the package.
  • The package archive tool builds a jar file from the class files created by the package compiler. The smallest unit of archive is the package.
For example, I have the original code for many different projects on my D: drive. I use Windows 95 as my operating system on the client side. I use Linux exclusively as my operating systems on the server side.

When it comes time to distribute the source code, I cannot distributed everything on my D: drive. I must selectively distribute packages for each product.

A package copier is a tool to copy the source code for a list of packages. The source code is copied to an src directory. I can control how much source code is copied by building a package copier script. The script for "Smart API - Release 1a" looks something like this: 

#gchii.packagecopier1a.dev.BatchCompiler
PROGRAM gchii.packagecopier1a.PackageCopierProgram
  path=D:&package=com.realignsoft.browserstudio1a
  path=D:&package=com.realignsoft.iniregistry1a
  path=D:&package=com.realignsoft.registry1a
  path=D:&package=com.realignsoft.registry1b
  path=D:&package=com.realignsoft.smartapi1a
  path=D:&package=com.realignsoft.streamfactory1a
  path=D:&package=com.realignsoft.stringregistry1a
  path=D:&package=com.realignsoft.stringregistry1b
  path=D:&package=com.realignsoft.textregistry1a
  path=D:&package=com.realignsoft.util1a
ENDPROGRAM

When original source code is distributed inside an archive, such as src.zip, the package copier uses the a slightly different path to accomplish the same thing: 

#gchii.packagecopier1a.dev.BatchCompiler
PROGRAM gchii.packagecopier1a.PackageCopierProgram
  path=[src.zip]&package=com.realignsoft.browserstudio1a
  path=[src.zip]&package=com.realignsoft.iniregistry1a
  path=[src.zip]&package=com.realignsoft.registry1a
  path=[src.zip]&package=com.realignsoft.registry1b
  path=[src.zip]&package=com.realignsoft.smartapi1a
  path=[src.zip]&package=com.realignsoft.streamfactory1a
  path=[src.zip]&package=com.realignsoft.stringregistry1a
  path=[src.zip]&package=com.realignsoft.stringregistry1b
  path=[src.zip]&package=com.realignsoft.textregistry1a
  path=[src.zip]&package=com.realignsoft.util1a
ENDPROGRAM

Package compiler

The package compiler uses the javac tool to compile source code. Only the command line is created dynamically based on the actual source code found inside a directory (package). The name of every source code file is passed on one long command line.

Package archive

I am working on the package archive tool. It shouldn't take much to create an archive, using the jar tool.

The package archive will run the jar tool with a wild card, something like this:</P> 

jar cvf smartapi1a-1.1.jar srccomrealignsoftsmartapi1a*.class

Everything is automatic because the class files are created by the package compiler. There are no "stray" class files.

Clean

The clean step of the bulk make process removes all source and class files from the temporary bulk-make directory. Because the package copier uses the Smart API to create output files, directories are created automatically, as needed. When the project is cleaned, the temporary source files and class files are deleted.

Problems

There are a couple of problems that I am working out.
  • The package compiler must be able to use any compiler. At the moment, it is hard-coded to use my favorite compiler. The package compiler already has a configuration file; but, the mechanism to get the compiler configuration to the package compiler is undefined.
  • The package compiler runs out of memory (?) when running a Java 1 Platform compiler from the Java 0 virtual machine. In theory, the Java 0 virtual machine can run any native OS process. I have run into other processes that will not run from the Java 0 virtual machine.
  • The package copier and package compiler should always use the temporary bulk-make directory.
  • The entire process should be scripted from creating the bulk-make directory, copying the source code, compiling and creating archives.
