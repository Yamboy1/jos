Article contributed by GilbertHerschberger (13 March 2000).

Introduction

About servlets

A servlet is a special kind of Java program. Servlets run inside a servlet environment. Everyone expects a servlet environment to use a custom class loader to load a servlet. By using a custom class loader, a servlet can be re-loaded without restarting your servlet environment.

An HTTP service is one way to provide a servlet environment for your servlets. A servlet can output to a popular web browser through the HTTP protocol.

About JOS

A process is a special kind of Java program. Processes run inside a process environment. Everyone expects a process environment to use a custom class loader to load a process. By using a custom class loader, a process can be re-loaded without restarting your process environment.

JOS is an operating system. An operating system provides three basic functions:

  • Start a process

    Create, launch, open, start, run, invoke are other names for starting a process.

  • Stop a process

    Stop, close, end and shut down are other names for stopping a process politely.

  • Kill a process

    Kill is reserved for stopping a process impolitely.

Processes run within a process environment, not an operating system. In other words, an operating system provides a process environment like a HTTP service provides a servlet environment. A process environment and an operating system are closely related, but very different.

With greatest precision, only a process environment can create a new process. If it exists, a process environment must provide a mechanism for one process to create another.

  • An operating system can create and destroy a process environment. Strictly speaking, an operating system does not provide a low-level mechanism to create/destroy another process; it uses a process environment.

  • Strictly speaking, a shell process does not provide a low-mechanism to create/destroy another process; it uses a process environment.

  • Strictly speaking, a desktop does not provide a low-mechanism to create/destroy another process; it uses a process environment.

  • Strictly speaking, a browser does not provide a low-level mechanism to create/destroy another process; it uses the process environment.

This distinction is clearly demonstrated in any operating system that does not support processes. Any operating system that does not support processes excludes a process environment. Look at embedded devices and controllers, Timex/Sinclair 1000, TRS-80, Commodore 64/PET, Gameboy and Atari. The Java Virtual Machine is just another example of an operating system that does not (yet) support processes. It has no process environment.

Justification

A servlet environment uses one mechanism as the primary mechanism to isolates servlets: a custom class loader. All of the servlets in a servlet environment share the following components:

  • security manager

    An HTTP service is expected to install its own security manager before loading the servlet environment (especially before loading its first servlet. This prevents a servlet from shutting down a virtual machine with exit().

  • standard in, standard out, standard error

    All servlets use the static fields of the java.lang.System class

  • java.net.URL

    All servlets share one URLConnectionFactory. A servlet cannot implements a custom URL scheme without the help of the HTTP service.

  • java.awt.Toolkit

    Servlet are not expected to make use of the AWT (or Swing). But, nothing prevents a servlet from using the AWT (or Swing). It is part of the Java standard class libraries.

    Again, a HTTP service is one place for a servlet environment. With a client running on one machine and an HTTP service running on another, window are displayed on the server machine when a servlet uses the AWT (or Swing).

About Universal Browser

An HTTP service is not the only program capable of providing a servlet environment. Soon, the UniversalBrowser will include a servlet browser plug-in. A servlet browser will provide a servlet environment. Servlets will run in this servlet environment. A servlet browser will be automatically selected when the servlet: scheme is used. The following URI invokes the org.jos.demo.DemoServlet servlet:

servlet:org.jos.demo.DemoServlet

With this configuration, a servlet browser and servlet environment run in the same process. When you close a servlet browser, all servlets are unloaded and the servlet environment is destroyed. No interprocess communication is needed. Notice how "servlets" run on the client-side.

About threads

A servlet is invoked for each request. A servlet does not run in its own thread. Rather, the servlet environment invokes a servlet by some custom mechanism. The mechanism is determined by the servlet environment.

Summary

Understand servlets. They will help us define and refine the process model for JOS.

  • A servlet is isolated from all other servlets. Can we use a servlet-like mechanism to isolate processes?

  • A servlet can be loaded without starting up a new virtual machine. Can we use a servlet-like mechanism to start a process without starting up a new virtual machine?

  • A servlet can be re-loaded without shutting down a virtual machine. Can we use a servlet-like mechanism to reload a process without shutting down a virtual machine.
