Introduction

According to the JavaNativeInterface (JNI), more than one instance of a virtual machine can a created within a native process. With more than one instance, JNI offers instance management. A program can create a new instance. A program can enumerate existing instances.

JNI rules

The following rules are a little different than the classic design.

  • Each virtual machine has a separate environment.

    A virtual machine might be optimized to run multiple instances within a single process. Otherwise, each virtual machine must follow the classic design. Each virtual machine appears to a Java application just like a virtual machine running in a separate process. Each Java application requires a separate virtual machine.

    Each environment is independent from all others, even within the same native process. Each virtual machine appears to have its own CLASSPATH, its own primordial class loader, its own JIT compiler. Each virtual machine appears to have its own garbage collector thread, its own object heap, its own threads and thread groups.

    Each virtual machine has its own instance of java.lang.Runtime, java.net.URL, java.awt.Toolkit and other classes with static fields.

  • Each virtual machine runs in a separate thread.

    The JNI design assumes a native thread is used to run each virtual machine. It is as if a virtual machine is running in a separate process, without the overhead.

  • Upon Runtime.exit(), a virtual machine dies.

    Like a classic design, the JNI design supports the full semantics of Runtime.exit(). All objects created by a virtual machine are garbage collected when the virtual machine dies. When a virtual machine dies, its native thread dies.

  • Loading libraries

    A shared library loaded by one virtual machine appears to be unavailable to other virtual machines. A shared library written for a classic design probably won't run in the JNI design. Shared libraries must be rewritten to manage instance variables, one instance per virtual machine.

Process-wide

Process-wide variables can be shared among different instances of a virtual machine. It is easy to intern() class files. It is easy to offer an object-based inter-process communication.

JNI support

The JNI supports multiple virtual machine per native process because of the demand for this design on the high-end. A business server can be optimized for improved performance by providing a virtual machine service. One process is responsible for running many Java applications. It is optimized by pre-compiling a list of packages. From the viewpoint of a java tool, loading the java packagestakes no time at all.

On the low-end, many virtual machines lack full support for the JNI. Once a virtual machine has been written to the classic design, it must be ported to JNI. Porting to the JNI is not a trivial project.

Testing

A shared library is identified by its unique module name. A kernel can be forced to load two or more instances of a shared library by re-linking with different module names. Each instance of a shared library must be written to its own executable image.

This technique is useful for regression testing. One version of a virtual machine is tested against another within a single process.

About exec()

One Java application should be able to invoke another. One virtual machine could create another with the same process.

How can a program do that?

A custom class would have to define a native method that calls the JNI to create a new virtual machine. The JNI is incomplete. It does not provide an API for one virtual machine to invoke another. Even inside the JNI design, Runtime.exec() method creates a brand new process.
