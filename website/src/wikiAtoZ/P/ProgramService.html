Starting the Program Service

The following procedures illustrate how to run the Program Service. So far, it has been tested exclusively on Windows(tm) 95. As a pure bytecode application, it should also run--unchanged--on Linux(r).

  1. To start the Program Service using the UniversalBrowser, type in the following location:

    run:program:gchii.tools.programservice1a.ProgramService

  2. To start the Program Service with the eric tool, double-click on the programservice.eric file. It is a shortcut to the program service utility. Here is its contents.

    location=run:program:gchii.tools.programservice1a.ProgramService

  3. To start the Program Service with the java tool, run something like the following command:

    java gchii.tools.programservice1a.Main

Program Service utility

The source code to this utility will be available soon.

A separate virtual machine is optional.By default, all subprograms run inside a program service inside a single virtual machine. Each connection to the program service runs in a separate thread. The number of simultaneous connections is determined by the count property. The program service is capable of launching another virtual machine and another program service.

A separate class loader is optional. By default, there is only a single class loader for the program service. Once a subprogram is loaded this way it cannot be unloaded.

Subprograms can communicate directly. Static fields can be used to pass objects of any size from one subprogram to another. A chat program, for example, could easily pass messages from one connection to another.

Each subprogram is given a subprogram environment. The subprogram environment provides its parameters, result code, standard in and standard out. The subprogram environment is encapsulated in a ProgramContext object. Subprograms are capable of redirection, piping and such within a single virtual machine. In the program service environment, all programs share one instance of system properties.

The main-style subprogram API is optional. Any one of dozens of object-oriented subprogram APIs are available, including the Applet API.

Note: The above examples do not include an example of the main-style subprogram API, such as

$ run:main:com.ice.jcvs.CVSApplication
$ run:main:sun.servlet.http.HttpServer
$ run:main:sun.tools.javac.Main
$ run:main:sun.tools.javadoc.Main
