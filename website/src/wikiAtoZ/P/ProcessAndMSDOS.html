ProcessPages

Article contributed by GilbertHerschberger (20 June 2000).

Introduction

Process is a concept even within a non-preemptive operating system like MS-DOS. Each executable image is loaded by the MS-DOS kernel and bound to the kernel. Each process has a program segment prefix. It also has the concept of terminate-and-stay-resident programs.

Benefits

The simplicity of MS-DOS is an important factor in its success. It proves that it is possible to build a successful non-preemptive operating system for personal computers. MS-DOS is not dead. It is still being installed and used on personal computers today.

Memory

Sometimes, the non-preemptive version of JOS reminds me about MS-DOS. There are many similarities. A non-preemptive version of JOS could follow the same kind of program segment prefix (PSP) design of MS-DOS. With each PSP, there is a linked list of memory owned by a program.

public class MemoryBlock {
  public MemoryBlock( MemoryDescriptor v ) {
	 m = v;
  }
  public MemoryBlock getNext() {
	 return next;
  }
  public void setNext( MemoryBlock v ) {
	 next = v;
  }
  private MemoryDescriptor m;
  private MemoryBlock next;
}

When a program (or bytecode process) is closed, all of the memory allocated by a program is automatically freed by following the linked list that starts with the PSP.

Files

When a file is opened, it is opened by a program (or bytecode process). A PSP also contains a list of open files. When a program opens a file, the MS-DOS API adds a file handle to the open file list. When a program invokes the abort() method, all open files are closed without saving changes.

public class FileBlock {
  public FileBlock( FileDescriptor v ) {
	 fd = v;
  }
  public FileBlock getNext() {
	 return next;
  }
  public void setNext( FileBlock v ) {
	 next = v;
  }
  private FileDescriptor fd;
  private FileBlock next;
}

Proposal

The program segment prefix (PSP) for a bytecode process might look something like this:

public class ProgramSegmentPrefix {
  public MemoryBlock getMemoryBlock() {
	 return pmemory;
  }
  public FileBlock getFileBlock() {
	 return pfile;
  }
  private MemoryBlock pmemory;
  private FileBlock pfile;
}

Polymorphism

Every system resource should be treated the same way. Since a memory block, shared memory, file and socket are all system resources, they can be added to a single shared resource list.

public interface SharedResource {
  public abstract SharedResource getNext();
  public abstract void setNext( SharedResource v );
  public abstract void abort();
}

A program segment prefix (PSP) would have a single pointer to the top of the shared resource list, like this:

public class ProgramSegmentPrefix {
  public SharedResource getSharedResource() {
	 return ptop;
  }
  private SharedResource ptop;
}

A memory block, file block and other system resources would implement a well-known shared resource interface.

public class MemoryBlock
	 <STRONG>implements SharedResource</STRONG> {
  public MemoryBlock( MemoryDescriptor v ) {
	 m = v;
  }
  public SharedResource getNext() {
	 return next;
  }
  public void setNext( SharedResource v ) {
	 next = v;
  }
  private MemoryDescriptor m;
  private SharedResource next;
}

public class FileBlock
	 <STRONG>implements SharedResource</STRONG> {
  public FileBlock( FileDescriptor v ) {
	 fd = v;
  }
  public SharedResource getNext() {
	 return next;
  }
  public void setNext( SharedResource v ) {
	 next = v;
  }
  private FileDescriptor fd;
  private SharedResource next;
}

Therefore, both the memory block and open files are added to a single system resource list. By polymorphism, all system resources can be abort'd with the following code:

  public void example() {
	 SharedResource r = psp.getSharedResource();
	 for (;;) {
		if ( r == null ) {
		  break;
		}

		try {
		  r.abort();
		}
		catch( Throwable e ) {
		}

		r = r.getNext();
	 }
  }
