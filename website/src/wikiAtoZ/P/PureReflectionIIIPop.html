BytecodePages; PureReflectionIII

Article contributed by GilbertHerschberger (21 August 2000).

Introduction

The stack passes a parameter from a thread to instruction inside an virtual machine.

About pop

Each thread has its own last-in, first-out (LIFO) stack. In the Code attribute of a method, the pop instruction removes an object from the stack. The internal state of the stack is changed as a side-effect.

  • pop

    A four-octet value is removed with this instruction. This includes these types:

    • boolean,
    • char,
    • short,
    • int,
    • float and
    • object-reference.

    Note: Or, a four-octet value might be one half of a long or double type.

  • pop2

    An eight-octet value is removed with this instruction. It is equivalent to two consecutive pops. This includes the following types:

    • long and
    • double.

Instruction

An instruction is a collection, with an opcode, operand(s) and parameter(s).

  • An opcode is part of an instruction. It is first. It is constant. An opcode is a one or two octet value.

  • An operand is part of an instruction. It is optional. It is constant. When used, it follows the opcode in the Code attribute.

  • A parameter is a part of an instruction. It is variable. A parameter is passed to an opcode method through a thread's stack. Like operands, an instruction has zero or more parameters. Each parameter has an expected types. When given a list of expected parameters, a function can map parameters to a parameter list.

Parameter type

Each opcode expects specific parameter type(s). The list of expected types is a property of an opcode. By extending the Opcode class, the parameter list can be correctly attached to each opcode. Each parameter in a list is one character and needs no separator.

  • D

    When an opcode expects an eight-octet double, this parameter type is used. This includes the double primative type.

  • F

    When an opcode expects a four-octet float, this parameter type is used. This includes the float primative type.

  • I

    When an opcode expects a four-octet integer, this parameter type is used. This includes the boolean, char, short and int primative typess.

  • J

    When an opcode expects an eight-octet long, this parameter type is used. This includes the long primative type.

  • O

    When an opcode expects a four-octet object reference, this parameter type is used. This is a reference to an instance of java.lang.Object.

Return type

Each opcode has a return type. All return types are a single character. Return types are encoded like parameter types.

Example 2

iadd is a simple instruction. It adds two integers at the top of the stack. It pop's two integers from the stack and push's an integer. Its parameter list is II. Its entry in the opcode table looks like this:

  IADD = new Opcode( 96, "iadd", 1, "I", "II" );

The new opcode class has a new constructor, with return type (r) and parameter list (p), something like this:

private Opcode( int i, String n, int s, String r, String p );

A prototype of the corresponding opcode method in C would look something like this:

int iadd( int p1, int p2 );

By storing the expected parameter types with each opcode, it is possible to perform minimalist bytecode validation.
