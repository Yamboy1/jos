ProcessPages; PrimordialClassLoader

Article contributed by GilbertHerschberger (11 April 2000).

Introduction

This article provides a few hints on the process design. It is currently under discussion on the architecture mailing list. We have recently clarified the relationship between a process and its primordial class loader. [1] In bytecode, a process model depends upon a primordial class loader. A primordial class loader is a critical component of a bytecode process.

  • When it supports only one primordial class loader, a virtual machine can support only one process.

  • When it supports more than one primordial class loader, a virtual machine can support more than one process.

Classic design

The classic design requires each Java Virtual Machine to have exactly one primordial class loader. This relationship between virtual machine and primordial class loader limits a classic virtual machine to support exactly one bytecode process.

Proposed MPCL design

The MPCL design enables a virtual machine to have more than one primordial class loader. This relationship enables a virtual machine to support more than one bytecode process.

Compare to classic design

The classic design is simple. It works well on a pre-existing model for machine code processes. The classic design makes these assumptions:

  • A machine code process is equal to a virtual machine;

  • A virtual machine is equal to a primordial class loader;

  • A primordial class loader is equal to a bytecode process;

  • Therefore, a machine code process is equal to a bytecode process.

Unlike the classic design, the MPCL design enables a single virtual machine to run many independent bytecode processes. A virtual machine can be optimized to start, stop and restart a process.

For more information on the classic design, see also ProcessDiagramClassic.

Compare to JNI design

The JNI design introduces the infinite complexity of programming around native shared libraries. Shared libraries must be completely rewritten, moving from process-wide to vm-wide global variables. The JNI design makes these assumptions:

  • A machine code process has one or more virtual machines;

  • A virtual machine is equal to a primordial class loader;

  • A primordial class loader is equal to a bytecode process;

  • Therefore, a machine code process has one or more bytecode processes, each with its own virtual machine.

Unlike the JNI design, the MPCL design does not require separate virtual machines for each bytecode process. A virtual machine can optimize inter-process communication. A virtual machine might provide a mechanism to share objects among two or more processes.

The system process (init) may be the only process with permission to plug in a kernel module. A kernel module is a shared machine code library. Unlike the JNI design, a kernel module is global. Its static variables are system-wide. When one process plugs in a kernel module, the kernel module is available across all bytecode processes.

For more information on the JNI design, see also ProcessDiagramJNI.

Summary

The classic design might not have gone far enough. The JNI design might have gone too far. The MPCL design is just right.

Ultimately, the virtual machine is the only machine code process running on a JOS kernel. The virtual machine supports multiple concurrent bytecode processes by launching each process with its own primordial class loader.
