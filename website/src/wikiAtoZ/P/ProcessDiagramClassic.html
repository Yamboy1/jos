Introduction

Before we can improve the classic design, we must understand it. This article provides more background on the classic design of a Java process. The classic description of a virtual machine is found in the Java Virtual Machine Specification. It describes a process. A process is equal to a native process on a foreign operating system.

Classic rules

The classic design follows certain rules. While these rules can be broken, it requires great effort to make it work.

  • The native process is the only process.

    The classic design assumes a foreign operating system. It assumes a foreign operating system supports heavy-weight process. It assumes that all processes are native. The java.lang.Process interface is a proxy for a native process. The exec() method of java.lang.Runtime assumes that a command and environment is required by a foreign operating system to create a native process.

  • Each process has standard in, out and err.

    The classic design assumes a process is a native process with three I/O streams: a standard input stream, a standard output stream and a standard error stream. This is reflected in the design of java.lang.Process.

  • Each process uses standard in, out and err for inter-process communication.

    The classic design assumes a parent process communicates with a child process. A child process might read more than its command line arguments. The parent creates a child process and writes to its standard in, reads from its standard out and standard err.

    Three I/O streams are used to cross the persistence barrier. Within one process, an object is marshalled into its raw data and written to the other through an I/O stream. Within the other process, raw data is read from standard in and unmarshalled into a new object.

  • Each process returns a result code.

    The classic design assumes the entire result of a process can be expressed as a result code. A result code is a single octet. By convention, a result code is interpreted. 0x00 means success and everything else means failure.

  • The native process has zero or one virtual machine.

    From the ground up, a classic design assumes that there will only be one virtual machine running in a native process. It takes advantage of this assumption. It makes extensive use of global variables. Global variables are attached to a process, not thread.

    Dynamic loading of a shared machine code library assumes a single process. Once linked, a machine code library is available to THE virtual machine.

  • Each Java application must run in a separate process.

    A classic design implies that each Java application has total control over a virtual machine. There is only one instance of java.lang.Runtime and other critical objects. Each Java application must run in a separate virtual machine. Each virtual machine must run in a separate process. Therefore, each Java application must run in a separate process.

  • A virtual machine is a shared library.

    A classic design implies that a virtual machine must be implemented as a shared library. As a shared library, it can be embedded in all kinds of native programs. A web browser, for example, can use it when a virtual machine is implemented as a shared library.

Optimization

Machine code is read-only. Because it is read-only, it cannot be written to a swap file. Machine code has already been optimized in a foreign operating system, such as Linux or Windows. A native operating system provides the following optimization for shared libraries.

Only one copy of a shared library is mapped to real memory.

Machine code in shared libraries is system-wide. Machine code is "mapped" into virtual memory from its original shared library image on disk. Executable code is special because it is read-only. It must be loaded into real memory just before invocation. There is no need for a shared library to be fully loaded into real memory. Small chunks of the shared library are read as needed by a machine code processor. Machine code might have the lowest priority in a virtual memory manager. Machine code is the easiest kind of memory to discard -- it can always be read again.

If there are 100 processes running and each process "loads" a virtual machine, there is only one copy of the virtual machine's code.

"Global" variables are process-wide. If there are 100 processes, there are 100 copies of "global" variables. Except for read-only machine code, each virtual machine is independent of the others.

Shared class files

The operating system loads only one copy of a shared library into real memory. If class files are "linked" to a shared library, and marked as native executable code, the operating system provides one copy of a class file. The core classes, for example, can be statically linked to a shared library. An instance of java.lang.Class can be created from the system-wide class file, saving real memory.

System-wide variables

With extraordinary effort, system-wide variables can be shared among all instances of a shared library. Using a classic design, a virtual machine can intern() classfiles in a system-wide cache. Any class file used by any instance is shared, saving additional memory.

With the ability to share system-wide variables, shared libraries can provide object-based inter-process communication at the machine code level. One process writes data to a shared object and another reads it.

When it is implemented as a shared library, a virtual machine might provide system-wide bytecode cache. A virtual machine might extend object-based inter-process communication to the Java level. One process writes data to a shared Java object and another reads it.
