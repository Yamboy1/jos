BytecodePages

Article contributed by GilbertHerschberger (25 May 2000).

Introduction

It might be easier to understand the class file format from the viewpoint of anchors. An anchor is a well-known place. After working with pure reflection for a while, there are certain places in a class file that can be used as anchors to reference other things.

Start of file

The first anchor point is always the start of a class file. Other anchors are codepool, fields, methods and class attributes. A class file looks something like this from a viewpoint of anchors:

class file
magic/version
  • codepool
inheritence
  • fields
  • methods
  • class attributes

Codepool

If the first octet in the size of codepool is used as an anchor, the entire codepool can be interpreted/translated by a codepool tool (in C++).

class CodepoolTool {
  public:
	 virtual void setPayload( const void *v );
	 virtual unsigned short getCodepoolCount();
	 virtual const void *getCodepool( unsigned short index );
	 virtual const void *getAfter();
};

Attribute

If the first octet in attribute count is used as an anchor, the entire attribute structure can be interpreted/translated by a attribute tool (in C++).

class AttributeTool {
  public:
	 virtual void setPayload( const void *v );
	 virtual unsigned short getAttributeCount();
	 virtual const void *getAttribute( unsigned short index );
	 virtual const void *getAfter();
};

The attribute tool can be reused. There are class attributes, field attributes and method attributes with the same structure.

Fields

If the first octet in the count of fields is used as an anchor, the entire field structure can be interpreted/translated by a field tool (in C++).

class FieldTool {
  public:
	 virtual void setPayload( const void *v );
	 virtual unsigned short getFieldCount();
	 virtual const void *getField( unsigned short index );
	 virtual const void *getAfter();
};

Methods

If the first octet in the count of methods is used as an anchor, the entire method structure can be interpreted/translated by a method tool (in C++).

class MethodTool {
  public:
	 virtual void setPayload( const void *v );
	 virtual unsigned short getMethodCount();
	 virtual const void *getMethod( unsigned short index );
	 virtual const void *getAfter();
};

Basically

The After property of a tool returns a pointer to the octet just after then end of its structure.

  • To find the codepool anchor, add eight (8) to the start of a class file.

  • To find the inheritence structure, use codepoolTool.getAfter().

  • To find the field structure, add 2 for thisClass, 2 for superClass, 2 for the interface count. Then add 2 for each interface.

  • To find the method structure, use fieldTool.getAfter().

  • To find the class attribute structure, use methodTool.getAfter().

An implementation of rc_Bytecode could be implemented something like this (in C++):

public class BasicBytecode : public rc_Bytecode {
  public:
	 BasicBytecode( const void *v );
	 :

  protected:

  private:
	 CodepoolTool cpTool;
	 FieldTool fieldTool;
	 MethodTool methodTool;
	 AttributeTool attributeTool;

};

BasicBytecode::BasicBytecode( const void *v ) {
  const unsigned char *p = (const unsigned char *) v;
  p += 4; // magic
  p += 4; // version
  cpTool.setPayload( p );
  p = (const unsigned char *) cpTool.getAfter();
  p += 2; // thisClass
  p += 2; // superClass
  unsigned short c = getUnsignedShort( p );
  p += 2; // interface count
  p += c * 2; // interface list
  fieldTool.setPayload( p );
  methodTool.setPayload( fieldTool.getAfter() );
  attributeTool.setPayload( methodTool.getAfter() );
}

Summary

Bytecode can be interpreted directly, and should be. Bytecode is stored in a byte array. A byte array class in C++ should "throw" an exception when an array index is out of bounds. Bytecode does not have to be interpreted by a virtual machine as if it were an input stream.
