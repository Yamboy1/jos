Introduction

Many people have worked on a computer. Within the last five decades, this work has become so widespread that we often say program when we mean computer program. We often say programming when we mean computer programming. And, we often say programmer when we mean computer programmer.

The word program has been around for thousands of years. A program has always been a part of accounting, theater and government. A program is always a product of work called programming. A person who does the work is called a programmer. These words have been adapted to the computer. This article refines these terms for the JOS Project.

Program

In general, a computer program is something that is intended to run on a computer. Without a computer program, a computer is useless. A program is a container for instructions. In exhausting detail, it explains to a computer what we want it to do.

Size alone might be the most difficult part of this concept. The size of a program is almost limitless. A program is still a program even if it has only one instruction. There is no upper limit to program size. A program may contain a billion or more instructions.

We speak of a program as if it works or not. We like programs that work; we hate those that don't. A program is still a program even if it doesn't work. Like the conductor of an orchestra, a program works in concert with a processor to get its work done.

Subprogram

A program may be designed to work with other programs. A program and all its subprograms work together. In general, a computer subprogram is something that is intended to run on a computer and invoked by other computer programs. The following diagram illustrates four programs, a main program and three subprograms.

main program --> subprogram 1
--> subprogram 2
--> subprogram 3

Do all programs work together?

No, conflicts are likely. One program can interfere with the proper operation of another. Conflicts can cause unexpected behavior in both programs. When two or more programs share the same subprogram, for example, preferences of one might be changed by the other. It is not a good idea to run two conflicting programs at the same time.

With so many programs, we can assume two programs won't work together. When we want to run both at the same time, we need a way to isolate them. We need a good way to keep them apart. We need a barrier between them. Such a barrier is found in a process model.

Process

A process is a special container for programs that work together. It is an isolation chamber. It protects conflicting programs from each other. The following diagram illustrates a typical program running inside a process with its subprograms.

subprogram 1 --> sub-subprogram 1
--> sub-subprogram 2
--> sub-subprogram n

Issues

The following issues may help define what a program is.

  • A program is invoke-able.

    A program must have at least one entry point. An entry point is the location of a program's first instruction. From a structured approach, a program should have only one entry point. From an object-oriented approach, a program is an object with both code and data.

  • A program is separate from the mechanism to load and run it.

    A program always needs help to run on a computer. The computer needs explicit instructions telling it how to load a program's executable image and how to find a program's entry point. Instructions for loading a program are always found in another program. This recursion ends in hardware. A program, such as PC BIOS, can be stored in ROM.

  • A program is not a process.

    A program runs inside a process. A process invokes a main program. In turn, a main program can invoke subprograms. In turn, subprograms can invoke sub-subprograms and so on.

  • A program can share its objects.

    A program and its subprogram can share objects. They share all system classes and their static fields. They share one standard in, standard out and standard error. They share one AWT toolkit. They share one URL connection factory. By default, they share objects without inter-process communication (IPC).

  • Subprograms may or may not run in their own thread.

    A thread is optional. It depends on the needs of a program. If it intends to wait for a subprogram to finish, a program should not create a separate thread. Otherwise, a program should run a subprogram in a separate thread.

  • Any method can invoke a program.

    When a program invokes another program, the invoking program is called a program and the invoked program called a subprogram. In parallel terminology, the invoking program is called a parent program and the invoked program called a child program.

  • A program is not a library.

    A program uses a library. A library has many methods. Each method has an entry point. When it is loaded, a library may be initialized but not invoked. A library waits for a program to invoke one of its methods.

Conserving resources

Conserving resources is an issue of software quality. High quality programs conserve the right resources. Time is a limited resource. A high quality program might waste other resources for the sake of time. Space is a limite resource. A high quality program might waste other resources for the sake of space.

  • Conserving threads

    A thread should be used only when a thread is needed. It might be wasteful to invoke a subprogram in a separate thread and then do nothing but wait.

  • Conserving processes

    A process should be used only when a process is needed. A process should use inter-process communication for loosely-coupled programs. It might be wasteful to invoke a subprogram in an exclusive process and then share all objects through inter-process communication.

Multiple process

Any method can invoke a program in another process.

main program 1 -->
process 2
main program 2 --> subprogram 1
--> subprogram 2
--> subprogram n

Summary

As a programmer, you may be uncertain how instructions should be organized. There are so many options to choose from. A program is a popular option because it has a well-known entry point.
