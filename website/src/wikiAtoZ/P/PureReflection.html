Article contributed by GilbertHerschberger (17 September 1999).

Introduction

There is a difference between practical and pure reflection. The JOS Project has a need for pure reflection as the first step to build a comprehensive bytecode verifier. It needs pure reflection in order to verify the bytecode emitted by its javac tool.

The org.jos.reflection package will be a pure reflection API. It provides a way to statically analyse any Java class from any Java platform.

  • While using a Java 2 Platform, pure reflection is able to analyse classes from the Java 0 and 1 Platforms, including classes in their java.* packages.

  • While using a Java 0 Platform, pure reflection is able to analyse classes from the Java 1 and 2 Platforms, including classes in their java.* packages.

The java.lang.reflect package is a practical reflection API. It provides a way to use classes that were unknown at compile-time. Practical reflection is limited to reflect classes compatible with a specific platform.

Bytecode

reflection.Bytecode is an interface. It defines methods to get properties from bytecode.

reflection.BasicBytecode is a basic implementation of the Bytecode interface. It implements methods to get properties from bytecode.

ClassFile

For bytecode stored in a class file, the reflection.ClassFile reads a class file the easy way. This class extends reflection.BasicBytecode, and therefore, implements the reflection.Bytecode interface.

  public void example() {
	 String cf = "/java1_0/java/lang/Object";
	 Bytecode bc = new ClassFile( cf );
	 System.out.println( bc.getThisClass() );
	 System.out.println( bc.getSuperClass() );
  }

Malformed bytecode

The pure reflection class has a general purpose exception called MalformedBytecodeException. This exception is used when bytecode cannot be analysed further. For example, bytecode without a known version cannot be analysed and the MalformedBytecodeException exception is thrown.

Byte array

The class loader treats all bytecode as a byte array. This is a good approach for a pure reflection API because it builds a model that does not depend on the java.io package. It builds a model that can be implemented in C/C++.

Other projects use a stream-based approach to bytecode. So far, the reflection project has not used a stream-based approach. Instead of using a java.io.DataInputStream to read a class file, this project uses a byte array.

This is one of the first steps to build a comprehensive bytecode verifier.

  1. Decode magic number.
  2. Decode major and minor version.
  3. Decode constant pool.
  4. Decode class access.
  5. Decode class name.
  6. Decode superclass name.
  7. Decode interface table.
  8. Decode field table.
  9. Decode method table.
  10. Decode attribute table.

    Before a class is defined

    There are many places where information about a class needs to be known before the class is defined. When using dynamic class loading, a mechanism usually has a class name or bytecode. It needs to know if a class implements a well-known interface before it gets an instance of java.lang.Class.

    A pure reflection API would be helpful to various applications, including

    • large bytecode respositories,
    • agent environments for distributed computing,
    • device drivers in JOSBox,
    • static analysis of an API or framework,
    • custom controllers in SmartAPI.

    Contrast

    Pure reflection can reflect without invocation. It reflects a class without creating a java.lang.Class object.

    On the other hand, practical reflection works when all classes are written properly because it cannot reflect without invocation. It reflects a class through its java.lang.Class object. It expects a class to be used. Invoking the static method is based on an assumption that everything is going to go well. Consider the following class.

    public class Example {
  static {
	 System.loadlibrary( "applib" );
  }
}

    This example shows that executable bytecode is written into the static block (). When searching for classes that match a pattern, pure reflection will not invoke the static block but practical reflection will.

    Scenario

    Imagine you have been asked to write the following program using the Java programming language.

    Build a servlet to compare bytecode from three Java platforms. This servlet run in both Java 1 and 2 Platforms. It must be sub-platform-independent.

    Bytecode storage

    This is a design decision. The .class files for three platforms will be stored in three zip files.
    java0.zip

    All .class files from the Java 0 Platform.
    java1.zip

    All .class files from the Java 1 Platform.
    java2.zip

    All .class files for the Java 2 Platform.

    Pick a platform

    At the top level, you can explore one of three platforms. When picking a platform, this determines which zip file is used.

    Class details

    When looking at class details, the servlet decodes all of the data in a class file. Show public/protected classes. It displays the constant pool in a separate HTML table.

    Compare

    When looking at the details for a class from one platform, the HTML provides links to compare the current class with the corresponding class of other platform(s). If a class does not exist in another platform, this is indicated after a link.

    • Show new public/protected fields.
    • Show new public/protected methods.
    • Show public/protected methods with different access.
    • Show public/protected methods with different exception list.

    Practically useless

    The practical reflection API is not up to this task. It is practically useless because a class loader must not create classes from another virtual machine.
