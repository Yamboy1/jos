ArchitectureGroup

Article contributed by GilbertHerschberger (14 June 2000).

Introduction

Java has a very large namespace. This namespace is very accomodating for open source development. A large namespace could be compared to a large staduim, a large sports arena.

A person can only sit in one seat. Imagine a small group of 30 people is all alone inside a large staduim with 50,000 seats. There are plenty of seats to go around.

A very large namespace has many implications. It is difficult, for example, to fill the Java namespace. Like a very large stadium, Java's namespace cannot be "sold out". Java has plenty of namespace for everyone. There is plenty of namespace to go around. A collision is when two or more people want the same "seat" or package name. As 30 people should not fight over a seat in a stadium with 50,000 seats, Java programmers should not fight over package names in a very large namespace. All classes from an organization should go in a superpackage with an organization's name on it. A superpackage, like java and sun, contains nothing but other packages.

One implication is package ownership (or package stewardship). Is there a systematic way to prevent namespace collisions? All programmers to manage Java package names responsibly.

  • All classes should go in a package; the default package not be used.

  • Two different packages should not have the same name.

    In general, a package written by one author should not have the same name as a package written by any other author. Ambiguous package names needlessly complicate distribution, compilation, CLASSPATH management and runtime linking. A dynamically extendable application usually stops working when package names are ambiguous.

  • A package should not have two different names.

    In general, a virtual machine should not play tricks with package names so that bytecode is renamed on the fly. There are exceptional techniques where a few classes might be identical -- except for package name. These classes might be stored once and renamed by a virtual machine on the fly.

First come, first serve

In the tradition of open source development, the first programmer to use a package name "wins". The first person to use a package name usually gets to keep the package name. Other programmers are strongly encouraged to find another name if a package has already been used.

Package-oriented

In general, entire packages should be distributed. While it might be possible to distribute individual classes, it is not recommended. Other programmers are strongly encouraged to distribute the whole package even if only one class is used.

Likewise, package designers are strongly encouraged to organize classes in tightly-coupled packages. If a class is independent, it should go in a package by itself. Yes, a highly independent class can be in a package by itself, and should be.

For more information on package-oriented thinking, see also PackageOriented.

Rules

For an open source project like this one, programmers should work first as individuals. They should be able to experiment and develop packages with impunity. Here is how package ownership is applied to a team.

  1. Personal superpackage

    Put all those classes that you do not intend to distribute in your individual package. All your custom classes should go inside a superpackage with your name on it. Put your name, nick name or initials on it. If it is never redistributed, many of these rules do not apply.

    For example, my personal superpackage is gchii. It contains nothing but other packages.

  2. Team superpackage

    To redistribute packages to other members of the team, create a package for yourself within the team's superpackage. All custom classes shared by a team should go inside a superpackage with the team's name on it. Again, create another package for yourself with your name, nick name, or initials on it. So you'll have an individual namespace and a namespace as part of each team.

  3. Organization superpackage

    To publish packages from your organization, create a package for yourself within your organization's superpackage. This applies to authors who publish Java source code in books, magazines and across the Internet. All custom classes published by an organization should go inside a superpackage with the organization's name on it. Again, create another package for yourself with your name, nick name, or initials on it. So you'll have an individual namespace and a namespace as part of each team and a namespace as part of each organization.

    For example, my superpackage within org.jos is org.jos.gchii. The JOS Project maintains a organization superpackage package called org.jos.

Summary

A few simple rules are necessary for extreme software reuse, where loosly-coupled subsystems work together. By following rules like these, Java namespace can work for everyone.
