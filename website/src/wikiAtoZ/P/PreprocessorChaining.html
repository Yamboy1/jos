Introduction

We must not ignore the preprocessor market. We need a preprocessor framework, like JavaDoc, so that all preprocessors can live together in harmony.

Standard compiler

The standard Java compiler is monolythic, inseparable, as if the whole compiler is one giant component.

Source reader

The first component in the chain is a source code reader. A source code reader can read source code from an OS file system or SQL service on a network. It should return source code as an object to a preprocessor.

The class that reads source code must be part of the compiler configuration.

  • Local files A default reader must be able to read local files, including source stored in archives (.zip, .cab and .jar).
  • Remote files A URL reader should be able to download source code across the network. You should be able to compile directly out of an Internet CVS service.
  • SQL An SQL reader should be able to get source code out of an SQL service.
  • Secure (password-protected) files A secure reader should be able to decrypt source code files long enough for them to be compiled. This reader should ask for password before passing the plain text source on to the compiler.
  • Java-to-Java, Pascal-to-Java, COBOL-to-Java, CORBA/RMI IDL-to-Java and other language translators. A preprocessor can read COBOL code and translate it to Java source code. The intermediate source code is never stored. The original source file and original line numbers can be preserved for debugging.
  • Custom Translation You can write code that lasts longer. What you might have done with a time-consuming search and replace, you can do with a preprocessor. Don't touch all of those individual files. With a preprocessor framework, you can easily write your own custom translator. If you should have used the wrong namespace, you can write a custom translator to map old package names to new ones, recompile without rewriting all your old code.

Linked

Once we put a source code reader at the beginning of the food chain, a pre-processor is simply a source code reader that uses another source code reader. One preprocessor can be affectively linked to another, with the Java compiler at the end of the chain.

Benefits

This compiler architecture has many benefits.

  • Source code can be translated on its way to the compiler without storing additional copies of source code on an OS file subsystem.
  • You can compile directly from CORBA IDL to bytecode.
  • Many pre-processors can be chained to support all the different Java language extensions.
  • Java assembler can be mixed with Java, the programming language.
  • All pre-processors can be demonstrated by creating a consistent test framework.
  • You can manage source code like you manage other resources, with uniform resource locators.
  • It eliminates the needless assumption that source code must be stored in an OS file subsystem.
  • A framework would let many different researchers build compatible preprocessors and experiment with language extensions.
  • Since all of the pre-processed code is fed to your Java compiler, you're assured that it is standard bytecodes running in your JVM.
