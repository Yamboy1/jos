Introduction

Persistence has been plagued by persistent problems of its own.

A barrier is an obstruction that's impossible to work around. While a general obstruction slows you down, a barrier stops you. While an obstruction is an irritation, a barrier is impassable. Like anything that gets in your way, a barrier prevents you from going where you want to go or doing what you want to do.

A persistence barrier is the un-breakable wall between dynamically running programs and dormant static data. Persistence in software is the notion that something can live on, past the end of a program.

What is the persistence barrier? And, why is it so important? This article reminds us of when the barrier was invented and why it remains an important part of modern computing.

Independent Process

In the beginning, computer processors were limited to running only one program at a time on their own dedicated console. The persistence barrier was limited to communication between independent computing devices. It was limited to inter-device communication. It was part of peripheral programming, like attaching a printer to a host. It was part of modem programming, like the communications libraries made popular by bulletin board system (BBS). It was part of network programming, attaching one host to another and running hosts with services across a network. Where there are two independent computing devices, there is a persistence barrier between them.

Life was simpler then. It was easier on the operating system, too. The operating system only needed to load and run one program at a time.

And then, a process was invented. An expensive computer could be divided into logically independent processes, enabling it to run more than one program at a time. Programmers worked long and hard to make sure processes were as independent as possible. Virtual devices were created. Hardware devices were limited resources and only available to one device driver at a time. Where there are two independent processes, there is a persistence barrier between them.

So many protocols

How many protocols are there? How many will there be? There are hundreds of thousands of protocols already invented. There are thousands of protocols already published. There is a special purpose protocol for every thing imaginable. There are a hundred more protocols invented every day. Even with the increased attention of the Internet, the rate of invention hasn't slowed down. There's no end in sight.

A protocol is a mechanism to get around the persistence barrier. The protocol provides a way to move objects from one computing device to another or from one process to another.

If all protocols are trying to overcome the same barrier, why are there so many protocols? The persistence barrier is real. It is a wall. It is a road block. It is more than a simple irritation. Many programmers have tried to break the persistence barrier and failed.

So many programmers

Some programmers aren't aware of the barrier. They go about their work finding ways to ignore it. They mistakenly believe it will go away by itself. They use ad hoc socket programming to coerce data across the persisence barrier.

Some programmers mistakenly believe the persistence barrier was broken long ago. They go about their work thinking the problem was solved. They point to their network API ( NetworkAPI ) or SQL service ( SQLService ) as if the struggle were over. If the persistence problem was truely solved, we would see the end. There would be no new protocols.

Some programmers believe the problem is solved if we keep up the fantasy that everything is one big machine. The network is the computer, right? A good example of this approach is clustering technology, where 16 or 64 independent processor can act as one. Clustering is an old idea. Lots of bright people have been struggling for years to get it to work. It has proven to be fragile. This makes it difficult for programmers to write high quality software, which must be robust, rugged and dependable. We have a long way to go before all the processing power of the Internet becomes one giant cluster.

Some believe in the myth of virtual RAM. If only everything were virtual RAM, we would have no need for any other kind of persistence, and thus, no persistence barrier. A good example of this is OS/400. While virtual RAM helps many processes running on one machine, even if a machine is a cluster, it doesn't help transient inter-machine communication.

I am not a fan of Java serialization, either. This makes persistence obtuse and more difficult for programmers to write high quality software, with stability and compatibility.

Same old problems

These problems have plagued the software industry for decades. Our nagging problems with persistence must be taken more seriously. This problem is far more than an obstacle; it is a barrier. It needs to be thoroughly researched, well defined and understood. Only then will we break the persistence barrier once and for all time.

Now that Java puts distributed and network programming in easy reach, more programmers are going to hit this barrier. More programmers than ever before are going to be adversely affected by this barrier. And I'm worried, as so few programmers are prepared for it.
