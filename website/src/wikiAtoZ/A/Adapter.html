Introduction

Adapter is a buzzword for object-oriented programmers. In Java, the adapter has a more specific usage in class names. A custom adapter for an interface should be named customAdapter. What is an adapter? When should it be used?

An interface for a light, for example, could be designed like this: 

public interface Light {
  public void turnOff();
  public void turnOn();
  public void dim( int v );
  public void flash();
  public void strobe();
}

An adapter is a class. It has a relationship with an interface. An adapter for the Light interface is called LightAdapter. There are many different kinds of lights. These methods are part of the interface just in case a light could do something unique. Many lights won't be able to all of the functions of the light interface, so an adapter class is provided. It does not have to be used. 

public class LightAdapter
	 implements Light {
  public void turnOff() {}
  public void turnOn() {}
  public void dim( int v ) {}
  public void flash() {}
  public void strobe() {}
}

Required

When an interface requires a class to do something unique, this can be accomplished in one of two ways.
  1. Drop the adapter. If a class is required to do something unique for every method in an interface, do not provide an adapter class. Drop the adapter class entirely. Any class that implements the interface will either remain abstract or implement the required method.
  2. Make an abstract adapter. If a class is required to do something unique for some of the methods in an interface, provide an abstract adapter. An abstract adapter is not called customAdapter, but Basecustom. If the turnOff() and turnOn() methods must implemented for every Light, the base class should look like this: 
public abstract class BaseLight
	 implements Light {
  public void dim( int v ) {}
  public void flash() {}
  public void strobe() {}
}

Just-in-Case

Sometimes an interface is designed around just-in-case scenarios. An interface has methods that might do something unique. Or, it might not.

We see this a lot in the design of interfaces in the java.awt.event package. A class seldom implements something unique for windowIconized().

When there is a possibility that some class needs to implement something unique for an abstract method, it makes sense to design corresponding adapter class for an interface.

Operating System API

A modern operating system should be friendly to distributed computing. Throughout its API, it should be designed around interfaces and adapters. It should not be based on abstract base classes alone.

When designing for system objects in an operating system, it is critical to design an interface for each system object. It makes sense to design an interface and an adapter class for each system object, even if the adapter class remains abstract. An adapter class becomes the starting point for implementing system objects.

Short Shelf Life

Imagine how anyone could build a single-process operating system in this day and age (circa. 1999)! It is unthinkable. How would you get the market get excited about something so obviously short-sighted? Such an operating system would be a hard sell and have very short shelf life, right? And yet, Sun Microsystems created a single-process, multiple threaded JVM. The JVM is a single process operating system. The official way to support multiple processes is to start multiple copies of the JVM. JOS, jsh and Enchida are just a few efforts underway to force the JVM to support multiple processes.

A single process operating system could be built with abstract base classes. Designing system objects around abstract base classes alone, while convenient in the short-term, require lots of painful redesign down the road. An abstract base class is trouble for distributed computing, because distributed computing requires an interface. A modern operating system should be able to distribute system objects, too. Designing pairs of interfaces and their corresponding adapters leaves system objects open for distributed computing and future implementations of the system object.

After defining an interface, an adapter for the interface could be used as a base class for your implementation of a system object. From the single-process operating system's point of view, it sounds like a burden to have two classes where one would do. There is little difference, in fact, between multiple processes and network computing in the design of an operating system. Breaking the barrier between processes is also the key to breaking the barrier between machines.
