Article contributed by GilbertHerschberger (revised 15 June 2001).

Introduction

Surely, it must be a simple misunderstanding. A custom class loader is supposed to work inside a browser, right? Isn't that the point of having Java with a ClassLoader that's extendable?

I expect a Java-based operating system (JOS) to support HTML browsers from many different suppliers. I expect to start JOS on my computer and then ask it to launch my favorite HTML browser.

My favorite HTML browser must be written in Java in order to run on JOS. On the other hand, my favorite HTML browser will run on any other JVM, too.

Question: How does a browser set the security manager for applets when both the applets and the browser are running inside the same JVM?

I believe there is a parallel here between running applets in a browser and running applications in JOS.

  • Applets are expected to run semi-independently, as are other applications.
  • Applets can find and know about all loaded applets. Applications should be able to find and know about loaded applications.
  • The browser provides a security manager corresponding to the network-able class loader. Each application needs to set its own security manager.
  • Here goes. If an application happens to be a shell, it needs to set its own security manager and enable each application it loads to set its own security manager.
  • So, the operating system itself can load an application that's a shell. In turn, the shell can load an application that's an HTML browser. In turn, the HTML browser can load applets. Each application must have its own security manager in order to preserve the semantics of getting/setting the security manager in each process.
  • And, since the operating system is already implemented in Java, why can't it download an application using a network-able class loader?

Example

The security headache grows bigger because, I think, something is fundamentally wrong with Java's security manager in general. It always assumes the worst of the Internet. I am working with an intranet, not Internet. Defects in the security manager make turns a powerful Java-enabled HTML browser into the weakest possible platform. So far, every solution has been a dead-end. Every attempt to get Java to work is stopped cold by the security manager. The security manager has been a barrier to get the intranet done.

I put together an applet and class loader to find out if it works in a browser. I get nothing but a security exception. Using a custom class loader isn't allowed in my browser.

Here is an applet: 

public class MinimumApplet
	 extends Applet {
  public void init() {
	 MinimumClassLoader loader =
		  new MinimumClassLoader();
  }
}

Here is a class loader: 

public class MinimumClassLoader
	 extends ClassLoader {
  public MinimumClassLoader() {
  }

  protected Class loadClass( String name, boolean resolve )
		throws ClassNotFoundException {
	 throw new ClassNotFoundException( name );
  }
}

It seems to be a catch-22, unless you write your own browser.

How do you get an applet to load with a custom class loader?

The code comes down to this: 

public class MyApplet
	 extends Applet {
  public void init() {
	 String componentName = getProperty( "COMPONENT" );

	 ClassManager manager = new ClassManager( this );
	 Class c = manager.findClass( panelName );
	 Object o = c.newInstance();
	 add( (Component) o );
  }
}
