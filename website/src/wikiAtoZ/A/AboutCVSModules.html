Introduction

A snapshot is a distribution option. It is an archive used to distribute other files. It is a popular alternative to CVS-compatible tools. For those with a limited Internet connection and for those without CVS-compatible tools, a snapshot is a great way to download software.

The JOS Project has many subprojects, Some of these projects are supported by Concurrent Versioning System (CVS). Within a CVS service, files are organized by module (or subproject). A snapshot is a great way to download a CVS module.

Strengths

Snapshots have many strengths and few weaknesses.

  • Most of us

    A snapshot meets the needs of most of us. Most of us want to use ready-to-run software. Most of us are not frequent contributors to a open source project. Unless you're a frequent contributor to a project, a snapshot is probably your best option.

  • Client-side

    On the client-side, downloading a snapshot is simpler. A snapshot does not require you to install CVS-compatible client software. Only your favorite Internet-ready HTML browser is required.

    The results are the same. A snapshot contains you all the public (anonymous) files. A snapshot is created by using anonymous as a user name. It is downloaded with a CVS-compatible client. An archive is created, both .zip and .tgz, and uploaded to a website.

  • Server-side

    On the server-side, snapshots require less software. Snapshots can be distributed through FTP and HTTP. Snapshots can be distributed on CD-ROM. Mirror sites can (re)distribute snapshots; a CVS-compatible service is not required. Snapshots are available long after someone shuts down the original CVS service. To comply with popular open-source and free-license software agreements, this can be significant strength of a snapshot.

  • History

    It is far easier to download a snapshot when interested in project history. A snapshot can provide the most reliable way to perform long-term, distributed testing, especially when testers are also volunteers. Anyone working from the snapshot should get identical results. A snapshot can be well documented, including how to fix known bugs. While distributed testing can be done with CVS, results are less than optimal because each volunteer works with slightly different software. Snapshots are more likely to result in high quality software.

  • Multiple platforms

    Snapshots can provide simple support for multiple platforms. By providing a snapshot in multiple archive formats, a wide audience of users and co-developers can be reached. Instructions for downloading new releases are similar among different platforms.

Weaknesses

A snapshot is delayed. Snapshots are typically created on schedule. It does not provide up-to-the-second access to a module. If you're a frequent contributor, you probably require require up-to-the-second access. It may be worthwhile to spend time learning CVS-compatible client software.

A snapshot contains the whole module; it is not a patch. Frequent use of a typical CVS-compatible client may be faster for some because only new and changed files are downloaded.

Download

Download a snapshot. Uncompress it. More instructions are found inside a README or similar file. For more information on downloading CVS modules, see also DownloadCVSModules .
