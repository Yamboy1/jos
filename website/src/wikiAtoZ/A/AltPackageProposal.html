AltPackagePages; ClasspathProject; VersionlessVirtualMachine

Article contributed by GilbertHerschberger (28 December 1999).

Introduction

Most of us want compatibility across virtual machines. The solution to compatibility across all virtual machines seems elusive. Should we have a universal virtual machine? or a version-less virtual machine? Or, should we reinvent the common platform using namespace?

Inclusive

When we choose namespace as a mechanism to provide compatibility across all virtual machines, we are able to do something that catches the imagination of the Java community. We can provide a common platform that works on top of virtual machines from Sun Microsystems, too. None of the virtual machines are left out. All of them are included in a namespace solution to compatibility.

Namespace

Unlike the ClasspathProject, we should organize to produce the alt namespace. The alt packages should be developed to run inside existing virtual machines as well as experimental virtual machines. You can use the result of the alt packages in your applications today and far into the future.

Do something different. Use alt.ion instead of java.io. The alt.ion package will be implemented the same way on every machine because it will be the same on every machine. It is written once and distributed as bytecode to every machine. As a programmer, you should choose the best I/O package for input/output. When you can't count on the java.io package to be implemented the same way on every virtual machine, it might not be the best I/O package after all.

How will you test a replacement for the java.io package? If you call your new package alt.ion you can use any virtual machine.

How Sun does it

We can test the same way Sun Microsystems does. How did they develop classes in the java. packages? The secret is this: they don't. All of the classes under development are written first in an alternative namespace (like javax.). After they have designed, developed, and debugged a class, it's recompiled into the java.* namespace.

A working virtual machine is used to build a new class library. JDK 0.0 was used to build JDK 1.0. JDK 1.0 was used to build JDK 1.1. JDK 1.1 was used to build JDK 1.2. Sun Microsystems can use virtual machines to build virtual machines by developing in an alternative namespace.

About JOS

As members of the JOS Project, we might be a long way from having a class library that works. Developing in an alternative namespace would shorten the distance from where we are to a full implementation of the Java 2 Platform.

We should not build in a JOS-specific alternative namespace. Rather, we should join the Java community in building in an alternative namespace that belongs, not only to JOS, but to everyone.

alt.* namespace

It is a good idea to build many of the following packages: See correction below.

alt.applet.0
alt.applet.1
alt.applet.2
alt.awt.0
alt.awt.1
alt.awt.2
alt.io.0
alt.io.1
alt.io.2
alt.net.0
alt.net.1
alt.net.2
alt.util.0
alt.util.1
alt.util.2

These packages can be built in any order. When a package has been built, it can be used by everyone -- immediately -- in their Java applications. There is no need to be perfectly compatible with the class design from Sun Microsystems.

Which Java compiler should we use? For the most part, it does not matter how the classes are created. An assembler, like Jasmin, could be used and still meet the goal of alternative namespace.

Over time, it keeps getting better. Alternative packages will eventually depend little if at all on the java.* packages. The alternative packages will assume the BytecodeNativeInterface, rather than the JavaNativeInterface. These packages will be available in every virtual machine without rewriting a single line of code.

Just-in-time class libraries

Distribute as few or as many of the alternative packages as you see fit. The alternative packages should be distributed just-in-time, rather than just-in-case. There will be lots of megabytes worth of bytecode in the alternative packages, and there should be. Size will not be a problem when packages are correctly organized in package files.

The alternative class libraries are the same on every virtual machine. Only the JVM-specific classes are different (as necessary) for each virtual machine. Through the alternative class libraries, and only through the alternative class libraries, you can hope to run your application unchanged on any operating system, including JOS.

Can't be fixed

Sun Microsystems broke run-anywhere when they released JDK 1.1. It was incompatible with everything prior to it. Once that mistake was made, there was no hope of providing perfect compatibility inside the java. namespace. Sun Microsystems "owns" the java. namespace anyway. If they want to reuse the namespace -- failing to reuse the software -- that is completely up to them. There's nothing that can be done about that.

Even if Sun Microsystems were to become convinced that reasonable incompatibility was a mistake, the problem doesn't go away by changing the classpath. Other mechanisms are bundled with Sun's definition of the Java platforms.

Serialization

Serialization is bad, very bad. Serialization has negative effects on compatibility. The reason is provided by members the ClasspathProject. It has proven to be quite impossible to make a class library in a clean room that also provides compatibility with serialized object. If compatibility with public and protected methods wasn't difficult enough, the goal of perfect compatibility with serialization is impossible to reach. It can't be done.

Feedback

The Java community needs to think about the technical implications of run-anywhere. It means that all mechanisms must be independent of the JVM, except JVM-specific classes. The BytecodeNativeInterface is an example of a mechanism that's independent of any virtual machine implementation.

Should the JOS Project wait until the Java community figures out that an alternative namespace is needed? Or, should we lead the way?

Do we have a choice beyond an alternative namespace? The search for a solution continues.

Excellent! Your questions have brought my attention directly to my assumptions. Here are some rules we must use for the alternative namespace if the plan can succeed:

  1. We need an alternative namespace for different packages with the same name, like java.applet, java.awt, java.io, java.lang, java.net, java.sql, java.util, javax.servlet and others.

    Alternative packages are not for everything. We need an alternative namespace for packages originally published by Sun Microsystems under java. and sometimes javax.. Alternative namespace is useful when resolving the ambiguity between different packages with the same name.

    Our org.jos namespace is perfectly adequate for our own packages. Our packages are under jos.* and org.jos, and should be.

  2. Alternative namespace is copied to java. and javax.

    Unlike my previous thinking, I believe that we must copy classes from the alternative namespace to the java. namespace, so that Java applications might find them there. It is not necessary to have a class in java. use a class in alt.*.

    In order for the java. namespace to come out perfectly compatible with the Java class libraries from Sun Microsystems, nothing must be allowed in alt. except classes design-compatible with Java Platform Specification.

  3. decaf-specific classes must be written in a jos.decaf. or org.jos.decaf. namespace.

    When classes in the alternative namespace need JVM-specific classes, they should use the BytecodeNativeInterface (or BCNI). And when we copy packages from the alternative namespace to java.*, they will continue to use BCNI.

  4. Focus on the libraries that truely make up the Java "platform".

    The alternative namespace is useful for developing a cleanroom implementation of the Java class libraries. It should not be abused for writing extensions to the language or platform.

    Specifically, the alternative namespace must be perfectly compatible with the Java Platform Specification. Class design has already been determined in advance for all classes in the alternative namespace. The only question is package name(s).

    Do we make our packages conform exactly with the interfaces in the respected Java releases?

    We make packages in the alternative namespace conform exactly and precisely with the interfaces in the respective Java releases. It is nothing more. It is nothing less.

    If anyone wants to go beyond the design of Java releases, they must use another namespace, such as their own personal namespace or an organizational namespace.

  5. There is no complete-ness requirement.

    The alternative namespace does not need to implement all of the classes in java.*. If a class is implemented, it must be implemented exactly. We can choose not to implement a class. We can choose not to implement a package.

JVM-specific classes must never exist inside the java. or alt. namespace. This will never work. As soon as one person abuses the alt.* namespace with classes beyond the Java release, the intention of the namespace has been compromised. -- GilbertHerschberger (30 December 1999).
